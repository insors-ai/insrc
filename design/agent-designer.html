Agent Design Document

DESIGNER Designer Agent

insrc coding agent — persona detail — March 2026

Overview

The Designer agent produces grounded, validated design documents through an iterative, user-validated workflow. Rather than generating a monolithic design in one pass, the Designer breaks the work into individual requirements, sketches each one against the actual codebase, and validates with the user at every significant decision point.

The Designer also handles review intents (code critique). The core design flow and review are distinct workflows sharing the same persona and tool restrictions.

PropertyValue
Intentsrequirements, design, review
Default providerClaude (always escalated — both stages)
Claude tierstandard (Sonnet); @opus for deep architectural reviews
Output formatsMarkdown or HTML — configurable via design templates
Tool usageRead-only — graph_*, Read, Grep, Glob, git diff/log
Session carry-forwardStores [requirements] and [design] tags in L2
MutationsNone — the Designer never calls Write, Edit, or Bash with side effects
Read-only guarantee: The orchestrator enforces that the Designer agent has no access to mutating tools. Its tool schema omits Write, Edit, and mutating Bash commands entirely. This is a hard constraint, not a prompt instruction — the agent physically cannot produce a file-write tool call.

Settings

Design templates

The Designer ships with out-of-the-box templates for Markdown and HTML output. Templates define the document skeleton — section structure, heading hierarchy, CSS (for HTML), and placeholder slots that the Designer fills during the per-requirement iteration.

TemplateFormatWhen used
default-md Markdown Default for REPL / inline display. Sections as ## headings.
default-html HTML When --html flag is set, or user requests HTML. Dark-theme styled document with ToC sidebar.

Template structure

// personas/designer/templates.ts
interface DesignTemplate {
  /** Template identifier */
  id: string;
  /** Output format */
  format: 'markdown' | 'html';
  /** Document skeleton with named slots */
  skeleton: string;
  /** Section definitions — order and heading text */
  sections: Array<{
    id: string;
    heading: string;
    /** Prompt hint for this section (injected into LLM prompt) */
    guidance: string;
  }>;
  /** CSS (HTML templates only) */
  css?: string;
  /** Whether this is a built-in or user-defined template */
  builtin: boolean;
}

Template override

Users can override the built-in templates by placing custom templates in the project's .insrc/templates/ directory. Custom templates follow the same DesignTemplate schema. The Designer resolves templates in order:

  1. User-specified template via --template <id> flag
  2. Project-level template in .insrc/templates/design-*.{md,html}
  3. Built-in default-md or default-html
// Template resolution
function resolveTemplate(opts: { format?: 'markdown' | 'html'; templateId?: string }): DesignTemplate {
  if (opts.templateId) {
    const custom = loadTemplate(`.insrc/templates/${opts.templateId}`);
    if (custom) return custom;
    throw new Error(`Template "${opts.templateId}" not found`);
  }
  const format = opts.format ?? 'markdown';
  const projectTemplate = loadTemplate(`.insrc/templates/design-default.${format === 'html' ? 'html' : 'md'}`);
  if (projectTemplate) return projectTemplate;
  return BUILTIN_TEMPLATES[format === 'html' ? 'default-html' : 'default-md'];
}

Flow

The Designer follows an iterative workflow with user validation gates. Every significant decision is surfaced to the user before proceeding, ensuring the final design reflects the user's intent rather than the model's assumptions.

Input (user message + optional requirements doc) | v [1] Extract concise requirements list LOCAL + CLAUDE | v VALIDATE User confirms / edits requirements | v [2] Create a todo for each requirement | v [3] For each requirement todo: | | | v | [3a] Write short sketch LOCAL | | - Analyze codebase for reusable modules | | - Check other indexed projects for services/modules | | - Produce summary implementation flow | | | v | [3b] Claude reviews & fixes sketch CLAUDE | | - Validate reuse choices against codebase | | - Fix incorrect entity references | | - Strengthen or correct summary flow | | | v | VALIDATE User confirms summary flow | | | v | [3c] Write detailed design section LOCAL | | | v | [3d] Claude reviews & fixes detail CLAUDE | | - Validate interface contracts | | - Fix integration point accuracy | | - Ensure cross-requirement consistency | | | v | VALIDATE User confirms detail | | | v | [mark todo done] | v [4] Assemble final document from template + sections

Step 1 — Extract requirements

The Designer takes the user's message (and optional requirements doc) and produces a concise, numbered list of functional and system requirements. This is not a full requirements document — it is a short, scannable list the user can quickly approve, edit, or reject.

Local sketch

The local model (qwen3-coder) receives the user message plus L4 graph context and produces the initial requirements list. If a requirements doc is provided, the local model extracts and condenses rather than generating from scratch. 

// personas/designer.ts — requirements extraction (Stage 1: local)
const REQ_EXTRACT_SYSTEM = `You are a requirements analyst for a software project.
You have the user's request and relevant code context from the codebase knowledge graph.

Your task: produce a concise numbered list of functional and system requirements.
Each requirement should be:
- One sentence, testable, specific
- Grounded in the codebase context (reference existing entities where relevant)
- Classified as FUNCTIONAL or SYSTEM

If a requirements document is provided, extract and condense it — do not regenerate.
If no requirements document is provided, derive requirements from the user's request
and the codebase context.

Output format:
1. [FUNCTIONAL] Requirement statement — references: entity_name, file.ts
2. [SYSTEM] Requirement statement — references: entity_name
...

Keep the list to 5–15 items. Merge overlapping requirements. Surface ambiguities
as open questions at the end (max 3).`;

Claude enhancement

Claude receives the local model's list plus the same context and sharpens it: filling gaps, resolving obvious ambiguities, flagging conflicts with existing architecture. The output remains a concise list — not a full document yet. 

// personas/designer.ts — requirements extraction (Stage 2: Claude)
const REQ_ENHANCE_SYSTEM = `You are a senior engineer reviewing a requirements list
produced by a local model. You have the user's original request, the local model's
list, and the relevant code context.

Your task:
1. Sharpen each requirement into a concrete, testable statement.
2. Add missing requirements the local model overlooked (cross-repo impact,

4. Flag any requirements that conflict with existing architecture.

Output the same numbered format. Keep it concise — this list will be shown to the
user for validation, not stored as a final document.`;

Step 2 — Validate requirements

The requirements list is presented to the user. This is the first VALIDATE gate. The user can:

  • Approve — proceed as-is
  • Edit — add, remove, or reword requirements; the Designer re-numbers and continues
  • Reject — start over with additional guidance
// personas/designer.ts — validation gate for requirements
interface ValidationGate {
  /** What is being validated */
  stage: 'requirements' | 'summary-flow' | 'detail';
  /** The content presented to the user */
  content: string;
  /** Requirement index (for per-requirement gates) */
  requirementIndex?: number;
  /** User's response */
  response: 'approve' | 'edit' | 'reject';
  /** User's edits or rejection reason */
  feedback?: string;
}

Step 3 — Create todos

Once the user approves the requirements list, the Designer creates one todo item per requirement. Each todo tracks the state of that requirement through the sketch → validate → detail → validate cycle. 

// personas/designer.ts — requirement todo
interface RequirementTodo {
  /** Requirement index (1-based, matching the validated list) */
  index: number;
  /** The requirement statement */
  statement: string;
  /** Requirement type */
  type: 'functional' | 'system';
  /** Referenced entities from the requirements extraction */
  references: string[];
  /** Current state */
  state: 'pending' | 'sketching' | 'sketch-reviewing' | 'sketch-validated' | 'detailing' | 'detail-reviewing' | 'done' | 'skipped';
  /** Sketch output (populated after Step 3a) */
  sketch?: RequirementSketch;
  /** Detailed section output (populated after Step 3b) */
  detail?: string;
}

Step 4 — Per-requirement iteration

The Designer iterates over each requirement todo sequentially. For each requirement, it performs two sub-steps with a validation gate after each.

Step 4a — Sketch

For each requirement, the Designer produces a short sketch that answers: how will this be implemented? The sketch is grounded in the actual codebase through two analyses:

Codebase analysis

The local model searches the current project's knowledge graph for existing modules, patterns, and utilities that can be reused or extended for this requirement. This prevents the design from proposing new code that duplicates what already exists.

Cross-project analysis

Services and external modules may be available in other indexed projects in the dependency closure. The Designer queries the graph across all session.closureRepos — not just the current project. This surfaces reusable services, shared libraries, and API contracts that the design should integrate with rather than rebuild.

Summary implementation flow

The sketch concludes with a summary flow: a concise description of how data moves through existing and proposed components to satisfy this requirement. 

// personas/designer.ts — per-requirement sketch
interface RequirementSketch {
  /** Requirement index */
  index: number;
  /** Existing modules/entities that can be reused */
  reusable: Array<{
    entity: string;      // entity ID or name
    project: string;     // which indexed project it's from
    relevance: string;   // one-line description of how it's relevant
  }>;
  /** Proposed new components (if reuse is insufficient) */
  proposed: Array<{
    name: string;
    kind: 'function' | 'class' | 'module' | 'interface' | 'type';
    file: string;        // proposed file location
    purpose: string;
  }>;
  /** Summary implementation flow */
  summaryFlow: string;
  /** Open concerns surfaced during analysis */
  concerns: string[];
}

// Stage 1: local model sketch prompt
const SKETCH_SYSTEM = `You are a software architect analyzing a single requirement
against an existing codebase. You have the requirement, the full codebase context
(including entities from other indexed projects in the dependency closure), and the
approved requirements list for overall context.

Your task for this ONE requirement:
1. REUSABLE MODULES — Search the provided context for existing modules, functions,
   types, or services that can be reused or extended. Include entities from other
   indexed projects if relevant (name the project). Do not propose new code if
   existing code already does the job.
2. PROPOSED COMPONENTS — If reuse is insufficient, propose new components. Name
   them, specify where they belong in the file tree, and describe their purpose.
3. SUMMARY FLOW — Describe how data moves through existing + proposed components
   to satisfy this requirement. Keep it to 3-5 sentences. Name the entities.
4. CONCERNS — Note any integration risks, performance concerns, or dependencies
   that the user should be aware of.

Be concrete. Name files, functions, types. Do not be vague.`;

Step 4b — Claude reviews sketch

Before presenting the sketch to the user, Claude reviews and fixes the local model's output. The local model may misidentify entities, propose reuse of unsuitable modules, or produce an inaccurate summary flow. Claude corrects these issues so the user sees a high-quality, validated sketch at the gate — not a raw local draft. 

// personas/designer.ts — Claude sketch review
const SKETCH_REVIEW_SYSTEM = `You are a senior engineer reviewing a design sketch produced
by a local model for a single requirement. You have:
- The requirement statement
- The local model's sketch (reusable modules, proposed components, summary flow)
- The codebase context (entity bodies, signatures, types)
- The full requirements list for cross-requirement context

Your task:
1. VALIDATE REUSE — Check that each "reusable" entity actually exists in the context
   and is genuinely relevant. Remove false positives. Add missed reuse opportunities.
2. VALIDATE PROPOSALS — Check that proposed new components don't duplicate existing code.
   Verify proposed file locations match the project's conventions.
3. FIX SUMMARY FLOW — Correct any inaccurate entity references, wrong call chains, or
   missing steps. The flow should be traceable through the actual codebase.
4. FLAG CONCERNS — Add any integration risks or cross-requirement conflicts the local
   model missed.

Output the corrected sketch in the same format. Mark corrections with [FIXED] and
additions with [ADDED] so the user can see what changed.`;
// personas/designer.ts — sketch review execution
async function reviewSketch(
  sketch: RequirementSketch,
  requirement: RequirementTodo,
  context: AssembledContext,
): Promise<RequirementSketch> {
  const messages: LLMMessage[] = [
    { role: 'system', content: SKETCH_REVIEW_SYSTEM },
    { role: 'user', content: `## Requirement\n${requirement.statement}` },
    { role: 'user', content: `## Local Sketch\n${formatSketch(sketch)}` },
    { role: 'user', content: `## Code Context\n${formatGraphContext(context)}` },
  ];

  const reviewed = await claudeProvider.complete(messages, {
    maxTokens: 2048,
    model: resolveModel('designer.sketch-review'),
  });

  return parseSketch(reviewed);
}

Step 4c — Validate summary flow

The Claude-reviewed sketch (reusable modules + proposed components + summary flow) is presented to the user. This is the second VALIDATE gate (per requirement). Because Claude has already validated and fixed the sketch, the user sees a high-quality proposal — not a raw local draft.

This gate is critical: it is where the user corrects the approach before the detailed design is written. Catching a wrong direction here saves significant LLM cost and user time.

  • Approve — proceed to detailed section
  • Edit — user adjusts the approach (e.g., "use X instead of Y", "don't create a new module, extend Z"); the Designer re-runs the local sketch + Claude review cycle with the feedback injected
  • Reject — skip this requirement or re-derive it

Step 4d — Write detailed section

Once the sketch is validated, the Designer generates the full detailed design section for this requirement. The section is scoped to the design template's slot for this requirement and includes:

  • Interface contracts (TypeScript signatures or pseudocode)
  • Data flow with concrete entity references
  • Integration points with existing code (from the sketch's reuse analysis)
  • Migration notes (if modifying existing entities)
  • Risk callouts (from the sketch's concerns)
// personas/designer.ts — detailed section generation
const DETAIL_SYSTEM = `You are a senior engineer writing one section of a design document.
You have:
- The requirement statement
- The approved sketch (reusable modules, proposed components, summary flow)
- The codebase context (entity bodies and signatures)
- The overall requirements list (for cross-requirement context)

Write a detailed design section covering:
1. INTERFACE CONTRACTS — TypeScript signatures or pseudocode for new/modified entities
2. DATA FLOW — Concrete path through existing and new components, with entity names
3. INTEGRATION POINTS — How this connects to existing code. Name the files and functions.
4. MIGRATION NOTES — If existing entities are modified, describe the change and impact.
5. RISKS — Anything from the sketch concerns that needs explicit handling.

Output as markdown. Use code blocks for interface definitions. Be specific — name every
entity, file, and function. Do not repeat the requirement statement as a preamble.`;

Step 4e — Claude reviews detail

Before the user sees the detailed section, Claude reviews and fixes it. This ensures interface contracts are valid, integration points reference real entities, and the section is consistent with other already-completed requirement sections. 

// personas/designer.ts — Claude detail review
const DETAIL_REVIEW_SYSTEM = `You are a senior engineer reviewing a detailed design section
produced by a local model. You have:
- The requirement statement and approved sketch
- The local model's detailed section
- The codebase context
- Previously completed requirement sections (for cross-requirement consistency)

Your task:
1. VALIDATE INTERFACES — Check that TypeScript signatures are syntactically correct
   and compatible with existing types in the codebase.
2. VALIDATE INTEGRATION — Verify that named files, functions, and call chains exist
   in the context. Fix incorrect references.
3. CROSS-REQUIREMENT CONSISTENCY — Check for conflicts with previously completed
   sections (duplicate entity names, incompatible interfaces, contradictory data flows).
4. COMPLETENESS — Ensure migration notes cover all modified entities. Ensure risks
   from the sketch concerns are addressed.

// personas/designer.ts — detail review execution
async function reviewDetail(
  detail: string,
  sketch: RequirementSketch,
  requirement: RequirementTodo,
  completedSections: string[],
  context: AssembledContext,
): Promise<string> {
  const messages: LLMMessage[] = [
    { role: 'system', content: DETAIL_REVIEW_SYSTEM },
    { role: 'user', content: `## Requirement\n${requirement.statement}` },
    { role: 'user', content: `## Approved Sketch\n${formatSketch(sketch)}` },
    { role: 'user', content: `## Local Detail\n${detail}` },
    { role: 'user', content: `## Code Context\n${formatGraphContext(context)}` },
    ...(completedSections.length > 0
      ? [{ role: 'user' as const, content: `## Completed Sections\n${completedSections.join('\n---\n')}` }]
      : []),
  ];

  const reviewed = await claudeProvider.complete(messages, {
    maxTokens: 4096,
    model: resolveModel('designer.detail-review'),
  });

  return reviewed;
}

Step 4f — Validate detail

The Claude-reviewed detailed section is presented to the user. This is the third VALIDATE gate (per requirement). Claude has already validated interfaces, fixed entity references, and checked cross-requirement consistency.

  • Approve — mark the requirement todo as done, move to next
  • Edit — user provides feedback; the Designer revises the section
  • Reject — go back to sketch stage for this requirement

Once all requirement todos are done, the Designer assembles the final document from the template skeleton plus all validated detail sections.

Validation Gates

Gate mechanics

Validation gates are the Designer's primary mechanism for keeping the user in the loop. Each gate pauses execution and presents a summary to the user. The orchestrator manages the gate interaction — the Designer persona itself is stateless and receives the user's response as input to the next step. 

// orchestrator/validation.ts — gate execution
async function executeGate(gate: ValidationGate): Promise<ValidationGate> {
  // Present content to user
  render(gate.stage, gate.content, gate.requirementIndex);

  // Wait for user response
  const response = await waitForUserInput();

  if (response.type === 'approve') {
    return { ...gate, response: 'approve' };
  }
  if (response.type === 'edit') {
    return { ...gate, response: 'edit', feedback: response.text };
  }
  return { ...gate, response: 'reject', feedback: response.text };
}

User responses

ResponseEffectAvailable at
Approve Proceed to next step. Content is accepted as-is. All gates
Edit User provides textual feedback. The Designer re-runs the current step with the feedback injected as an additional user message. Max 3 edit rounds per gate before the Designer asks for explicit direction. All gates
Reject At requirements gate: restart extraction with user's rejection reason as guidance. At sketch gate: skip this requirement or re-derive. At detail gate: go back to sketch for this requirement. All gates
Skip Skip this requirement entirely. The todo is marked skipped and no section is generated for it. Available only at per-requirement gates. Sketch, detail gates

Auto-approve mode

For batch or CI usage, the Designer supports an --auto-approve flag that skips all validation gates and proceeds with the LLM's output at each step. This is equivalent to approving every gate automatically.

Not recommended for first runs: Auto-approve is useful for regenerating a design with known-good requirements, or for CI pipelines. For initial design work, the validation gates are where most value is created — catching wrong assumptions early saves significant rework.

Codebase Analysis

The sketch step (4a) relies on thorough codebase analysis to ground the design in reality. This analysis happens in two scopes.

Local project analysis

The Designer queries the current project's knowledge graph for entities that match the requirement's domain: 

// personas/designer.ts — local codebase analysis
async function analyzeLocalCodebase(
  requirement: RequirementTodo,
  context: AssembledContext,
): Promise<ReusableEntity[]> {
  // Vector search for entities relevant to this requirement
  const vec = await embed(requirement.statement);
  const hits = await db.searchCode(vec, {
    repos: [context.currentRepo],
    limit: 15,
  });

  // Expand to neighbours for integration context
  const neighbours = await db.neighbors(
    hits.map(h => h.id),
    { hops: 1, signaturesOnly: true },
  );

  // Fetch referenced types for contract compatibility
  const types = await db.referencedTypes(hits.map(h => h.id));

  return hits.map(h => ({
    entity: h.id,
    project: context.currentRepo,
    body: h.body,
    signature: h.signature,
    neighbours: neighbours.filter(n => n.sourceId === h.id),
    types: types.filter(t => t.referencedBy === h.id),
  }));
}

Cross-project analysis

Services, shared libraries, and external modules may live in other indexed projects within the dependency closure. The Designer queries across all closure repos to find reusable components the design should integrate with rather than rebuild. 

// personas/designer.ts — cross-project analysis
async function analyzeCrossProject(
  requirement: RequirementTodo,
  context: AssembledContext,
): Promise<ReusableEntity[]> {
  const otherRepos = context.session.closureRepos.filter(
    r => r !== context.currentRepo,
  );

  if (otherRepos.length === 0) return [];

  const vec = await embed(requirement.statement);
  const hits = await db.searchCode(vec, {
    repos: otherRepos,
    limit: 8,
  });

  // For cross-project hits, fetch signatures only (not full bodies)
  // to keep context budget manageable
  return hits.map(h => ({
    entity: h.id,
    project: h.repo,
    body: null,  // not included — only signatures for cross-project
    signature: h.signature,
    neighbours: [],
    types: [],
  }));
}
Cross-project is signatures-only: To stay within context budget, cross-project hits include only signatures (not full bodies). If the user approves a sketch that references a cross-project entity, the detail step can fetch the full body on demand.

Output

Schema

Session carry-forward

Review Intent

The review intent is a separate workflow from the design flow above. It critiques existing code rather than designing new code. The review workflow does not use the iterative todo-based pattern — it is a single-pass Claude analysis.

Context assembly

The local model uses read-only tools to build the review payload:

  • If a diff is in scope (staged changes, a PR, a specific commit): git diff or git show — the diff is the primary input
  • Bodies of touched entities — fetched by matching diff hunks to indexed entity ranges
  • Signatures of first-degree callers and callees — for call-site compatibility context
  • If no diff (reviewing an existing file or function): the entity body + its immediate neighbours' signatures

Claude review

// personas/designer.ts — review prompt (system)
const REVIEW_SYSTEM = `You are a senior engineer performing a code review. Be specific —
cite line numbers or entity names for every finding.

Review for:
- CORRECTNESS: logic errors, off-by-ones, unhandled edge cases, race conditions
- SECURITY: injection, unvalidated input, exposed internals, over-permissive access
- PERFORMANCE: unnecessary allocations, N+1 queries, blocking calls in hot paths
- STYLE: consistency with surrounding code patterns shown
- COMPLETENESS: missing error handling, missing tests, undocumented public API

Format: one section per category. Skip categories with no findings.
Severity for each finding: CRITICAL / WARN / NOTE.
For CRITICAL and WARN findings, include a suggested fix (code snippet or description).`;

Escalation to Opus

Sonnet handles most reviews. The user can prefix with @opus to escalate for large diffs (> 500 lines), security audits, performance-critical paths, or cross-repo changes. Cost estimate shown before any Opus call.

Context budget

Claude calls are bounded to keep costs predictable:

ComponentBudgetContents
System prompt~500 tokensPersona + step-specific instructions
User request~200 tokensThe user's original message
Requirements list~300–800 tokensThe validated requirements (all steps)
Local sketch~1–2K tokensSketch output for current requirement
Graph context~4–8K tokensEntity bodies + signatures + types
Cross-project signatures~1–2K tokensSignatures from other indexed projects
Validation feedback~200 tokensUser edits from prior gate (if any)
Total input~8–14K tokens
Response budget4096 tokensClaude's output for this step

A full design run with 10 requirements will make ~40 Claude calls: requirements enhancement + 10 sketch reviews + 10 detail reviews + 10 detail generations + assembly overhead. At Sonnet pricing (~$3/M input, ~$15/M output), a typical full design costs $0.80–$2.00. The orchestrator tracks cumulative spend via /cost.

Error handling

FailureBehaviour
Local sketch fails (Ollama timeout) Skip local stage. Claude works without sketch — quality degrades but turn completes. User notified.
Claude fails (API error, timeout) Return local sketch as-is with warning. For per-requirement steps, mark the todo as failed and continue to next requirement.
Claude returns malformed output Retry once. If still malformed, present raw text to user at the validation gate — user can edit it into shape.
Graph context empty Proceed without graph grounding. User warned that results may reference entities that don't exist.
User abandons mid-flow Completed requirement todos are preserved. The Designer can resume from the last completed todo in a subsequent turn (state stored in session).

Open Questions

  1. Validation gate UX in REPL vs CLI — In the REPL, gates are natural (user types a response). In CLI one-shot mode, how should gates work? Options: --auto-approve for batch, or an interactive prompt sequence.
  2. Parallelism within requirements — Some requirements may be independent and could be sketched/detailed in parallel. Should the Designer detect independence (via graph overlap analysis) and parallelise, or always iterate sequentially?
  3. Cross-project index freshness — If another indexed project has changed since last indexing, the Designer's cross-project analysis may surface stale signatures. Should the Designer check index freshness and warn, or trigger a re-index?
  4. Design versioning — When the user re-runs the Designer on the same topic, should it produce a diff against the previous design (stored in L2), or a fresh document?
  5. Structured output enforcement — Should the requirement list and sketch be enforced via Claude's structured output mode, or extracted post-hoc from markdown with a parsing pass?
