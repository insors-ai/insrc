Agent Design Document

DESIGNER Designer Agent

insrc coding agent — persona detail — March 2026 (updated: agent framework v1, search planner, context compression)

Overview

The Designer agent produces grounded, validated design documents through an iterative, user-validated workflow built on the agent framework — a step-based state machine with file-based checkpointing, crash recovery, and resume support. Rather than generating a monolithic design in one pass, the Designer breaks the work into individual requirements, sketches each one against the actual codebase, and validates with the user at every significant decision point.

The Designer also handles review intents (code critique). The core design flow and review are distinct workflows sharing the same persona and tool restrictions.

Read-only guarantee: The orchestrator enforces that the Designer agent has no access to mutating tools. Its tool schema omits Write, Edit, and mutating Bash commands entirely. This is a hard constraint, not a prompt instruction — the agent physically cannot produce a file-write tool call.

Agent framework architecture

The Designer is implemented as an AgentDefinition<DesignerState> driven by the framework's runAgent() step loop. The framework provides:

  • Channel abstraction — ReplChannel for terminal I/O, TestChannel for scripted auto-approve in tests
  • Typed message protocol — AgentMessage envelopes with kinds: emit, progress, gate, checkpoint, done, error
  • Atomic checkpoints — temp-file + rename after every step, with heartbeat and run-lock for crash detection
  • Artifact persistence — step outputs written to ~/.insrc/agents/<runId>/artifacts/, validated on resume
  • CLI management — insrc agent list|resume|discard|prune commands for run lifecycle

Settings

Design templates

The Designer ships with out-of-the-box templates for Markdown and HTML output. Templates define the document skeleton — section structure, heading hierarchy, CSS (for HTML), and placeholder slots that the Designer fills during the per-requirement iteration.

TemplateFormatWhen used
default-md Markdown Default for REPL / inline display. Sections as ## headings.
default-html HTML When --html flag is set, or user requests HTML. Dark-theme styled document with ToC sidebar.

Template structure

Template override

Users can override the built-in templates by placing custom templates in the project's .insrc/templates/ directory. Custom templates follow the same DesignTemplate schema. The Designer resolves templates in order:

  1. User-specified template via --template <id> flag
  2. Project-level template in .insrc/templates/design-*.{md,html}
  3. Built-in default-md or default-html
// src/agent/tasks/designer/templates.ts — template resolution
function resolveTemplate(opts: {
  format?: 'markdown' | 'html';
  templateId?: string;
  repoPath?: string;
}): DesignTemplate {
  // 1. Explicit template ID — check builtins, then project templates
  if (opts.templateId) {
    if (opts.templateId in BUILTIN_TEMPLATES) return BUILTIN_TEMPLATES[opts.templateId]!;
    if (opts.repoPath) {
      const custom = loadProjectTemplate(opts.repoPath, opts.templateId);
      if (custom) return custom;
    }
    // Degrade gracefully — fall through to default
  }

  const format = opts.format ?? 'markdown';

  // 2. Project-level override
  if (opts.repoPath) {
    const ext = format === 'html' ? 'html' : 'md';
    const custom = loadProjectTemplate(opts.repoPath, `design-default.${ext}`);
    if (custom) return custom;
  }

  // 3. Built-in default
  return format === 'html' ? DEFAULT_HTML : DEFAULT_MD;
}

Input

The Designer accepts two inputs:

InputSourceRequiredDescription

L2 carry-forward: If the user ran a requirements task earlier in the session, the compressed requirements are available in L2 under the [requirements] tag. The Designer detects this automatically and uses it as the requirementsDoc input — no file path needed.

Flow

The Designer follows an iterative workflow with user validation gates. Every significant decision is surfaced to the user before proceeding, ensuring the final design reflects the user's intent rather than the model's assumptions.

Input (DesignerInput: message + codeContext + template + session) | v extract-requirements LOCAL extract + CLAUDE enhance | checkpoint → state.json v validate-requirements GATE User confirms / edits / rejects | (reject: re-extract with guidance) | (edit: Claude re-extract with feedback, max 3 rounds) v parse-requirements String → ParsedRequirement[] → RequirementTodo[] | artifact → requirements.md v pick-next-requirement Select next todo with state='pending' | (none left → jump to assemble) v +———— Per-requirement loop ————+ | | v | sketch LOCAL write + CLAUDE review | | artifact → sketch-N.md | v | validate-sketch GATE approve/edit/reject/skip | | (skip → pick-next) | | (edit → re-sketch) | v | detail LOCAL write detailed section | | artifact → detail-N.md | v | validate-detail GATE approve/edit/reject/skip | | (reject → reset to pending, re-sketch) | (approve → mark done) | v | +—— back to pick-next-requirement —+ | v assemble Template + all done todos → final document artifact → assembled.{md|html} emit → result output
Checkpoint after every step: Each step writes an atomic checkpoint to ~/.insrc/agents/<runId>/state.json. On crash or user abort, the run can be resumed from the last checkpoint via insrc agent resume <runId> or the REPL's automatic crash detection. Artifacts are validated on resume — missing artifacts trigger rollback to the step that produced them.

Step: extract-requirements

The first step takes the user's message (and optional requirements doc) and produces a concise, numbered list of functional and system requirements. This is not a full requirements document — it is a short, scannable list the user can quickly approve, edit, or reject.

This step checks daemon availability (for codebase analysis) before proceeding, then runs two LLM calls in sequence:

Local extraction

The local model (qwen3-coder) receives the user message plus codebase context and produces the initial requirements list. If a requirements doc is provided, the local model extracts and condenses rather than generating from scratch. 

// src/agent/tasks/designer/prompts.ts — requirements extraction (Stage 1: local)
const REQ_EXTRACT_SYSTEM = `You are a requirements analyst for a software project.
You have the user's request and relevant code context from the codebase knowledge graph.

Your task: produce a concise numbered list of functional and system requirements.
Each requirement should be:
- One sentence, testable, specific
- Grounded in the codebase context (reference existing entities where relevant)
- Classified as FUNCTIONAL or SYSTEM

If a requirements document is provided, extract and condense it — do not regenerate.
If no requirements document is provided, derive requirements from the user's request
and the codebase context.

Output format:
1. [FUNCTIONAL] Requirement statement — references: entity_name, file.ts
2. [SYSTEM] Requirement statement — references: entity_name
...

Keep the list to 5–15 items. Merge overlapping requirements. Surface ambiguities
as open questions at the end (max 3).`;

Claude enhancement

Claude receives the local model's list plus the same context and sharpens it: filling gaps, resolving obvious ambiguities, flagging conflicts with existing architecture. The output remains a concise list — not a full document yet. 

// src/agent/tasks/designer/prompts.ts — requirements extraction (Stage 2: Claude)
const REQ_ENHANCE_SYSTEM = `You are a senior engineer validating a requirements list
produced by a local model. You have the user's original request, the local model's
list, and the relevant code context.

Your job is to VALIDATE, not extend. The granularity of the input list is correct —
do not split requirements into finer-grained items.

Rules:
1. ENHANCE IN PLACE — If a requirement is missing details, add those details to
   that existing requirement. Do not create a new item for the missing detail.
2. ONLY ADD if the user's original request contains a requirement that the local
   model completely missed. This should be rare (1–2 items at most).
3. Do not change the granularity. If the input has one item covering "markdown
   serialization", keep it as one item — do not split into read/write/partial-update.
4. Remove exact duplicates (keep the better-worded version).
5. Flag conflicts with existing architecture.
6. Keep open questions to genuine ambiguities (max 3).

Output the same numbered format. Keep it concise — this list will be shown to the
user for validation, not stored as a final document.`;

Step: validate-requirements

The enhanced requirements list is presented to the user via a GATE. The gate uses the framework's ctx.gate() method, which sends a GatePayload message and blocks until the channel returns a ReplyPayload.

  • Approve — proceed to parse-requirements
  • Edit — Claude re-extracts with feedback injected (max 3 rounds per MAX_EDIT_ROUNDS)
  • Reject — re-extract from scratch with rejection reason as guidance (resets edit counter)
// src/agent/tasks/designer/steps.ts — validate-requirements step (agent framework path)
const reply = await ctx.gate({
  stage: 'requirements',
  title: 'Requirements Validation',
  content: state.enhancedRequirements,
  actions: [
    { name: 'approve', label: 'Approve' },
    { name: 'edit', label: 'Edit', hint: '<feedback>' },
    { name: 'reject', label: 'Reject', hint: '<reason>' },
  ],
});
// src/agent/framework/types.ts — framework gate types
interface GatePayload {
  gateId: string;
  stage: string;
  title: string;
  content: string;
  actions: GateAction[];
  context?: Record<string, unknown>;
}

interface GateAction {
  name: string;       // e.g. 'approve', 'edit', 'reject', 'skip'
  label: string;      // display label
  hint?: string;      // e.g. '<feedback>'
  needsInput?: boolean;
}

interface ReplyPayload {
  gateId: string;
  action: string;
  feedback?: string;
}
Two gate implementations: The framework types above are used by the agent framework path (steps.ts). The async generator path uses its own simpler ValidationGate and GateResponse types defined in src/agent/tasks/designer/validation.ts.

Step: parse-requirements

Parses the enhanced requirements text into structured ParsedRequirement[] and creates one RequirementTodo per requirement. Writes the requirements.md artifact. If parsing fails (0 items), falls back to a minimal document with the raw enhanced text and terminates. 

// src/agent/tasks/designer/types.ts — requirement todo
interface RequirementTodo {
  /** Requirement index (1-based, matching the validated list) */
  index: number;
  /** The requirement statement */
  statement: string;
  /** Requirement type — clarifications are skipped during design */
  type: 'functional' | 'system' | 'clarification';
  /** Referenced entities from the requirements extraction */
  references: string[];
  /** Current state */
  state: 'pending' | 'sketching' | 'sketch-reviewing' | 'sketch-validated' | 'detailing' | 'detail-reviewing' | 'done' | 'skipped';
  /** Sketch output (populated after Step 3a) */
  sketch?: RequirementSketch;
  /** Detailed section output (populated after Step 3b) */
  detail?: string;
}
Clarification detection: Requirements that are questions (ending with ? or starting with question words like 'Should', 'What', 'How') are auto-classified as clarification and skipped during design. They are noted in progress output for user awareness.

Step: pick-next-requirement

Finds the next todo with state === 'pending'. If all todos are done or skipped, transitions to assemble. Otherwise sets currentTodoIndex and transitions to sketch. This step contains no LLM calls — it is pure state routing.

Step: sketch

For each requirement, the Designer produces a short sketch that answers: how will this be implemented? The local model writes the sketch, then Claude reviews and fixes it in the same step. The sketch is grounded in the actual codebase through two analyses:

LLM-driven search planning

Before querying the codebase, the local model analyzes the requirement and generates 3–6 targeted search queries with category filters. This replaces the naive approach of vector-searching the raw requirement text. 

// src/agent/tasks/designer/search-planner.ts
interface PlannedSearch {
  query: string;           // Short, focused (2-6 words)

  category: string;        // "interfaces", "configs", "schemas", etc.
  limit: number;           // 5-15 per query
}

Codebase analysis

The local model searches the current project's knowledge graph for existing modules, patterns, and utilities that can be reused or extended for this requirement. This prevents the design from proposing new code that duplicates what already exists.

Cross-project analysis

Services and external modules may be available in other indexed projects in the dependency closure. The Designer queries the graph across all session.closureRepos — not just the current project. This surfaces reusable services, shared libraries, and API contracts that the design should integrate with rather than rebuild.

Summary implementation flow

The sketch concludes with a summary flow: a concise description of how data moves through existing and proposed components to satisfy this requirement. 

// src/agent/tasks/designer/types.ts — per-requirement sketch
interface RequirementSketch {
  /** Requirement index */
  index: number;
  /** Existing modules/entities that can be reused */
  reusable: Array<{
    entity: string;      // entity ID or name
    project: string;     // which indexed project it's from
    relevance: string;   // one-line description of how it's relevant
  }>;
  /** Proposed new components (if reuse is insufficient) */
  proposed: Array<{
    name: string;
    kind: 'function' | 'class' | 'module' | 'interface' | 'type';
    file: string;        // proposed file location
    purpose: string;
  }>;
  /** Summary implementation flow */
  summaryFlow: string;
  /** Open concerns surfaced during analysis */
  concerns: string[];
}

// Stage 1: local model sketch prompt
const SKETCH_SYSTEM = `You are a software architect analyzing a single requirement
against an existing codebase. You have the requirement, the full codebase context
(including entities from other indexed projects in the dependency closure), and the
approved requirements list for overall context.

Your task for this ONE requirement:
1. REUSABLE MODULES — Search the provided context for existing modules, functions,
   types, or services that can be reused or extended. Include entities from other
   indexed projects if relevant (name the project). Do not propose new code if
   existing code already does the job.
2. PROPOSED COMPONENTS — If reuse is insufficient, propose new components. Name
   them, specify where they belong in the file tree, and describe their purpose.
3. SUMMARY FLOW — Describe how data moves through existing + proposed components
   to satisfy this requirement. Keep it to 3-5 sentences. Name the entities.
4. CONCERNS — Note any integration risks, performance concerns, or dependencies
   that the user should be aware of.

Be concrete. Name files, functions, types. Do not be vague.

Output as structured sections:

## Reusable Modules
- [project] entity_name — file:line — relevance

## Proposed Components
- ComponentName (kind) — proposed/file/path.ts — purpose

## Summary Flow
Narrative description...

## Concerns
- Concern description`;

Claude review (within sketch step)

Before presenting the sketch to the user, Claude reviews and fixes the local model's output — this happens within the same sketch step. The local model may misidentify entities, propose reuse of unsuitable modules, or produce an inaccurate summary flow. Claude corrects these issues so the user sees a high-quality, validated sketch at the gate — not a raw local draft. Artifact: sketch-N.md 

// src/agent/tasks/designer/prompts.ts — Claude sketch review
const SKETCH_REVIEW_SYSTEM = `You are a senior engineer reviewing a design sketch produced
by a local model for a single requirement. You have:
- The requirement statement
- The local model's sketch (reusable modules, proposed components, summary flow)
- The codebase context (entity bodies, signatures, types)
- The full requirements list for cross-requirement context

Your task:
1. VALIDATE REUSE — Check that each "reusable" entity actually exists in the context
   and is genuinely relevant. Remove false positives. Add missed reuse opportunities.
2. VALIDATE PROPOSALS — Check that proposed new components don't duplicate existing code.
   Verify proposed file locations match the project's conventions.
3. FIX SUMMARY FLOW — Correct any inaccurate entity references, wrong call chains, or
   missing steps. The flow should be traceable through the actual codebase.
4. FLAG CONCERNS — Add any integration risks or cross-requirement conflicts the local
   model missed.

Output the corrected sketch in the same format. Mark corrections with [FIXED] and
additions with [ADDED] so the user can see what changed.`;
// src/agent/tasks/designer/sketch.ts — Claude sketch review
async function reviewSketch(
  sketch: RequirementSketch,
  todo: RequirementTodo,
  allRequirements: ParsedRequirement[],
  input: DesignerInput,
  claudeProvider: LLMProvider,
): Promise<RequirementSketch> {
  const reqListContext = formatRequirementsList(allRequirements);

  const messages: LLMMessage[] = [
    { role: 'system', content: SKETCH_REVIEW_SYSTEM },
    { role: 'user', content: `## Requirement\n${todo.statement}` },
    { role: 'user', content: `## Local Sketch\n${formatSketch(sketch)}` },
    { role: 'user', content: `## All Requirements\n${reqListContext}` },
    ...(input.codeContext
      ? [{ role: 'user', content: `## Code Context\n${input.codeContext}` }]
      : []),
  ];

  const response = await claudeProvider.complete(messages, {
    maxTokens: 2048,
    temperature: 0.2,
  });

  return parseSketch(response.text, todo.index);
}

Step: validate-sketch

The Claude-reviewed sketch is presented to the user via a GATE. Because Claude has already validated and fixed the sketch within the sketch step, the user sees a high-quality proposal — not a raw local draft.

This gate is critical: it is where the user corrects the approach before the detailed design is written. Catching a wrong direction here saves significant LLM cost and user time.

  • Approve — mark sketch-validated, transition to detail
  • Edit — Claude re-sketches with feedback (max 3 rounds), stays on validate-sketch
  • Reject — mark skipped, transition to pick-next-requirement
  • Skip — mark skipped, transition to pick-next-requirement

Step: detail

Once the sketch is validated, the local model generates the full detailed design section for this requirement. Artifact: detail-N.md. The section includes:

  • Interface contracts (TypeScript signatures or pseudocode)
  • Data flow with concrete entity references
  • Integration points with existing code (from the sketch's reuse analysis)
  • Migration notes (if modifying existing entities)
  • Risk callouts (from the sketch's concerns)
// src/agent/tasks/designer/prompts.ts — detailed section generation
const DETAIL_SYSTEM = `You are a senior engineer writing one section of a design document.
You have:
- The requirement statement
- The approved sketch (reusable modules, proposed components, summary flow)
- The codebase context (entity bodies and signatures)
- The overall requirements list (for cross-requirement context)

Write a detailed design section covering:
1. INTERFACE CONTRACTS — TypeScript signatures or pseudocode for new/modified entities
2. DATA FLOW — Concrete path through existing and new components, with entity names
3. INTEGRATION POINTS — How this connects to existing code. Name the files and functions.
4. MIGRATION NOTES — If existing entities are modified, describe the change and impact.
5. RISKS — Anything from the sketch concerns that needs explicit handling.

Output as markdown. Use code blocks for interface definitions. Be specific — name every
entity, file, and function. Do not repeat the requirement statement as a preamble.`;
Note: Unlike the sketch step, the detail step currently uses only the local model. Claude review of details happens only when the user provides edit feedback at the validate-detail gate (via reDetailWithFeedback).

Claude review (on edit feedback only)

Claude reviews and revises the detail only when the user provides edit feedback at the validate-detail gate. This saves Claude API calls for the common case where the local model's detail is acceptable. 

// src/agent/tasks/designer/prompts.ts — Claude detail review
const DETAIL_REVIEW_SYSTEM = `You are a senior engineer reviewing a detailed design section
produced by a local model. You have:
- The requirement statement and approved sketch
- The local model's detailed section
- The codebase context
- Previously completed requirement sections (for cross-requirement consistency)

Your task:
1. VALIDATE INTERFACES — Check that TypeScript signatures are syntactically correct
   and compatible with existing types in the codebase.
2. VALIDATE INTEGRATION — Verify that named files, functions, and call chains exist
   in the context. Fix incorrect references.
3. CROSS-REQUIREMENT CONSISTENCY — Check for conflicts with previously completed
   sections (duplicate entity names, incompatible interfaces, contradictory data flows).
4. COMPLETENESS — Ensure migration notes cover all modified entities. Ensure risks
   from the sketch concerns are addressed.

Output the corrected section. Mark corrections with [FIXED].`;
// src/agent/tasks/designer/detail.ts — Claude detail review
async function reviewDetail(
  detail: string,
  todo: RequirementTodo,
  allTodos: RequirementTodo[],
  input: DesignerInput,
  claudeProvider: LLMProvider,
): Promise<string> {
  const messages: LLMMessage[] = [
    { role: 'system', content: DETAIL_REVIEW_SYSTEM },
    { role: 'user', content: `## Requirement\n${todo.statement}` },
    { role: 'user', content: `## Approved Sketch\n${formatSketch(todo.sketch)}` },
    { role: 'user', content: `## Local Detail\n${detail}` },
  ];

  if (input.codeContext) {
    messages.push({ role: 'user', content: `## Code Context\n${input.codeContext}` });
  }

  const history = compressHistory(allTodos);
  if (history) {
    messages.push({ role: 'user', content: `## Design History\n${history}` });
  }

  const response = await claudeProvider.complete(messages, {
    maxTokens: 4096,
    temperature: 0.2,
  });

  return response.text;
}

Step: validate-detail

The detailed section is presented to the user via a GATE.

  • Approve — mark todo done, transition to pick-next-requirement
  • Edit — Claude revises the detail with feedback (max 3 rounds), stays on validate-detail
  • Reject — reset todo to pending (clears sketch and detail), transition to pick-next-requirement which re-selects it for a fresh sketch
  • Skip — mark todo skipped, transition to pick-next-requirement

Step: assemble

Once all requirement todos are done or skipped, the Designer assembles the final document from the template skeleton plus all validated detail sections. Derives a title from the user's input message and writes the assembled.{md|html} artifact. Emits the full document via ctx.emit() and sets assembledOutput and summary on the final state.

Validation Gates

Gate mechanics

Validation gates are the Designer's primary mechanism for keeping the user in the loop. Each gate uses ctx.gate() which sends a GatePayload message through the Channel abstraction and blocks until a ReplyPayload is returned. The channel implementation determines how the gate is presented:

  • ReplChannel — renders the gate to terminal with ANSI formatting, prompts via readline
  • TestChannel — returns pre-scripted replies (for automated tests and CI)
// src/agent/tasks/designer/validation.ts — gate rendering (async generator path)
function renderGate(gate: ValidationGate): string {
  // Renders title, content, and action hints: [a]pprove  [e]dit <feedback>  [r]eject  [s]kip

Codebase Analysis

The sketch step (4a) relies on thorough codebase analysis to ground the design in reality. This analysis happens in two scopes.

Local project analysis

The Designer queries the current project's knowledge graph for entities that match the requirement's domain: 

// src/agent/tasks/designer/sketch.ts — local codebase analysis
async function analyzeLocalCodebase(
  provider: ReturnType<typeof createDaemonContextProvider>,
  currentRepo: string,
  searches: PlannedSearch[],
): Promise<AnalysisEntity[]> {
  // Execute all planned searches in parallel
  const allHits = await Promise.all(
    searches.map(s => provider.search(s.query, s.limit, s.filter)),
  );

  // Deduplicate by entity ID, keep first occurrence
  const seen = new Set<string>();
  const localHits: Entity[] = [];
  for (const hits of allHits) {
    for (const h of hits) {
      if (h.repo === currentRepo && !seen.has(h.id)) {
        seen.add(h.id);
        localHits.push(h);
      }
    }
  }

  // Expand top 5 for neighbour context (keep budget manageable)
  const toExpand = localHits.slice(0, 5);
  const expanded = await Promise.all(
    toExpand.map(async h => ({
      entity: h,
      neighbours: await provider.expand(h.id),
    })),
  );

  // Include remaining hits without expansion
  const rest = localHits.slice(5).map(h => ({
    entity: h,
    neighbours: { callers: [], callees: [] },
  }));

  return [...expanded, ...rest];
}

Cross-project analysis

Services, shared libraries, and external modules may live in other indexed projects within the dependency closure. The Designer queries across all closure repos to find reusable components the design should integrate with rather than rebuild.

Context Compression

As the Designer iterates through requirements, completed designs accumulate. Without compression, by requirement 10 the context would include 9 full detail sections — blowing the token budget. The compressHistory() function in src/agent/tasks/designer/context.ts solves this.

Budget

Approximately 500–800 characters per completed requirement.

What is extracted

  • Type/interface/enum signatures — compact form, no body
  • Function signatures — name, parameters, return type
  • Key design decisions — one-line summaries

Where it is used

The compressed history is injected into both the sketch and detail steps so that later requirements have awareness of earlier designs. This enables cross-requirement consistency (e.g., referencing interfaces defined in requirement 2 when designing requirement 7) without repeating full detail sections.

Review Intent

The review intent is a separate workflow from the design flow above. It critiques existing code rather than designing new code. The review workflow does not use the iterative todo-based pattern — it is a single-pass Claude analysis.

Context assembly

The local model uses read-only tools to build the review payload:

  • If a diff is in scope (staged changes, a PR, a specific commit): git diff or git show — the diff is the primary input
  • Bodies of touched entities — fetched by matching diff hunks to indexed entity ranges
  • Signatures of first-degree callers and callees — for call-site compatibility context
  • If no diff (reviewing an existing file or function): the entity body + its immediate neighbours' signatures

Claude review

Output schema

The review intent does not use a separate ReviewResult interface. Instead, it returns a DesignerResult with kind: 'review' and empty requirements and sketches arrays. The review content is in the output field.

Sonnet handles most reviews. The user can prefix with @opus to escalate for large diffs (> 500 lines), security audits, performance-critical paths, or cross-repo changes. Cost estimate shown before any Opus call.

Error handling

FailureBehaviour
Local sketch fails (Ollama timeout) Step throws — framework writes error checkpoint with status: 'failed'. Run can be resumed after fixing Ollama.

Run locked by another process acquireLock() uses O_CREAT | O_EXCL. If lock held by dead process (stale lock), it is broken and retried once.

Agent CLI

Agent runs are managed through insrc agent subcommands (registered in src/cli/commands/agent.ts):

CommandDescription
insrc agent list List all runs with status, agent ID, repo, and last update time. Detects crashed runs (stale heartbeat / dead PID).

The REPL integration (gated by INSRC_NEW_AGENT env var) automatically checks for active or crashed designer runs when the user issues a design or requirements command, and offers to resume.

Open Questions

  1. Validation gate UX in REPL vs CLI — Resolved: ReplChannel handles REPL gates (readline prompt), TestChannel handles batch/CI (scripted replies). CLI management via insrc agent list|resume|discard|prune. Feature-flagged via INSRC_NEW_AGENT.
  2. Parallelism within requirements — Some requirements may be independent and could be sketched/detailed in parallel. Should the Designer detect independence (via graph overlap analysis) and parallelise, or always iterate sequentially? (Currently sequential — simpler checkpoint model.)
  3. Cross-project index freshness — If another indexed project has changed since last indexing, the Designer's cross-project analysis may surface stale signatures. Should the Designer check index freshness and warn, or trigger a re-index?
  4. Design versioning — When the user re-runs the Designer on the same topic, should it produce a diff against the previous design (stored in L2), or a fresh document? (inputHash in RunMeta could enable dedup detection.)
  5. Structured output enforcement — Resolved: Requirements are extracted post-hoc from markdown via parseRequirementsList() in the parse-requirements step. Sketches use structured RequirementSketch type. SEARCH_PLAN_SYSTEM is the 8th prompt in prompts.ts, used for LLM-driven search planning.
  6. Version migration — The framework supports migrate(state, fromVersion) for checkpoint schema evolution, but no migrations exist yet (version 1). Define migration strategy before bumping to version 2.
  7. Claude detail review — Currently the detail step uses only the local model. Claude only reviews on edit feedback. This is an intentional tradeoff: it saves API calls for the common case where the local model's detail is acceptable. Should Claude review all details for higher quality, at the cost of more API calls?
