Agent Design Document

insrc coding agent — v0.1 — March 2026

Overview

The insrc agent is a local-first hybrid coding assistant. The majority of tasks are handled autonomously by a local LLM running via Ollama — fast, private, and free. For tasks that require deeper reasoning, architectural judgment, or cross-repo design decisions, the agent transparently escalates to Claude.

Both providers draw context from the same source: the live Code Knowledge Graph maintained by the indexer daemon (see indexer.html). The agent never sends raw file dumps to any LLM — it assembles structured, minimal context from graph queries.

Indexer dependency: The agent requires the daemon to be running. On startup it connects via the Unix socket, resolves the dependency closure of the active repo, and holds that closure for the session duration.

LLM Providers

LOCAL qwen3-coder:latest

  • Runs via Ollama — no API key, no network
  • Code-specialized model, fast response
  • Handles the majority of day-to-day tasks
  • Auto-installed on daemon startup if missing
  • Privacy default — code never leaves the machine

CLOUD Claude (Anthropic)

  • Invoked only for complex / design tasks
  • Requires ANTHROPIC_API_KEY
  • Always announced to user before invocation
  • Receives structured graph context, not raw files
  • 200K token context — handles large result sets

Task Routing

Tasks are classified by scope and reasoning depth. Local handles everything it can; Claude is invoked only when the task genuinely exceeds local capability.

TaskProviderRouting Signal

Explain this function / classLOCALScope ≤ 1 file
Write unit tests for a functionLOCALSingle entity target
Fix a specific bugLOCALError + file pointed at
Rename / refactor within a fileLOCALSingle file scope
Summarize what a file doesLOCALSingle file scope
Find callers / usagesLOCALPure graph query — no LLM needed
Multi-file refactorCLAUDEEntities span > 2 files
Architecture / design questionCLAUDETask kind = design

Explain a complex interactionCLAUDEGraph depth > 2 hops, multi-repo

Graph queries are free: Tasks like "find all callers of X" or "what does this repo depend on" are answered directly from SurrealDB with no LLM involved. The agent resolves these as pure graph traversals and formats the result for the user.

Escalation Modes

1. Explicit — user-directed

User prefixes a message with @claude to bypass routing and always use Claude:

> @claude how should I restructure the payment service to support multiple providers?

2. Agent-suggested — local agent defers

When the local agent determines a task exceeds its confidence or scope, it pauses and asks rather than producing a low-quality answer: 

Local agent: This refactor touches 6 files across 2 repos (order-service, shared-lib).
             Would you like me to ask Claude for a plan? [y/N]

3. Automatic — threshold-based

The agent escalates automatically when any of these signals fire:

SignalThreshold
Files involved> 3 files
Repos involved> 1 repo
Context tokens needed> 8K tokens
Task kinddesign, review, document
Always transparent: The agent never silently calls Claude. Every escalation — whether suggested or automatic — is announced in the output before the API call is made. This keeps the user aware of cost and of code leaving the machine.

Context Assembly

Neither provider receives raw file dumps. The agent assembles a structured context package from the Code Knowledge Graph for each task. This keeps prompts tight, deterministic, and cost-efficient.

Local context (qwen3-coder)

Small, focused: 3–10 entities, direct neighbors only.

// Assembled by agent/context.ts
{
  task: "write unit tests for OrderService.create",
  entities: [
    { kind: "Function", name: "create", file: "src/services/order.ts", body: "..." },
    { kind: "Interface", name: "IOrderRepo", file: "shared-lib/repos/order.ts", body: "..." },
    { kind: "Type", name: "Order", file: "src/types/order.ts", body: "..." }
  ],
  relations: [
    "OrderService.create CALLS IOrderRepo.save",
    "OrderService.create CALLS PaymentService.charge"
  ]
}

Claude context (design / cross-repo)

Broader but still structured: dependency closure summary, key entity graph, relevant relations. Raw source is only included for the directly relevant entities. 

System: You are a senior engineer. Here is the relevant code graph context:

[Active repo]: order-service
[Dependency closure]: order-service, shared-lib, payment-service

[Key Entities]
- OrderService.create()     src/services/order.ts:42
- PaymentService.charge()   payment-service/src/service.ts:18  (external repo)
- IOrderRepo.save()         shared-lib/repos/order.ts:9        (external repo)
- PgOrderRepo               shared-lib/repos/pg-order.ts:5     implements IOrderRepo

[Relations]
OrderService.create  → CALLS      → PaymentService.charge
OrderService.create  → CALLS      → IOrderRepo.save
IOrderRepo           ← IMPLEMENTS ← PgOrderRepo

[Source: OrderService.create]
<body of function>

[User question]
How should I restructure this to support multiple payment providers?
Entity bodies are included only for the directly relevant entities. Related entities are described by name, file, and relation type only — not full source. This keeps Claude prompts well under the token budget even for large dependency closures.

Provider Interface

Both providers implement a common LLMProvider interface so the agent layer is decoupled from the underlying model. Task routing selects the provider; the agent code does not branch on provider identity. 

// shared/types.ts
interface LLMMessage {
  role: 'system' | 'user' | 'assistant';
  content: string;
}

interface LLMProvider {
  complete(messages: LLMMessage[], opts?: { maxTokens?: number }): Promise<string>;
  stream(messages: LLMMessage[], opts?: { maxTokens?: number }): AsyncIterable<string>;
}

// agent/providers/ollama.ts
class OllamaProvider implements LLMProvider {
  constructor(private model = 'qwen3-coder:latest') {}
  // uses ollama npm package
}

// agent/providers/claude.ts
class ClaudeProvider implements LLMProvider {
  constructor(private model = 'claude-sonnet-4-6') {}
  // uses @anthropic-ai/sdk
}

// agent/router.ts
function selectProvider(task: Task, config: AgentConfig): LLMProvider {
  if (task.explicit === 'claude') return claudeProvider;
  if (needsEscalation(task))     return claudeProvider;
  return ollamaProvider;
}

Session Management

The agent maintains per-session conversation history in two tiers: raw turns stored in SurrealDB during the session (for semantic retrieval), and a session summary promoted to persistent storage at session end (for cross-session continuity).

PropertyValue
Raw turn storageSurrealDB conversation_turn — session duration only, deleted on close
Session summaryPersisted to conversation_session — retained across sessions, pruned by TTL
Context window managementLayered memory model — see below
Provider switchingHistory is provider-agnostic; can switch mid-session
Dependency closureResolved once at session start, held for session duration
When the agent escalates to Claude mid-session, it passes L1 + L2 (rolling summary) + the assembled task context only. The full turn history is not forwarded — it adds noise without value for design-level questions.

Context Window & Memory

Ollama's context window is 64K tokens. This sounds generous but is quickly consumed by system prompt, conversation history, code context, and response headroom all competing for the same budget. A naive sliding-window approach discards turns arbitrarily and loses earlier context permanently. The layered memory model below allocates the budget deliberately, using recency for near-term turns and semantic retrieval for older history.

The index is the memory. The SurrealDB Code Knowledge Graph is the agent's persistent long-term memory for code structure. The context window only needs to hold what is immediately relevant to the current turn — everything else is re-queryable on demand.

Layer Budget

┌──────────────────────────────────────────────────────┐  64K total
│  L1  System context        ~1K   fixed               │  always present
│  L2  Session summary       ~3K   rolling             │  compressed older turns
│  L3a Recent turns          ~4K   recency-weighted    │  last 5 turns, tapered verbatim
│  L3b Semantic history      ~4K   retrieved per turn  │  relevant older turns via embedding
│  L4  Task context         ~16K   dynamic per turn    │  graph-fetched, not accumulated
│  L5  Response headroom     ~8K   reserved            │  never used for input
│  L6  Overflow buffer      ~28K   elastic             │  absorbed by L3/L4 as needed
└──────────────────────────────────────────────────────┘
LayerBudgetContentsEviction policy
L1 — System ~1K Agent persona, active repo name, dependency closure repo list (names only) Never evicted
L2 — Summary ~3K Rolling compressed summary of older conversation turns Regenerated in place when L3a overflows; never grows beyond budget
L3a — Recent turns ~4K Last 5 turns verbatim, recency-weighted (newest = full, oldest = truncated) Oldest turn compressed into L2 when L3a overflows
L3b — Semantic history ~4K Top-K older turns retrieved by embedding similarity to current message Re-fetched every turn; not accumulated. Skipped for first 5 turns.
L4 — Task context ~16K Entities + relations fetched from code graph for current turn Replaced every turn — not accumulated across turns
L5 — Response ~8K Reserved for model output Hard reservation — input never fills this

Recency-Weighted Recent Turns (L3a)

Near-term turns are the most contextually connected. Rather than a flat window, L3a weights turns by age — the most recent turn is included in full, older turns are progressively truncated:

Turn ageInclusion
Turn −1 (most recent)Full text
Turn −275% — trim assistant response tail
Turn −350% — user message + first half of response
Turn −425% — user message + first paragraph of response
Turn −5User message only
Turn −6+Evicted to L2 (summarized) or L3b (embedded)

Semantic History Retrieval (L3b)

Every turn is embedded and stored in SurrealDB. When assembling context, the current user message is used to retrieve the most semantically relevant older turns — regardless of when they occurred. A turn from 30 exchanges ago resurfaces if it is relevant to the current question.

History embedding is skipped for the first 5 turns of a session (pure L3a recency is sufficient and cheaper when history is short).

Storage — per turn (end of turn, non-blocking)

// memory.ts — called after assistant response is streamed
async function storeTurn(session: Session, turn: ConversationTurn): Promise<void> {
  if (session.turnIndex < 5) return;  // skip embedding for early turns

  const vec = await embed(turn.userMessage);  // embed user message only

  await db.query(`
    INSERT INTO conversation_turn {
      session:   $session,
      idx:       $idx,
      user:      $user,
      assistant: $assistant,
      embedding: $vec,
      entities:  $entities,   -- graph entity IDs referenced in this turn
      createdAt: time::now()
    }
  `, { session: session.id, idx: session.turnIndex, ...turn, vec });
}

Retrieval — per turn (start of turn, on critical path ~5ms)

// memory.ts — called before assembling context
async function retrieveSemanticHistory(
  session: Session,
  vec: number[]
): Promise<ConversationTurn[]> {
  if (session.turnIndex < 5) return [];  // nothing stored yet

  return db.query(`
    SELECT idx, user, assistant,
           vector::similarity::cosine(embedding, $vec) AS score
    FROM conversation_turn
    WHERE session = $session
      AND idx < $currentIdx - 5   -- exclude what is already in L3a
      AND score > 0.70
    ORDER BY score DESC
    LIMIT 4
  `, { session: session.id, vec, currentIdx: session.turnIndex });
}

Rolling Summarization (L2)

When L3a overflows (turn −6 is evicted), that turn is compressed into the rolling L2 summary by the local model. The summary accumulates key decisions, entity names, and conclusions — not raw dialogue: 

// memory.ts
async function evictToSummary(session: Session): Promise<void> {
  const turn = session.recentTurns.pop();  // evict oldest (turn -6)

  const prompt = [
    { role: 'system', content:
        'Update the running summary to include the new exchange. ' +
        'Preserve: decisions made, file names, entity names, open questions. ' +
        'Be concise. Max 3 sentences total.' },
    { role: 'user', content:
        `Current summary:\n${session.summary}\n\nNew exchange:\n${formatTurn(turn)}` }
  ];

  session.summary = await ollamaProvider.complete(prompt, { maxTokens: 250 });
}

Dynamic Graph Retrieval (L4)

Task context is not accumulated — it is re-assembled from the code graph each turn. This keeps L4 fresh and avoids context drift from stale code that has since been edited by the indexer daemon. 

// context.ts
async function assembleContext(turn: string, session: Session): Promise<AssembledContext> {
  // Single embed call serves both code search (L4) and history retrieval (L3b)
  const vec = await embed(turn);

  const [codeHits, historyHits] = await Promise.all([
    db.searchCode(vec, { repos: session.closureRepos, limit: 8 }),   // L4
    retrieveSemanticHistory(session, vec),                            // L3b
  ]);

  const neighbors  = await db.neighbors(codeHits.map(h => h.id));
  const entities   = buildEntityContext(codeHits, neighbors, session.seenEntities);

  return fitToBudget({
    system:    session.systemContext,                    // L1
    summary:   session.summary,                          // L2
    recent:    weightedRecent(session.recentTurns),      // L3a
    semantic:  historyHits,                              // L3b
    code:      entities,                                 // L4
  });
}
Single embed per turn: The same embedding vector is reused for both the code graph search (L4) and the conversation history search (L3b). Only one Ollama call is needed.

Progressive Entity Disclosure (L4)

Entity bodies are large. Disclosure is staged to conserve L4 budget:

Entity stateWhat is included in L4
Never seen this sessionFull body (if budget allows) or signature + first 10 lines
Seen in last 5 turns (L3a)Signature only — body is still in recent history
Seen earlier, retrieved via L3bName + file + line reference only
Neighbor (not directly hit)Name + kind + relation type only

Token Budget Enforcement

Context assembly respects hard per-layer ceilings. L4 entities are dropped in reverse relevance order (lowest vector score first) if over budget. 

// budget.ts
const TOKEN_BUDGET = {
  system:   1_000,
  summary:  3_000,
  recent:   4_000,   // L3a
  semantic: 4_000,   // L3b
  code:    16_000,   // L4
  response: 8_000,   // reserved — never consumed by input
  total:   64_000,
} as const;

History Retention & Pruning

Raw turn embeddings are ephemeral. At session end, the final L2 summary is promoted to a persistent conversation_session record and raw turns are deleted. This gives cross-session continuity without unbounded storage growth.

Session close

// session.ts — on disconnect or explicit /exit
async function closeSession(session: Session): Promise<void> {
  // Promote summary to persistent store
  await db.query(`
    INSERT INTO conversation_session {
      id:        $sid,
      repo:      $repoId,
      summary:   $summary,
      entities:  $seenEntities,
      createdAt: time::now(),
      expiresAt: time::now() + 30d
    }
  `, { sid: session.id, repoId: session.repoId,
       summary: session.summary, seenEntities: [...session.seenEntities] });

  // Delete all raw turns — summary is the durable artifact
  await db.query(`DELETE conversation_turn WHERE session = $sid`, { sid: session.id });
}

Cross-session seeding

On session start, recent session summaries for the same repo seed the initial L2:

// session.ts — on session start
const priorSummaries = await db.query(`
  SELECT summary, createdAt FROM conversation_session
  WHERE repo = $repoId AND expiresAt > time::now()
  ORDER BY createdAt DESC
  LIMIT 3
`, { repoId });

session.summary = priorSummaries.map(s => s.summary).join('\n\n');

Nightly daemon maintenance job

// lifecycle.ts — runs daily via setInterval
async function pruneConversationHistory(): Promise<void> {
  // Delete expired summaries
  await db.query(`DELETE conversation_session WHERE expiresAt < time::now()`);

  // Cap at 20 summaries per repo — delete oldest beyond cap
  const repos = await db.query(`SELECT DISTINCT repo FROM conversation_session`);
  for (const { repo } of repos) {
    await db.query(`
      DELETE conversation_session
      WHERE repo = $repo
        AND id NOT IN (
          SELECT id FROM conversation_session
          WHERE repo = $repo
          ORDER BY createdAt DESC
          LIMIT 20
        )
    `, { repo });
  }
}
Claude escalation and memory: Claude has a 200K token context window, so budget pressure is much lower. When escalating, pass L1 + L2 (summary) + assembled task context (L4). Do not forward L3a/L3b turn history — it adds noise for design-level questions.

Model Bootstrap

On daemon startup, after the embedding model check, the agent model is also verified and pulled if missing. Both checks run sequentially before any index or agent jobs are processed. 

// lifecycle.ts
const MODELS = {
  embedding: 'qwen3-embedding:0.6b',
  agent:     'qwen3-coder:latest',
};

async function ensureModels(ollama: Ollama): Promise<void> {
  const { models } = await ollama.list();
  const installed = new Set(models.map(m => m.name));

  for (const [role, model] of Object.entries(MODELS)) {
    if (installed.has(model)) continue;
    log.info(`Pulling ${role} model: ${model}`);
    const stream = await ollama.pull({ model, stream: true });
    for await (const event of stream) {
      const pct = event.total ? Math.round((event.completed / event.total) * 100) : 0;
      setDaemonState({ modelPull: { role, model, pct } });
    }
    log.info(`${model} ready`);
  }
}

Implementation Phases

Phase 1 Local Agent (Ollama)

  • OllamaProvider with qwen3-coder:latest
  • Session REPL + layered memory model
  • Rolling summarization (L2) + sliding history (L3)
  • Dynamic graph retrieval + progressive disclosure (L4)
  • Token budget enforcement
  • Task handlers: explain, complete, test, edit
  • Model bootstrap in daemon lifecycle

Phase 2 Task Classification

  • Task kind inference from user message
  • Scope signals: file count, repo count, token estimate
  • Automatic escalation thresholds
  • Agent-suggested escalation prompts

Phase 3 Claude Integration

  • ClaudeProvider via @anthropic-ai/sdk
  • @claude explicit routing prefix
  • Structured context package for Claude
  • Escalation announcement before API call
  • Graceful degradation when no API key

Phase 4 Graph-Native Tasks

  • Pure graph answers (callers, dependents) with no LLM
  • Cross-repo impact formatted as structured report
  • Dependency closure visualization

Open Questions

  1. Summarization model — Rolling summarization (L2 eviction) uses qwen3-coder:latest for convenience, but a smaller/faster model (e.g. qwen3:0.6b) would be more efficient for this internal housekeeping task. Worth benchmarking summary quality vs. latency tradeoff.
  2. Token counting accuracy — Accurate token counting requires the model's tokenizer. Approximating with character count (~4 chars/token) is fast but imprecise. Consider using tiktoken-style counting or a lightweight tokenizer aligned to Qwen's vocabulary to avoid budget overruns.
  3. Task classification accuracy — Simple heuristics (file count, repo count) may misclassify. A lightweight classifier model or rule-based NLU might improve routing over time. Consider logging escalation decisions for later review.
  4. Claude model selection — Default is claude-sonnet-4-6 (balanced cost/quality). User-configurable; claude-opus-4-6 for highest quality, claude-haiku-4-5 for lowest cost.
  5. Session persistence — Currently ephemeral. Future option: persist session summaries to SurrealDB as a lightweight "conversation memory" per repo, giving the agent recall across sessions without full history replay.
  6. Streaming UX — Both providers support streaming. The REPL should stream tokens to the terminal in real time. Escalation announcement must appear before the first token from Claude.
