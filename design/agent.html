Agent Design Document

insrc coding agent — v0.2 — March 2026

Overview

The insrc agent is a local-first, multi-agent coding system. Four specialized agent personas — Designer, Developer, Tester, and Deployer — handle distinct phases of the software development lifecycle. An Orchestrator Pipeline classifies user intent, assembles context from the Code Knowledge Graph, and delegates to the right agent for each task.

The majority of work is handled by local LLMs running via Ollama — fast, private, and free. For tasks that require deeper reasoning, architectural judgment, or cross-repo design decisions, agents transparently escalate to Claude. Each persona has its own escalation behaviour, tool preferences, and prompt strategies, but all share the same infrastructure: LLM providers, tool execution pipeline, context assembly, and layered memory.

All agents draw context from the same source: the live Code Knowledge Graph maintained by the indexer daemon (see indexer.html). No agent sends raw file dumps to any LLM — they receive structured, minimal context assembled by the orchestrator from graph queries.

Indexer dependency: The agent system requires the daemon to be running. On startup the orchestrator connects via the Unix socket, resolves the dependency closure of the active repo, and holds that closure for the session duration.

Agent Architecture

Multi-agent model

The system is composed of an orchestrator and four agent personas. The orchestrator is not an LLM — it is a deterministic pipeline that classifies intent, selects the right persona, assembles context, and manages session state. Each persona is an LLM-backed agent with a distinct system prompt, tool set, escalation policy, and output format.

Orchestrator responsibilities

The orchestrator owns the session lifecycle and all cross-cutting concerns. Agents are stateless within a turn — they receive a fully assembled context package and return a result. They do not manage their own memory, session state, or plan progression.

Intent → agent routing

// orchestrator/router.ts
function selectAgent(intent: Intent): AgentPersona {
  switch (intent) {
    case 'requirements':
    case 'design':
    case 'review':
      return designerAgent;

    case 'plan':
    case 'implement':
    case 'refactor':
    case 'debug':
    case 'research':
    case 'document':
      return developerAgent;

    case 'test':
      return testerAgent;

    case 'deploy':
    case 'release':
    case 'infra':
      return deployerAgent;

    case 'graph':
      return null;  // orchestrator handles directly, no agent
  }
}

Agent lifecycle (per turn)

  1. Orchestrator classifies — local LLM returns intent + confidence + snippet via structured output
  2. Orchestrator assembles context — L1–L4 layers, plan step if active
  3. Orchestrator selects agent — intent → persona mapping
  4. Agent receives context package — system prompt (persona-specific) + assembled context + task description
  5. Agent executes — may involve multiple LLM calls, tool loops, retries, and internal escalation
  6. Agent returns result — structured output (diff, document, review, deployment manifest)
  7. Orchestrator commits result — writes files, updates plan state, evicts turns to L2, enqueues re-index
// orchestrator/pipeline.ts
interface AgentPersona {
  /** Persona identifier */
  readonly name: 'designer' | 'developer' | 'tester' | 'deployer';
  /** Intents this persona handles */
  readonly intents: Intent[];
  /** System prompt template — persona-specific framing, constraints, output format */
  readonly systemPrompt: string;
  /** Default LLM provider for this persona */
  readonly defaultProvider: 'local' | 'claude';
  /** Claude tier when Claude is used */
  readonly claudeTier: 'fast' | 'standard' | 'powerful';
  /** Execute a task with assembled context, return structured result */
  execute(task: AgentTask): Promise<AgentResult>;
}

interface AgentTask {
  /** Classified intent */
  intent: Intent;
  /** User message */
  message: string;
  /** Assembled context package (L1–L4) */
  context: AssembledContext;
  /** Active plan step, if any */
  planStep?: PlanStep;
  /** Attachments (text injected, binary forces Claude) */
  attachments?: Attachment[];
}

interface AgentResult {
  /** Output type */
  kind: 'diff' | 'document' | 'review' | 'plan' | 'deployment' | 'answer';
  /** The produced output */
  output: string;
  /** Files written or to be written */
  files?: string[];
  /** Whether Claude was used during execution */
  usedClaude: boolean;
  /** Execution summary for L2 */
  summary: string;
  /** Plan step state transition, if applicable */
  planTransition?: { stepId: string; state: 'done' | 'failed' | 'skipped'; note: string };
}
Agents are stateless across turns. An agent does not remember previous turns or maintain its own conversation history. All continuity is handled by the orchestrator via the layered memory model (L2 summary, L3a/L3b history). This keeps agents simple, testable, and replaceable — a persona can be swapped or upgraded without touching session logic.

LLM Providers

LOCAL qwen3-coder:latest

  • Runs via Ollama — no API key, no network
  • Code-specialized model, fast response
  • Handles the majority of day-to-day tasks
  • Auto-installed on daemon startup if missing
  • Privacy default — code never leaves the machine

CLOUD Claude (Anthropic)

  • Invoked only for complex / design tasks
  • Requires ANTHROPIC_API_KEY
  • Always announced to user before invocation
  • Receives structured graph context, not raw files
  • 200K token context — handles large result sets

Task Routing

Every user message is classified into one of the intents below before a provider is selected. Classification uses a combination of keyword signals, cursor context (active file, selected entity), and graph signals (entity count, repo count, graph depth). Local handles everything it can; Claude is invoked only when the task genuinely exceeds local capability.

Graph queries are free: Tasks like "find all callers of X" or "what does this repo depend on" are answered directly from Kuzu with no LLM involved. The agent resolves these as pure graph traversals and formats the result for the user.

Intent classification

Classification runs locally via the local LLM with structured output. The model receives the user message and session context, and returns a JSON classification with intent, confidence, and the relevant text snippet that drives each classification. This replaces brittle keyword matching — the LLM understands natural language like "the payment handler is returning 500s" (debug) without needing an explicit keyword hit.

Classification prompt

The orchestrator sends a single, focused classification call to the local model. The call uses structured output (JSON schema) to guarantee a parseable response. The model sees only the user message and lightweight session signals — no code context is assembled until after classification, since the intent determines what context to fetch. 

// orchestrator/classifier.ts — system prompt
const CLASSIFY_SYSTEM = `You are an intent classifier for a coding assistant.
Given a user message and session context, classify the primary intent.

Intents:
- requirements: clarify, capture, or refine what needs to be built
- design: architecture, API shape, module boundaries, tradeoff analysis
- review: critique correctness, style, security, or performance of code
- plan: break a feature into an ordered implementation checklist
- implement: write or edit code to satisfy a specification
- refactor: restructure code without changing observable behaviour
- debug: diagnose and fix a failing or incorrect behaviour
- research: understand existing code, explain, trace, find usages
- document: generate or update comments, READMEs, ADRs, changelogs
- test: generate, run, or repair tests
- deploy: execute deployment pipelines, apply manifests, manage rollouts
- release: cut releases, tag versions, generate changelogs, publish
- infra: infrastructure queries — pod status, logs, resource health, scaling
- graph: pure structural query — callers, dependents, closure (no LLM needed)

Respond with JSON only. Include the text snippet from the user message that
most strongly signals each intent you identify.`;

Structured output schema

// orchestrator/classifier.ts — response schema
interface ClassificationResult {
  /** Primary intent — the main thing the user wants */
  primary: {
    intent: Intent;
    confidence: number;       // 0.0–1.0
    snippet: string;          // verbatim text from user message that signals this intent
    reasoning: string;        // one sentence explaining the classification
  };
  /** Secondary intent, if the message contains a compound request (optional) */
  secondary?: {
    intent: Intent;
    confidence: number;
    snippet: string;
    reasoning: string;
  };
}

// → primary:   { intent: "debug",     confidence: 0.85, snippet: "returning 500s",    reasoning: "User reports a failure in production behaviour" }
//   secondary: { intent: "implement", confidence: 0.70, snippet: "add retry logic",   reasoning: "User requests new functionality after the fix" }

// Example: "explain how OrderService.create works"
// → primary:   { intent: "research",  confidence: 0.95, snippet: "explain how OrderService.create works", reasoning: "User wants to understand existing code" }

// Example: "who calls embedEntities?"
// → primary:   { intent: "graph",     confidence: 0.98, snippet: "who calls embedEntities", reasoning: "Pure structural caller query — no LLM needed" }

Performance budget

Classification is on the critical path — every user turn waits for it before context assembly begins. The call must be fast:

PropertyTarget
Input tokens~400 (system prompt ~300 + user message ~100)
Output tokens~80–150 (JSON with snippets and reasoning)
Latency target< 500ms on local Ollama (qwen3-coder, GPU)
Fallback on timeoutIf > 2s, fall back to research (safest default — read-only, no mutations)
Why not keyword matching? Keywords are brittle and ambiguous: "fix" could be debug or implement, "spec" could be test or requirements, "changelog" could be release or document. The local LLM resolves these naturally from context — "fix the payment handler to support retries" is implement, "fix why payments are failing" is debug. The snippet field in the response makes the classification auditable: the user (and the orchestrator's logs) can see exactly which words drove the decision.

Compound requests

When the classifier returns both a primary and secondary intent, the orchestrator handles them sequentially in a single turn:

  1. Execute the primary intent first (e.g. debug the 500 errors)
  2. On success, execute the secondary intent (e.g. implement retry logic)
  3. If the primary intent fails, surface the failure and do not proceed to the secondary

The secondary intent is announced to the user before execution: "Also detected: implement (add retry logic). Proceeding after debug completes."

Ollama unavailable fallback

If Ollama is not running at classification time (connection refused), the orchestrator cannot classify. In this case:

  • If the user used an explicit prefix (/intent, /plan, /deploy), the intent is known — proceed without classification
  • Otherwise, prompt the user: "Local model unavailable — cannot classify intent. Use a prefix (/intent debug, /plan, etc.) or start Ollama."
  • If @claude or @opus was specified, route to Claude with research as a safe default intent

Explicit prefix overrides

User prefixes always win over LLM classification. They bypass the classifier entirely:

PrefixEffect
@claudeRoute to Claude regardless of intent; use standard tier model for the classified intent role
@opusRoute to Claude with powerful tier model (Opus 4.6)
@localForce local model even for normally-escalated intents (requirements, design, plan, review); useful offline
/intent <name>Override classified intent explicitly — e.g. /intent debug @claude why is this slow?

Attachment modifier

Attachments are not a standalone intent — they are a context modifier that augments the following intent turn with additional input material. The user attaches a file (drag-and-drop, paste, or explicit path reference) and the framework resolves how to inject it before the intent pipeline runs.

Local LLM cannot process files directly. Neither text files nor binary files can be passed to the local model as raw bytes. Text-based attachments are read and injected as text into the prompt. Binary attachments (images, PDFs) force escalation to Claude regardless of the classified intent.

Attachment type routing

Attachment typeHandlingProvider effect
Text file — .txt, .csv, .log, .md, plain text Framework reads file via Read tool (auto-execute), injects content as text into L4 context. Truncated to L4 budget ceiling if large. No change — normal intent routing applies
Code file — .ts, .py, .go, etc. Treated as text; also resolved against the graph if the file is indexed. Graph entity body takes precedence over raw file content if available. No change — normal intent routing applies
Image — .png, .jpg, .gif, .webp Cannot be fed to local LLM. Passed to Claude as a base64 image content block. Forces Claude regardless of intent; uses standard tier unless @opus specified
PDF Cannot be fed to local LLM. Passed to Claude as a document content block (Files API or inline base64). Forces Claude regardless of intent; uses standard tier unless @opus specified

Forced-Claude augmentation and the implement pipeline

When an image or PDF forces escalation on an implement or test intent, the normal two-stage local→Claude pipeline collapses into a single Claude call: Claude produces the implementation directly (Stage 1 skipped) and the diff validation (Stage 2) is folded into the same response via a structured output format. The local model is not involved in the turn.

Supported tools

All tool execution happens in the framework — neither the local model nor Claude runs anything directly. Both receive tool schemas, emit tool-call JSON, and the framework executes and returns results.

Because insrc is integrated with Claude Code (Agent SDK), most standard tools can be delegated to Claude Code's built-in implementations rather than reimplemented. Claude Code already provides Read, Write, Edit, Glob, Grep, Bash, WebSearch, and WebFetch — complete with permission prompting, sandboxing, and confirmation flows. Only the knowledge graph tools are custom to insrc and must be implemented as MCP tools exposed by the daemon.

Tool execution pipeline

Claude never sees tool output — it only validates call requests for mutating or high-risk operations before execution. Read-only tool calls auto-execute without any Claude involvement. The local model analyses all output.

In practice the local LLM runs through most of a debug session — reading logs, grepping files, querying the graph, running tests — with zero Claude calls. Claude is only invoked when the local LLM wants to write a file, commit, or run a mutating command. 

// Tool execution pipeline
[Step 1] Local LLM emits tool_call_request
         { tool: "Bash", input: "kubectl logs payment-svc --tail=100" }

[Step 2] Framework checks routing table
         → read-only? auto-execute, skip to Step 4
         → mutating?  send to Claude/Haiku for validation

[Step 3] Claude/Haiku validates mutating call (~$0.0004)
         → sees: user intent + tool name + command string only
         → APPROVED | REJECTED (reason + safer alternative)
         → rejected calls returned to local LLM; local LLM retries or stops

[Step 4] Execute via Claude Code built-in or insrc MCP
         → output returned to local LLM for analysis — never sent to Claude

[Step 5] If local LLM stuck after 2 turns with output in hand:
         → local LLM writes ≤200-token summary of findings + specific question
         → Claude/Haiku receives summary (~$0.001), responds with direction
         → raw output never reaches Claude

The two cases where Claude sees content beyond a call request:

  • LLM-generated diffs (implement, refactor, test Stage 2) — the diff is bounded and is exactly what Claude needs to validate correctness
  • Stuck escalation summary — local LLM authored, ≤200 tokens, never raw tool output

Agent Personas

Each persona is a self-contained agent with a distinct system prompt, tool preferences, escalation policy, and output format. The orchestrator delegates to the appropriate persona based on the classified intent. Personas are stateless across turns — they receive a fully assembled context package and return a structured result.

Claude model selection is configurable per intent role in ~/.insrc/config.json. The config uses a two-level structure: tier defaults apply to all roles in that tier, and per-role overrides take precedence. This lets users tune cost vs. quality globally or surgically.

1. Explicit — user-directed

User prefixes a message with @claude or @opus to bypass routing:

Always transparent: The agent never silently calls Claude. Every escalation — whether suggested or automatic — is announced in the output before the API call is made. This keeps the user aware of cost and of code leaving the machine.

Neither provider receives raw file dumps. The agent assembles a structured context package from the Code Knowledge Graph for each task. This keeps prompts tight, deterministic, and cost-efficient.

Provider Interface

Both providers implement a common LLMProvider interface so the agent layer is decoupled from the underlying model. Task routing selects the provider; the agent code does not branch on provider identity. 

// shared/types.ts

Context Window & Memory

The default agent model (qwen3-coder) runs with a 64K-token context window — a property of the model, configurable in the Ollama Modelfile but defaulting to 64K for this model class. This sounds generous but is quickly consumed by system prompt, conversation history, code context, and response headroom all competing for the same budget. A naive sliding-window approach discards turns arbitrarily and loses earlier context permanently. The layered memory model below allocates the budget deliberately, using recency for near-term turns and semantic retrieval for older history.

The index is the memory. The Kuzu Code Knowledge Graph is the agent's persistent long-term memory for code structure. The context window only needs to hold what is immediately relevant to the current turn — everything else is re-queryable on demand.

Layer Budget

┌──────────────────────────────────────────────────────┐  64K total
│  L1  System context        ~1K   fixed               │  always present
│  L2  Session summary       ~3K   rolling             │  compressed older turns
│  L3a Recent turns          ~4K   recency-weighted    │  last 5 turns, tapered verbatim
│  L3b Semantic history      ~4K   retrieved per turn  │  relevant older turns via embedding
│  L4  Task context         ~16K   dynamic per turn    │  graph-fetched, not accumulated
│  L5  Response headroom     ~8K   reserved            │  never used for input
│  L6  Overflow buffer      ~28K   elastic             │  absorbed by L3/L4 as needed
└──────────────────────────────────────────────────────┘
LayerBudgetContentsEviction policy
L1 — System ~1K Agent persona, active repo name, dependency closure repo list (names only) Never evicted
L2 — Summary ~3K Rolling compressed summary of older conversation turns Regenerated in place when L3a overflows; never grows beyond budget
L3a — Recent turns ~4K Last 5 turns verbatim, recency-weighted (newest = full, oldest = truncated) Oldest turn compressed into L2 when L3a overflows
L3b — Semantic history ~4K Top-K older turns retrieved by embedding similarity to current message Re-fetched every turn; not accumulated. Skipped for first 5 turns.
L4 — Task context ~16K Entities + relations fetched from code graph for current turn Replaced every turn — not accumulated across turns
L5 — Response ~8K Reserved for model output Hard reservation — input never fills this

Recency-Weighted Recent Turns (L3a)

Near-term turns are the most contextually connected. Rather than a flat window, L3a weights turns by age — the most recent turn is included in full, older turns are progressively truncated:

Turn ageInclusion
Turn −1 (most recent)Full text
Turn −275% — trim assistant response tail
Turn −350% — user message + first half of response
Turn −425% — user message + first paragraph of response
Turn −5User message only
Turn −6+Evicted to L2 (summarized) or L3b (embedded)

Semantic History Retrieval (L3b)

Every turn is embedded and stored in LanceDB. When assembling context, the current user message is used to retrieve the most semantically relevant older turns — regardless of when they occurred. A turn from 30 exchanges ago resurfaces if it is relevant to the current question.

History embedding is skipped for the first 5 turns of a session (pure L3a recency is sufficient and cheaper when history is short).

Storage — per turn (end of turn, non-blocking)

// memory.ts — called after assistant response is streamed
async function storeTurn(session: Session, turn: ConversationTurn): Promise<void> {
  if (session.turnIndex < 5) return;  // skip embedding for early turns

  const vec = await embed(turn.userMessage);  // embed user message only

  // LanceDB: upsert into conversation_turns table
  await conversationTurnsTable.add([{
    id:        `${session.id}:${session.turnIndex}`,
    sessionId: session.id,
    idx:       session.turnIndex,
    user:      turn.userMessage,
    assistant: turn.assistantResponse,
    vector:    vec,            // float32[2048] for semantic retrieval
    entities:  turn.entities, // graph entity IDs referenced in this turn
    createdAt: new Date().toISOString(),
  }]);
}

Retrieval — per turn (start of turn, on critical path ~5ms)

// memory.ts — called before assembling context
async function retrieveSemanticHistory(
  session: Session,
  vec: number[]
): Promise<ConversationTurn[]> {
  if (session.turnIndex < 5) return [];  // nothing stored yet

  // LanceDB ANN vector search scoped to this session, excluding L3a window
  const minIdx = session.turnIndex - 5;  // exclude last 5 turns (already in L3a)
  return conversationTurnsTable
    .vectorSearch(vec)
    .where(`sessionId = '${session.id}' AND idx < ${minIdx}`)
    .distanceType('cosine')
    .limit(4)
    .toArray();
}

Rolling Summarization (L2)

When L3a overflows (turn −6 is evicted), that turn is compressed into the rolling L2 summary by the local model. The summary accumulates key decisions, entity names, and conclusions — not raw dialogue: 

// memory.ts
async function evictToSummary(session: Session): Promise<void> {
  const turn = session.recentTurns.pop();  // evict oldest (turn -6)

  const prompt = [
    { role: 'system', content:
        'Update the running summary to include the new exchange. ' +
        'Preserve: decisions made, file names, entity names, open questions. ' +
        'Be concise. Max 3 sentences total.' },
    { role: 'user', content:
        `Current summary:\n${session.summary}\n\nNew exchange:\n${formatTurn(turn)}` }
  ];

  session.summary = await ollamaProvider.complete(prompt, { maxTokens: 250 });
}

Dynamic Graph Retrieval (L4)

Task context is not accumulated — it is re-assembled from the code graph each turn. This keeps L4 fresh and avoids context drift from stale code that has since been edited by the indexer daemon. 

// context.ts
async function assembleContext(turn: string, session: Session): Promise<AssembledContext> {
  // Single embed call serves both code search (L4) and history retrieval (L3b)
  const vec = await embed(turn);

  const [codeHits, historyHits] = await Promise.all([
    db.searchCode(vec, { repos: session.closureRepos, limit: 8 }),   // L4
    retrieveSemanticHistory(session, vec),                            // L3b
  ]);

  const neighbors  = await db.neighbors(codeHits.map(h => h.id));
  const entities   = buildEntityContext(codeHits, neighbors, session.seenEntities);

  return fitToBudget({
    system:    session.systemContext,                    // L1
    summary:   session.summary,                          // L2
    recent:    weightedRecent(session.recentTurns),      // L3a
    semantic:  historyHits,                              // L3b
    code:      entities,                                 // L4
  });
}
Single embed per turn: The same embedding vector is reused for both the code graph search (L4) and the conversation history search (L3b). Only one Ollama call is needed.

Token Budget Enforcement

Context assembly respects hard per-layer ceilings. When a layer overflows, content is dropped in a defined priority order to preserve the most critical context. 

// budget.ts
const TOKEN_BUDGET = {
  system:   1_000,
  summary:  3_000,
  recent:   4_000,   // L3a
  semantic: 4_000,   // L3b
  code:    16_000,   // L4
  response: 8_000,   // reserved — never consumed by input
  total:   64_000,
} as const;

Overflow priority (what gets dropped first)

Each layer has a fixed ceiling. When assembling context exceeds the total budget, content is shed in this order — lowest priority first:

PriorityLayerDrop strategy
1st droppedL4 — code entitiesDrop lowest vector-score entities first; always keep entities directly named in the current turn
2nd droppedL3b — semantic historyDrop lowest-similarity retrieved turns first
3rd droppedL3a — recent turnsDrop oldest turns first (beyond the last 2 turns, which are always kept)
4th droppedL2 — session summaryTruncate to half; always keep the first sentence (repo + goal context)
Never droppedL1 — system contextHard floor — never truncated. If L1 alone exceeds budget, daemon logs an error.

When L4 is dropped below 4K tokens due to overflow, the agent prints a notice: "Context tight — some code entities were omitted. Use /intent graph to fetch specific entities by name."

History Retention & Pruning

Raw turn embeddings are ephemeral. At session end, the final L2 summary is promoted to a persistent conversation_session record and raw turns are deleted. This gives cross-session continuity without unbounded storage growth.

Session close

// session.ts — on disconnect or explicit /exit
async function closeSession(session: Session): Promise<void> {
  const now = new Date().toISOString();
  const expiresAt = new Date(Date.now() + 30 * 86_400_000).toISOString(); // 30 days

  // Promote summary to persistent LanceDB store
  await conversationSessionsTable.add([{
    id:           session.id,
    repo:         session.repoId,
    summary:      session.summary,
    seenEntities: [...session.seenEntities],
    createdAt:    now,
    expiresAt:    expiresAt,
    vector:       await embed(session.summary),  // for semantic cross-session seeding
  }]);

  // Delete all raw turns — summary is the durable artifact
  await conversationTurnsTable.delete(`sessionId = '${session.id}'`);
}

Cross-session seeding

On session start, recent session summaries for the same repo seed the initial L2:

// session.ts — on session start
// Embed the user's opening message as query vector for semantic similarity
const now = new Date().toISOString();
const queryVector = await embed(openingMessage);  // embed first user turn for relevance
const priorSummaries = await conversationSessionsTable
  .search(queryVector)                              // nearest-neighbour by summary vector
  .where(`repo = '${repoId}' AND expiresAt > '${now}'`)
  .limit(3)
  .toArray();
// Sort by recency (vector search returns by similarity; we prefer recent + relevant)
priorSummaries.sort((a, b) => b.createdAt.localeCompare(a.createdAt));

session.summary = priorSummaries.map(s => s.summary).join('\n\n');

// lifecycle.ts — runs daily via setInterval
async function pruneConversationHistory(): Promise<void> {
  const now = new Date().toISOString();

  // Delete expired summaries (LanceDB filter delete)
  await conversationSessionsTable.delete(`expiresAt < '${now}'`);

  // Cap at 20 summaries per repo — delete oldest beyond cap
  // Note: Map.groupBy requires Node.js ≥ 21. Use Object.groupBy polyfill for Node 18/20.
  const allSessions = await conversationSessionsTable.query().toArray();
  const byRepo = Map.groupBy(allSessions, s => s.repo);
  for (const [repo, sessions] of byRepo) {
    const sorted = sessions.sort((a, b) => b.createdAt.localeCompare(a.createdAt));
    const toDelete = sorted.slice(20).map(s => s.id);
    if (toDelete.length > 0) {
      const ids = toDelete.map(id => `'${id}'`).join(',');
      await conversationSessionsTable.delete(`repo = '${repo}' AND id IN (${ids})`);
    }
  }
}
Claude escalation and memory: Claude has a 200K token context window, so budget pressure is much lower. When escalating, pass L1 + L2 (summary) + assembled task context (L4). Do not forward L3a/L3b turn history — it adds noise for design-level questions.

Model Bootstrap

On daemon startup, after the embedding model check, the agent model is also verified and pulled if missing. Both checks run sequentially before any index or agent jobs are processed. 

// lifecycle.ts
const MODELS = {
  embedding: 'qwen3-embedding:0.6b',
  agent:     'qwen3-coder:latest',
};

async function ensureModels(ollama: Ollama): Promise<void> {
  const { models } = await ollama.list();
  const installed = new Set(models.map(m => m.name));

  for (const [role, model] of Object.entries(MODELS)) {
    if (installed.has(model)) continue;
    log.info(`Pulling ${role} model: ${model}`);
    const stream = await ollama.pull({ model, stream: true });
    for await (const event of stream) {
      const pct = event.total ? Math.round((event.completed / event.total) * 100) : 0;
      setDaemonState({ modelPull: { role, model, pct } });
    }
    log.info(`${model} ready`);
  }
}

Fault Tolerance

The agent system has two critical dependencies that can become unavailable at runtime: the local Ollama service (used by Developer, Tester, and Deployer personas) and the insrc daemon (which hosts the graph and plan MCP tools used by all personas). Both must degrade gracefully without data loss.

Ollama unavailability

ScenarioDetectionBehaviour
Ollama not running at session start Connection refused on first health-check ping Print: "Local model (Ollama) is not running. Start it with ollama serve, or use @claude / @opus prefix to route directly to Claude." Session continues in Claude-only mode.
Ollama goes away mid-session Request timeout or connection reset on tool call Print warning, offer to re-route current turn to Claude. If user declines, abort current turn and preserve session state.
Model not pulled ollama.generate returns 404 model not found Print: "Model qwen3-coder:latest not found. Pull with ollama pull qwen3-coder" and offer Claude fallback for current session.
Ollama OOM / crash Timeout > 30s on in-flight request Abort request, log error, fall back to Claude for the current turn. User-visible notice printed.

Daemon crash and stale graph

ScenarioDetectionBehaviour
Daemon not running MCP connection refused; graph_* / plan_* tools unavailable Print: "insrc daemon is not running. Start with insrc daemon start." Disable graph and plan tools for session. LLM still functional without graph context.
Daemon crashes mid-session MCP tool call returns transport error Mark graph tools unavailable. Print warning. Attempt daemon auto-restart if insrc daemon start is in PATH. Resume graph tools if restart succeeds within 10s.
Stale graph (index behind HEAD) Daemon reports indexStatus: stale (last index SHA ≠ HEAD SHA) Print: "Graph index is behind HEAD by N commits. Re-index with insrc index." Continue with stale graph; annotate graph results with "[stale — may not reflect recent changes]".
Kuzu DB corruption Kuzu query throws schema error or SIGABRT in daemon Daemon marks itself unhealthy. CLI prints: "Graph database may be corrupted. Run insrc index --rebuild to rebuild from source."

Plan pre-flight — local-only condensed path

When the user issues a plan intent, the local model runs a condensed pre-flight analysis before Claude is invoked (to assemble the context package):

  1. Entity resolution — identify entities touched by the feature request using graph traversal only (auto-execute, no LLM)
  2. Scope estimate — count files, repos, and estimate token budget for the full plan context
  3. Local sketch — local model generates a rough ordered list of steps (~200 tokens) to give Claude a starting structure
  4. Claude enhancement — Claude receives: user request + entity signatures + local sketch; produces the final plan with dependencies, estimates, and ordered steps
  5. Plan persisted — plan written to Kuzu Plan / PlanStep tables via plan_step_update

If Ollama is unavailable, step 3 is skipped and Claude receives only steps 1–2 output. If the daemon is unavailable, plan persistence is skipped and the plan is printed to the terminal only (user can copy it).

Re-indexing handoff

After any implement or refactor turn that writes files to disk, the agent enqueues a background re-index job for the changed files via the daemon. The REPL prints a non-blocking notice: 

✓ 2 files written. Re-indexing in background (src/payment/handler.ts, src/payment/types.ts)...
  (graph queries may return stale results until indexing completes — usually <5s)

The re-index is file-scoped (only the changed files are re-parsed and their graph nodes updated) rather than a full repo scan, keeping the feedback loop fast.

Multi-step plan execution UX

When an active plan exists, the REPL shows plan progress at the start of each turn: 

insrc> [Plan: Add Stripe support — Step 2/5: implement PaymentGateway interface]
       → developer/implement
       What's next? (enter to continue, /plan to view full plan, /skip to skip step)

Pressing enter starts the next in-progress step automatically. The orchestrator:

  1. Calls plan_get to fetch the current in-progress step
  2. Classifies the step description to select the right agent persona (e.g. implement → Developer, test → Tester)
  3. Delegates to the selected agent with assembled context
  4. On success: calls plan_step_update(status: 'done') and prints the next pending step
  5. On failure: calls plan_step_update(status: 'failed'), shows error, asks user to retry or skip

Plan steps may route to different agents as execution progresses — e.g. an implement step goes to Developer, the following checkpoint test step goes to Tester, and a deploy step at the end goes to Deployer. The orchestrator handles all routing transparently.

Open Questions

  1. Summarization model — Rolling summarization (L2 eviction) uses qwen3-coder:latest for convenience, but a smaller/faster model (e.g. qwen3:0.6b) would be more efficient for this internal housekeeping task. Worth benchmarking summary quality vs. latency tradeoff.
  2. Token counting accuracy — Accurate token counting requires the model's tokenizer. Approximating with character count (~4 chars/token) is fast but imprecise. Consider using tiktoken-style counting or a lightweight tokenizer aligned to Qwen's vocabulary to avoid budget overruns.
  3. Classification latency vs. accuracy — The local LLM classifier adds ~300–500ms to every turn. If this proves too slow on CPU-only machines, consider a tiered approach: fast regex for unambiguous prefixes (e.g. /plan, who calls X) to skip the LLM call entirely, falling back to LLM classification only for ambiguous messages. Log all classification results (intent, confidence, snippet) for offline analysis of misroutes.
  4. Agent handoff — Can one agent invoke another mid-turn? E.g. the Developer finishes an implement step and wants to hand off to the Tester for the next checkpoint step. Currently the orchestrator handles this at turn boundaries, but seamless intra-turn handoff would improve plan execution flow.
  5. Persona-specific memory — Should each persona maintain its own L2 summary tags (e.g. [designer:requirements], [developer:debug-findings]) or share a flat L2 namespace? Namespaced tags prevent accidental cross-persona interference but add complexity.
  6. Deployer safety model — The Deployer has the highest ratio of mutating-to-read-only tool calls. Should it default to dry-run for all mutations (user confirms each), or should the orchestrator allow auto-execution for known-safe patterns (e.g. kubectl get in infra intent)?
  7. Streaming UX — Both providers support streaming. The REPL should stream tokens to the terminal in real time. Escalation announcement and agent persona identification must appear before the first token from the LLM.
