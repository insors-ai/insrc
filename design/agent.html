Agent Design Document

insrc coding agent — v0.1 — March 2026

Overview

The insrc agent is a local-first hybrid coding assistant. The majority of tasks are handled autonomously by a local LLM running via Ollama — fast, private, and free. For tasks that require deeper reasoning, architectural judgment, or cross-repo design decisions, the agent transparently escalates to Claude.

Both providers draw context from the same source: the live Code Knowledge Graph maintained by the indexer daemon (see indexer.html). The agent never sends raw file dumps to any LLM — it assembles structured, minimal context from graph queries.

Indexer dependency: The agent requires the daemon to be running. On startup it connects via the Unix socket, resolves the dependency closure of the active repo, and holds that closure for the session duration.

LLM Providers

LOCAL qwen3-coder:latest

  • Runs via Ollama — no API key, no network
  • Code-specialized model, fast response
  • Handles the majority of day-to-day tasks
  • Auto-installed on daemon startup if missing
  • Privacy default — code never leaves the machine

CLOUD Claude (Anthropic)

  • Invoked only for complex / design tasks
  • Requires ANTHROPIC_API_KEY
  • Always announced to user before invocation
  • Receives structured graph context, not raw files
  • 200K token context — handles large result sets

Task Routing

Every user message is classified into one of the intents below before a provider is selected. Classification uses a combination of keyword signals, cursor context (active file, selected entity), and graph signals (entity count, repo count, graph depth). Local handles everything it can; Claude is invoked only when the task genuinely exceeds local capability.

Graph queries are free: Tasks like "find all callers of X" or "what does this repo depend on" are answered directly from Kuzu with no LLM involved. The agent resolves these as pure graph traversals and formats the result for the user.

Intent classification

Classification runs locally — no LLM call is required to route a message. It is a fast, deterministic two-pass pipeline that produces an Intent with confidence score, used to select the provider and pre-fetch the right graph context.

Pass 1 — keyword + prefix signals

A small vocabulary of high-precision trigger phrases is checked against the lowercased message. Hits above a confidence threshold short-circuit to Pass 2 for graph confirmation only.

When keyword confidence is below threshold (< 0.7) or two intents score similarly, graph signals resolve ambiguity:

Attachment modifier

Attachments are not a standalone intent — they are a context modifier that augments the following intent turn with additional input material. The user attaches a file (drag-and-drop, paste, or explicit path reference) and the framework resolves how to inject it before the intent pipeline runs.

Local LLM cannot process files directly. Neither text files nor binary files can be passed to the local model as raw bytes. Text-based attachments are read and injected as text into the prompt. Binary attachments (images, PDFs) force escalation to Claude regardless of the classified intent.

Supported tools

All tool execution happens in the framework — neither the local model nor Claude runs anything directly. Both receive tool schemas, emit tool-call JSON, and the framework executes and returns results.

Because insrc is integrated with Claude Code (Agent SDK), most standard tools can be delegated to Claude Code's built-in implementations rather than reimplemented. Claude Code already provides Read, Write, Edit, Glob, Grep, Bash, WebSearch, and WebFetch — complete with permission prompting, sandboxing, and confirmation flows. Only the knowledge graph tools are custom to insrc and must be implemented as MCP tools exposed by the daemon.

Tool execution pipeline

Claude never sees tool output — it only validates call requests for mutating or high-risk operations before execution. Read-only tool calls auto-execute without any Claude involvement. The local model analyses all output.

requirements

The user describes something they want built — a feature, a change, a constraint. The agent's job is to turn that raw intent into a structured, grounded requirements sketch: what already exists that is relevant, what is missing, what decisions still need to be made.

This is a two-stage process. The local model does the groundwork first; Claude enhances it.

Stage 1 — Local sketch

The local model (qwen3-coder) receives the user message plus the L4 graph context assembled for this turn: relevant entities, their signatures, and 1-hop relations. It produces a requirements sketch — a structured but intentionally rough draft that captures:

  • What already exists — existing entities, patterns, and conventions in the codebase that are relevant to the request
  • What needs to change or be added — gaps identified between the request and the current code
  • Open questions — ambiguities or decisions the user has not yet made
  • Constraints — inferred non-functional requirements (performance, privacy, compatibility) from the codebase context

The local model does not attempt to answer the open questions — it surfaces them. Its output is intentionally scoped to what it can see and is confident about. 

// Stage 1 prompt skeleton (system)
You are a requirements analyst for a software project.
You have been given the user's request and relevant code context from the codebase.

Your task:
1. Identify existing code that is relevant to this request.
2. List what needs to be built or changed.
3. List open questions — things the user has not specified and that have more than one reasonable answer.
4. Infer any constraints from the existing code (naming conventions, error handling patterns, etc.)

Be concise. Do not make decisions on behalf of the user. Do not write code.

Stage 2 — Claude enhancement

The sketch is always escalated to Claude regardless of scope. Claude receives the local sketch plus the same graph context and is asked to:

  • Fill in missing context the local model may have overlooked (cross-repo implications, dependency impacts)
  • Sharpen ambiguous requirements into concrete, testable statements
  • Propose answers to the open questions the local model surfaced, with tradeoffs where relevant
  • Flag any requirements that conflict with existing architecture or introduce significant risk

Claude's output is the final requirements document for this session, which is stored in L2 (session summary) and referenced by subsequent design, plan, and implement tasks in the same session. 

// Stage 2 prompt skeleton (system)
You are a senior engineer reviewing a requirements sketch produced by a local model.
Below is the sketch, the user's original request, and the relevant code context.

Your task:
1. Sharpen each requirement into a concrete, testable statement.
2. Propose answers to the open questions, noting tradeoffs where multiple choices exist.
3. Identify cross-repo or architectural implications not captured in the sketch.
4. Flag conflicts with existing patterns or risks introduced by this change.

Output a revised requirements document in markdown. Keep it actionable.
Always escalated: Requirements tasks always go through both stages. There is no local-only path. The sketch is Claude's input, not a substitute for Claude's output.

Session carry-forward

After Claude returns, the requirements document is compressed into L2 (rolling summary) with a [requirements] tag so subsequent turns in the same session can reference it without repeating the full text. The plan and design intents pick this up automatically.

design

The user wants to reason about how something should be built: API shape, module boundaries, data flow, tradeoffs between approaches. The agent's job is to produce a concrete design proposal grounded in the existing codebase — not a blank-sheet architecture, but one that fits the patterns and constraints already in place.

Like requirements, this is a two-stage process. The local model drafts a design sketch; Claude elevates it.

Stage 1 — Local sketch

The local model receives the user message, any [requirements] carried forward in L2, and the L4 graph context for this turn. It produces a design sketch covering:

  • Proposed structure — new modules, types, functions, or interfaces needed; how they map onto the existing file/package layout
  • Integration points — which existing entities are called, extended, or replaced; relations to adjacent code surfaced from the graph
  • Data flow — how data moves through the proposed change (inputs, outputs, side effects)
  • Alternatives considered — at least one alternative approach the local model can identify, with a one-line reason it was not chosen
  • Open design questions — decisions that depend on context beyond the local model's visibility

The sketch is expressed as prose + pseudocode or interface signatures — not full implementation. The local model stays within what it can see in the graph and does not speculate about cross-repo implications. 

// Stage 1 prompt skeleton (system)
You are a software architect working on an existing codebase.
You have been given the user's design question, prior requirements context (if any),
and relevant code entities from the codebase knowledge graph.

Your task:
1. Propose a concrete design that fits the existing codebase structure.
2. Identify which existing entities are touched and how.
3. Describe the data flow for the proposed change.
4. Name at least one alternative and why it was set aside.
5. List open design questions that exceed your visibility.

Use interface signatures or pseudocode where helpful. Do not write full implementations.

Stage 2 — Claude enhancement

Claude receives the local sketch, the requirements document (from L2 if present), and the full graph context. It is asked to:

  • Evaluate the proposed design against the requirements — does it satisfy them?
  • Identify cross-repo impact: which dependent repos would need to change and how
  • Assess tradeoffs between the proposed approach and the alternatives, with concrete reasoning
  • Resolve open design questions where possible, or reframe them as explicit decisions for the user
  • Produce a final design document: chosen approach, rationale, interface contracts, migration notes if applicable

Session carry-forward

Claude's design document is stored in L2 with a [design] tag. The plan intent reads both [requirements] and [design] tags from L2 to produce a sequenced implementation plan without re-running these stages.

plan

The user wants a concrete, ordered checklist of implementation steps for a feature or change. Planning is downstream of requirements and design — a good plan references both. If neither is present in the session, the agent runs a lightweight inline version before planning.

Same two-stage pattern: local draft, Claude refinement.

Pre-flight: context check

Before generating a plan, the agent checks L2 for [requirements] and [design] tags. If both are present, they are passed directly to Stage 1. If either is missing, the agent runs a condensed single-turn version of the missing stage (local only, no Claude escalation) to produce a minimal anchor — just enough context for the plan to be coherent.

Stage 1 — Local draft

The local model receives the requirements and design context plus the L4 graph for this turn. It produces an ordered implementation checklist:

  • Steps — each step is a self-contained unit of work, scoped to a single file or entity where possible
  • Dependencies between steps — which steps must precede others and why
  • Test checkpoints — after which steps a test can confirm progress
  • Unknowns — steps the local model cannot fully specify (e.g. cross-repo changes it cannot see)

Steps are expressed as imperative sentences, not prose. Each step names the file or entity it touches. The local model does not generate code at this stage. 

// Stage 1 prompt skeleton (system)
You are a senior engineer breaking a feature into an ordered implementation plan.
You have the requirements, the design, and the relevant code graph context.

Your task:
1. Produce a numbered checklist of implementation steps.
2. Each step must name the file or entity it touches and state what changes.
3. Mark steps that depend on a prior step completing first.
4. Insert test checkpoints after steps that produce a verifiable intermediate state.
5. Flag steps you cannot fully specify due to limited visibility.

Do not write code. Steps should be one sentence each.

Stage 2 — Claude refinement

Claude receives the local draft plan, requirements, design, and graph context. It:

  • Fills in steps the local model flagged as underspecified, using cross-repo graph visibility
  • Reorders steps if the local model's sequencing is suboptimal (e.g. a shared type should be defined before its consumers)
  • Adds rollback or migration steps if the change touches persistent data, public APIs, or shared contracts
  • Estimates relative complexity per step (simple / moderate / complex) to help the user prioritise or split the work
  • Outputs the final plan as a markdown checklist, ready to be saved or tracked

1. Fill in any underspecified steps using your broader context.

Plan persistence and checkpointing

The plan is a structured artifact — an ordered checklist with step identifiers, complexity labels, dependency annotations, and tick state. Running it through L2 rolling summarization would corrupt all of that: step order, tick state, and structured fields are not preserved by prose compression. The plan must be stored verbatim, outside the L2 pipeline entirely.

Plans are stored in Kuzu — the embedded property graph already running in the daemon. Plans have no embedding component, so LanceDB (the vector store) is the wrong tool; Kuzu handles structured node tables with Cypher queries and survives daemon restarts without any additional infrastructure.

Two node tables are added to the existing Kuzu schema: Plan and PlanStep, connected by a CONTAINS edge. Step dependency edges (DEPENDS_ON) are also stored as graph edges, making dependency resolution a simple Cypher reachability query rather than a string-parsing problem. 

// Kuzu schema additions (src/db/schema.ts)

CREATE NODE TABLE Plan (
  id          STRING,
  repo        STRING,
  title       STRING,
  createdAt   STRING,
  updatedAt   STRING,
  completedAt STRING,   // '' while in progress
  PRIMARY KEY (id)
)

CREATE NODE TABLE PlanStep (
  id          STRING,   // '{planId}:step{seq}' — stable and human-readable
  planId      STRING,
  seq         INT32,    // display order (1-based)
  body        STRING,   // imperative sentence describing the step
  complexity  STRING,   // 'simple' | 'moderate' | 'complex'
  checkpoint  BOOLEAN,  // true → test gate required before next step
  state       STRING,   // 'pending' | 'in_progress' | 'done' | 'skipped' | 'failed'
  fileHint    STRING,   // file or entity this step touches
  startedAt   STRING,   // '' until state → in_progress
  doneAt      STRING,   // '' until state → done | skipped | failed
  note        STRING,   // one-sentence outcome written by agent on completion
  PRIMARY KEY (id)
)

CREATE REL TABLE CONTAINS  (FROM Plan TO PlanStep)
CREATE REL TABLE DEPENDS_ON (FROM PlanStep TO PlanStep)  // step must complete before target
// Dependency resolution — Cypher (no string parsing)
// "Is step X unblocked?" — all predecessors must be done or skipped
MATCH (s:PlanStep {id: $stepId})
OPTIONAL MATCH (s)<-[:DEPENDS_ON]-(dep:PlanStep)
WHERE dep.state NOT IN ['done', 'skipped']
RETURN count(dep) AS blockers
// blockers = 0 → step is ready to start
// L2 carries only a lightweight reference

Pre-turn checkpoint gate

Before any implement, test, or debug turn executes, the agent runs a checkpoint gate against the active plan:

  1. Fetch all steps for the active plan from plan_steps.
  2. Find the next pending step whose dependsOn steps are all done or skipped.
  3. Confirm no step is currently in_progress from a prior session (crash recovery: reset stale in_progress → pending if startedAt is older than the current session start).
  4. Mark the target step in_progress and stamp startedAt before the LLM call, not after. This prevents a crash between generation and commit from leaving a step invisible.
  5. Inject only the target step's body and fileHint into L4 — not the entire plan — to keep context tight.

Post-turn commit

After the agent successfully writes or verifies code for a step:

  1. Transition step state → done, stamp doneAt, write a one-sentence note summarising what changed.
  2. If the step had checkpoint: true, the agent prompts the user to run tests before proceeding to the next step. The next step remains pending until the user confirms or the agent observes a passing test run.
  3. Update the L2 pointer with the new progress count.
  4. Surface the next pending step to the user so they can immediately continue or pause.

research

The user wants to understand something — how a piece of code works, what calls what, why a pattern was chosen, or how an external library or API behaves. Research spans two distinct sources: the Code Knowledge Graph (what exists in the codebase) and the web (what exists outside it). The agent decides which source to query, or both, based on the nature of the question.

Source selection

Question typeSourceExample
How does this function/class work?Graph only"explain IndexerService.processJob"
What calls / uses this entity?Graph only (pure Kuzu traversal)"who calls embedEntities?"
What does this repo depend on?Graph only"show me the dependency closure"
How do I use an external library?Web only"how does LanceDB vector search work?"
Is there a known bug in a dependency?Web only"any issues with kuzu@0.11.3 and node?"
How should I integrate X into this codebase?Graph + Web"how do I add OpenTelemetry tracing to the daemon?"
Why does the code do it this way?Graph + optionally web"why is the embedding dimension 1024?"

Graph research (local, no LLM for pure structural queries)

Questions scoped to the codebase are resolved from the Code Knowledge Graph first. Pure structural questions (callers, callees, dependents, file contents) are answered directly from Kuzu and LanceDB with no LLM — they are routed as graph intent and the result is formatted and returned immediately.

For interpretive questions ("explain", "why", "what does this do"), the graph provides the entity bodies and relations, and the local model narrates them: 

// Local model prompt skeleton — graph research
You are explaining code to a developer.
You have been given the relevant code entities and their relationships from the codebase graph.

Your task:
1. Explain what the code does in plain language.
2. Describe how the entities relate to each other (calls, implements, depends on).
3. If the code has non-obvious behaviour or constraints, call them out.
4. Do not speculate beyond what the code shows.

Be concise. Use the entity names and file paths provided.

Web research

When the question involves external libraries, APIs, tools, known bugs, or patterns outside the codebase, the agent performs a web search. Web search is triggered when the question contains signals that exceed the graph's visibility:

  • Names of packages, frameworks, or APIs not defined in the codebase (e.g. kuzu, ollama, tree-sitter)
  • Keywords: "documentation", "changelog", "release notes", "known issue", "bug", "how to", "example of"
  • Version-specific questions (e.g. "what changed in X@2.0?")
  • Questions about best practices or conventions not answerable from local code patterns
// Web search flow
1. Local model generates 1–3 targeted search queries from the user's question
2. Agent executes searches via SearchProvider and fetches top results (title + snippet, or full page if needed)
3. Local model synthesises the results into a concise answer grounded in the search findings
4. Sources are cited inline so the user can follow up directly

SearchProvider abstraction

Web search is exposed through a thin SearchProvider interface so the backend can be swapped without touching calling code — analogous to LLMProvider: 

interface SearchResult {
  title:   string;
  url:     string;
  snippet: string;
}

interface SearchProvider {
  search(query: string, limit?: number): Promise<SearchResult[]>;
}

Search backend — Brave Search API (preferred) or Claude fallback

The preferred backend is the Brave Search API: reliable, structured JSON, 2,000 free queries per month, one API key. If no key is configured, all web search turns are automatically escalated to Claude, which uses Anthropic's hosted web_search tool.

Combined graph + web research

For questions that require both (e.g. "how do I add tracing to the daemon?"), the agent runs both sources in parallel and merges the results before generating an answer:

  1. Graph query: fetch the daemon's existing entry points, middleware, and dependency injection patterns from Kuzu + LanceDB
  2. Web search: find documentation or examples for the tracing library being considered (via Brave or Claude)
  3. Local model receives both sets of results and produces an answer that explains how the external approach maps onto the existing codebase structure

Escalation

Research stays local by default. Claude is invoked for web search when no Brave key is configured, and also when:

  • The question spans more than 2 repos (cross-repo architectural explanation)
  • The user prefixes with @claude
  • The local model explicitly signals low confidence after seeing both graph and web results

implement

The user wants to write or modify code to satisfy a known specification — either an explicit instruction or an active plan step. Implementation is a two-stage pipeline: the local model writes the code, Claude validates it. The framework is responsible for ensuring Claude receives only the minimal set of code snippets necessary to validate the change — never whole files, never the full codebase.

Stage 1 — Local model writes the implementation

The local model receives the task description (from the active plan step body or the user's message), plus a tightly scoped context bundle assembled by the framework from the knowledge graph:

  • Target entities: the function(s), class(es), or file(s) to be created or modified — their current body and signature fields from LanceDB
  • Direct neighbours: callers and callees one hop away in the Kuzu graph (signatures only, not bodies, to keep context small)
  • Type signatures: any interface or type definitions referenced by the target entities
  • Plan step: the body and fileHint of the active PlanStep node
// Framework context assembly (Stage 1)
const target    = await getEntity(db, entityId);              // full body
const callers   = await getCallers(db, entityId, 1);          // signatures only
const callees   = await getCallees(db, entityId, 1);          // signatures only
const types     = await getReferencedTypes(db, entityId);     // full body (usually small)
const planStep  = await getPlanStep(db, activePlanStepId);    // body + fileHint

// Local model prompt
`Implement the following change.
Plan step: ${planStep.body}

Target (full):
${target.body}

Direct callers (signatures):
${callers.map(c => c.signature).join('\n')}

Direct callees (signatures):
${callees.map(c => c.signature).join('\n')}

Types in scope:
${types.map(t => t.body).join('\n')}

Output a unified diff only. No explanation.`

The local model outputs a unified diff. The framework applies it to the working tree with a dry-run check before writing, and records the set of touched entity IDs by parsing the diff's ---/+++ headers against the indexed file paths.

Stage 2 — Claude validates the diff

The framework assembles a second, minimal context bundle for Claude — derived entirely from the diff, not from re-reading the codebase. Only entities actually touched or directly affected by the change are included:

  • The diff itself — always included in full
  • Bodies of touched entities — fetched by matching diff hunks to indexed entity ranges (startLine/endLine in LanceDB)
  • Signatures of first-degree neighbours — callers and callees of any modified entity (signatures only)
  • Relevant type definitions — only types referenced in the diff, not the whole type graph

Entities that are not touched and not directly adjacent to a touched entity are never included. This is enforced by the framework — the validation prompt is constructed programmatically, not handed to Claude to decide what to fetch. 

// Framework context assembly (Stage 2 — minimal, diff-derived)
const touchedIds  = parseDiffToEntityIds(diff, db);           // hunk → entity lookup
const touched     = await getEntities(db, touchedIds);        // full bodies
const neighbours  = await getNeighbours(db, touchedIds, 1);   // signatures only
const types       = await getReferencedTypesFromDiff(diff, db); // full bodies

// Claude validation prompt
`You are reviewing a code change generated by a local model.

Diff:
${diff}

Touched entities (full):
${touched.map(e => e.body).join('\n\n')}

Adjacent entities (signatures only — for call-site compatibility):
${neighbours.map(n => n.signature).join('\n')}

Type definitions referenced in the diff:
${types.map(t => t.body).join('\n\n')}

Validate the diff. Check for:
- Correctness: does the implementation match the plan step intent?
- Compatibility: are all call sites still satisfied?
- Edge cases: null/undefined, empty collections, error paths
- Security: injection, unvalidated input, exposed internals
- Style: consistent with the surrounding code patterns shown

Respond with one of:
  APPROVED — brief rationale
  CHANGES_NEEDED — bullet list of specific issues with line references`

Outcome handling

  • APPROVED: diff is written to disk, plan step marked done via plan_step_update, index job enqueued for changed files
  • CHANGES_NEEDED: Claude's bullet-list issues are appended to the local model correction prompt as explicit fix directions (e.g. "Fix null check on line 12 — OrderService.create must throw on null input"); Stage 1 re-runs with these directions. Up to 2 retry rounds.
  • 2 retries exhausted: user is shown the diff, Claude's feedback, and a prompt to decide: accept as-is / reject / edit manually. No further automatic retry.
Context budget: The Stage 2 payload sent to Claude is bounded by the diff size plus one graph hop. For a typical single-function change this is well under 4K tokens. Multi-file changes that touch many entities are split into per-file validation rounds to keep each call focused and cheap.

refactor

The user wants to restructure existing code without changing observable behaviour — rename, extract, inline, split, or reorganise. Refactor follows the same two-stage pipeline as implement (local model produces a diff, Claude validates it) with two differences in prompt framing:

Stage 1 differences — local model prompt

  • The goal is stated as a structural change, not a feature addition
  • The local model is explicitly instructed that observable behaviour must not change — same inputs must produce the same outputs, same errors must be thrown
  • Neighbour context is weighted more heavily: because a refactor often ripples into callers (e.g. a rename or signature change), callers are included at 2 hops rather than 1
// Stage 1 prompt differences for refactor
`Refactor the following code as described. Do not change observable behaviour.
Same inputs → same outputs. Same errors → same error types and messages.

Refactor instruction: ${taskDescription}

Target (full):
${target.body}

Callers up to 2 hops (signatures — must remain compatible):
${callers2hop.map(c => c.signature).join('\n')}

Callees 1 hop (signatures):
${callees.map(c => c.signature).join('\n')}

Output a unified diff only.`

Stage 2 differences — Claude validation prompt

  • Correctness check is replaced with a behaviour-equivalence check: Claude is asked to reason about whether the refactored code preserves the contract of the original
  • Call-site compatibility is checked at 2 hops to match the wider context given to the local model
// Stage 2 validation differences for refactor
`Validate this refactor. The change must preserve observable behaviour.

Check for:
- Behaviour equivalence: same inputs → same outputs, same error paths
- Call-site compatibility across all adjacent signatures shown
- No unintended scope, visibility, or export changes
- Style consistency with surrounding code

Respond with APPROVED or CHANGES_NEEDED (with line references).`

Outcome handling, retry logic, and context budget rules are identical to implement.

test

The user wants test cases written for existing or newly implemented code. The pipeline has four phases that repeat as a loop: develop → validate → execute → fix. Test case authoring follows the same two-stage local/Claude pattern as implement. Execution and fixing introduce an additional repair loop with escalation on the 4th consecutive failure. 

// Test intent — overall loop
while (tests failing) {
  [Stage 1] Local model writes / rewrites tests
  [Stage 2] Claude validates test quality
  [Stage 3] Execute tests locally
  [Stage 4] Fix loop:
    for attempt in 1..3:
      local model fixes → re-execute → if pass: break
    if still failing:
      Claude fixes → re-execute → if pass: break
      // repeat Stage 4 with reset counter
}

Stage 1 — Local model develops test cases

The framework assembles a context bundle from the knowledge graph identical in structure to implement Stage 1, but scoped to the entity under test:

  • Full body of the entity under test
  • Its type signature and any referenced type definitions
  • Existing test file for that module (if any) — so the local model extends rather than duplicates
  • Active plan step body if the test task came from a plan
// Stage 1 local prompt (test)
`Write test cases for the following entity.
Cover: happy path, boundary values, error/exception paths, null/undefined inputs.
Use the existing test style shown. Add to the existing test file — do not rewrite it.

Entity under test:
${entity.body}

Existing tests (if any):
${existingTestBody}

Output a unified diff against the test file only.`

Stage 2 — Claude validates test quality

Same diff-derived minimal context as implement Stage 2. Claude receives the test diff, the entity under test (full body), and its type definitions. Claude does not receive the full test suite or unrelated files. 

// Stage 2 Claude validation prompt (test)
`Review these test cases. The entity under test is provided in full.

Test diff:
${diff}

Entity under test:
${entity.body}

Check for:
- Coverage: are happy path, boundaries, and error paths all exercised?
- Assertion correctness: do assertions actually verify the right thing?
- Independence: do tests share mutable state that could cause order-dependence?
- Completeness: are there obvious cases missing given the implementation?

Respond with APPROVED or CHANGES_NEEDED (specific gaps with line references).`

Stage 3 — Execute tests locally

The framework applies the validated diff and runs only the affected test file (not the full suite) to keep the feedback loop fast. Test runner output — pass/fail counts, failure messages, and stack traces — is captured verbatim for the fix loop.

Stage 4 — Fix loop with escalation

On any test failure, the framework enters a repair loop. The local model attempts fixes for up to 3 rounds; the 4th failure escalates to Claude. After Claude's fix is applied and executed, the counter resets and the local-fix cycle resumes for any subsequent failures. 

// Fix loop state machine
localAttempts = 0

loop:
  execute tests
  if all pass → done

  if localAttempts < 3:
    // Local fix round
    local model receives: failing test body, error output + stack trace, entity under test
    local model produces: fix diff (may fix test OR implementation — model decides)
    apply diff
    localAttempts++
    goto loop

  else:
    // Escalate to Claude (4th failure)
    Claude receives:
      - failing test body (full)
      - entity under test (full body)
      - error output + stack trace from all 3 local attempts
      - each local fix diff that was tried
    Claude produces: fix diff
    apply diff
    localAttempts = 0   // reset — local cycles resume after Claude's fix
    claudeRounds++
    goto loop

// Terminal conditions (exit loop without surfacing as error):
//   all tests pass → done, plan_step_update(status: 'done')
//
// Terminal conditions (surface to user):
//   claudeRounds >= 2 AND tests still fail → show last diff + all errors, ask user to decide:
//     [a] accept current state  [b] discard changes  [c] open editor
//   localAttempts >= 3 AND Claude unavailable → same prompt as above

The fix diff may target either the test file or the implementation — the model decides which is wrong. If the implementation is changed, the indexer re-indexes the affected file before the next execution round.

Context budget for escalation: Claude receives only the single failing test, the entity under test, and the three prior diffs with their error outputs — not the full test suite or codebase. Each prior attempt adds roughly one diff + one error block, keeping the escalation payload small even after 3 local rounds.

debug

The user wants to diagnose and fix a failing or incorrect behaviour — a crash, a test failure, a bad response, or an unexpected side effect. Debug is the most tool-heavy intent: the local model drives a tool loop gathering evidence (logs, traces, code), analyses it locally, and only touches Claude to validate mutating commands or when it is stuck.

Tool loop

The local model iterates freely over read-only tools at zero Claude cost, following the tool execution pipeline rules:

  • Auto-execute (no validation): kubectl get/logs/describe, git log/diff/blame, Read, Grep, Glob, graph_*, Bash (test runner / build / lint)
  • Validate before execute: kubectl apply/exec/delete, git commit/reset, Write, Edit, arbitrary shell
// Typical debug tool sequence — kubectl scenario
1. graph_search("payment service")          → auto-execute, local analysis
2. kubectl get pods -n prod                 → auto-execute, local analysis
3. kubectl logs payment-svc --tail=100      → auto-execute, local analysis
4. kubectl describe pod payment-svc-7d9f   → auto-execute, local analysis
5. Read(src/payment/handler.ts)             → auto-execute, local analysis
6. Grep("processPayment", src/)             → auto-execute, local analysis
   → local LLM identifies root cause
7. Edit(src/payment/handler.ts, fix)        → VALIDATE (Claude/Haiku, ~$0.0004)
8. Bash("kubectl apply -f deploy/...")      → VALIDATE (Claude/Haiku, ~$0.0004)
   ─────────────────────────────────────────────────────────
   Total Claude cost for session: ~$0.0008

Stuck escalation

If the local model cannot identify a root cause after 2 tool-loop turns with evidence in hand, it writes a ≤200-token summary of what it found and what it needs, which is sent to Claude/Haiku for direction. Raw tool output (logs, traces, file contents) is never forwarded to Claude. 

// Stuck escalation prompt (local LLM authors this)
`Debugging: payment-svc pods crash on startup.

Found:
- OOMKilled in describe output; memory limit 256Mi
- heap allocation spike in logs at PaymentHandler.processPayment line 42
- No recent changes to memory limits in git log

Need: guidance on whether to raise the limit, fix the allocation, or both.`

→ Claude/Haiku responds with direction (~200 tokens)
→ local LLM continues tool loop with that direction

Fix validation

Once the local model produces a fix diff, it follows the same Stage 2 validation as implement: Claude/Haiku receives the diff plus the touched entity bodies (diff-derived, minimal context) and checks correctness before the fix is written to disk. If the fix requires a mutating shell command (apply, restart, rollout), that command is validated separately via the call-validation pipeline.

Plan step lifecycle in debug

If the session is running under an active plan, the local model calls plan_get at session start (auto-execute, no cost) to identify the relevant plan step. On successful fix: plan_step_update(status: 'done'). If the debug session ends with the bug unresolved, the step remains in_progress; the user is shown the step ID and told they can resume in a new session.

review

The user wants a critique of existing code — correctness, style, security, or performance. Review is Claude-primary: the local model assembles and scopes the context bundle, Claude does the analysis. The local model never generates a review draft; its only role is deciding what to include.

Context assembly (local model)

The local model uses read-only tools (all auto-execute, zero Claude cost) to build the review payload:

  • If a diff is in scope (staged changes, a PR, a specific commit): git diff or git show — the diff is the primary input
  • Bodies of entities touched by the diff, fetched via graph_entity
  • Signatures of first-degree callers and callees (not bodies) — for call-site compatibility context
  • If no diff (reviewing an existing file or function): the entity body + its immediate neighbours' signatures from the graph

The local model does not add commentary, score, or pre-judge the code. It constructs the bundle and forwards it to Claude.

Claude review (Sonnet by default)

// Review prompt sent to Claude
`Review the following code change. Be specific — cite line numbers or entity names.

${diff ? `Diff:\n${diff}` : `Code under review:\n${entityBody}`}

Touched entities (full bodies):
${touchedEntities.map(e => e.body).join('\n\n')}

Adjacent signatures (call-site context):
${neighbours.map(n => n.signature).join('\n')}

Review for:
- Correctness: logic errors, off-by-ones, unhandled edge cases
- Security: injection, unvalidated input, exposed internals, over-permissive access
- Performance: unnecessary allocations, N+1 queries, blocking calls in hot paths
- Style: consistency with surrounding code patterns shown
- Completeness: missing error handling, missing tests, undocumented public API

Format: one section per category, skip categories with no findings.
Severity: CRITICAL / WARN / NOTE per finding.`

Escalation to Opus

Sonnet handles most reviews well. The user can prefix with @opus to escalate to Opus for deep architectural reviews — large diffs touching many subsystems, security audits, or performance-critical paths where maximum reasoning depth is justified.

The user can request Claude review by appending --review to the command or replying review after seeing the draft. Claude/Haiku checks for accuracy, completeness, and clarity — one round, no retry loop.

Cross-cutting docs (ADRs, changelogs)

For ADRs and changelogs spanning multiple subsystems, the local model produces a draft and flags it as cross-cutting. The user is prompted to optionally escalate to Claude/Sonnet for enhancement — same two-stage local→Claude pattern as requirements and design. Escalation is never automatic.

graph

The user wants a pure structural answer from the code knowledge graph — who calls this, what does this depend on, what would break if this changed. No LLM is involved at any point. The query is resolved entirely from Kuzu and LanceDB and the result is formatted and returned directly.

Escalation

Graph stays LLM-free by design. If the user's question cannot be answered by a structural query alone — e.g. "why does X call Y?" or "is this dependency intentional?" — the handler recognises it as a research intent and re-routes rather than attempting to answer from graph structure alone.

Model configuration

Claude model selection is configurable per intent role in ~/.insrc/config.json. The config uses a two-level structure: tier defaults apply to all roles in that tier, and per-role overrides take precedence. This lets users tune cost vs. quality globally or surgically. 

// ~/.insrc/config.json — model configuration
{
  "models": {
    // Tier defaults — apply to all roles assigned to that tier
    "tiers": {
      "fast":     "claude-haiku-4-5",    // call validation, diff validation, fix escalation
      "standard": "claude-sonnet-4-6",   // requirements, design, plan, review
      "powerful": "claude-opus-4-6"      // explicit @opus escalation only
    },

    // Per-role overrides — take precedence over tier defaults
    "roles": {
      "call_validation":       "fast",      // validate mutating tool call requests
      "implement.validate":    "fast",      // Stage 2 diff correctness check
      "implement.escalation":  "fast",      // fix direction after 3 local failures
      "refactor.validate":     "fast",      // Stage 2 behaviour-equivalence check
      "test.validate":         "fast",      // Stage 2 test quality check
      "test.escalation":       "fast",      // fix direction on 4th failure
      "debug.validate":        "fast",      // mutating command validation
      "debug.escalation":      "fast",      // stuck diagnosis direction
      "requirements":          "standard",  // sketch enhancement
      "design":                "standard",  // architecture enhancement
      "plan":                  "standard",  // plan ordering and dependency resolution
      "research.escalation":   "standard",  // cross-repo or low-confidence answers
      "review":                "standard",  // full code review
      "document.review":       "fast",      // user-requested Claude doc review (opt-in)
      "document.cross_cutting": "standard"  // cross-cutting doc direction for multi-file doc tasks
    }
  }
}

To upgrade all validation to Sonnet (higher quality, ~5× more expensive per call): 

{ "models": { "tiers": { "fast": "claude-sonnet-4-6" } } }

To upgrade only code review to Opus: 

{ "models": { "roles": { "review": "powerful" } } }

To use a specific model ID directly for a role (bypassing tier names): 

{ "models": { "roles": { "review": "claude-opus-4-6" } } }
Opus default off: powerful tier is never assigned to any role by default. It is only reachable via an explicit role override or the @opus user prefix. This prevents accidental Opus spend on routine validation calls.

1. Explicit — user-directed

User prefixes a message with @claude or @opus to bypass routing:

When the local agent determines a task exceeds its confidence or scope, it pauses and asks rather than producing a low-quality answer:

Context Assembly

Neither provider receives raw file dumps. The agent assembles a structured context package from the Code Knowledge Graph for each task. This keeps prompts tight, deterministic, and cost-efficient.

Small, focused: 3–10 entities, direct neighbors only.

Context Window & Memory

The default agent model (qwen3-coder) runs with a 64K-token context window — a property of the model, configurable in the Ollama Modelfile but defaulting to 64K for this model class. This sounds generous but is quickly consumed by system prompt, conversation history, code context, and response headroom all competing for the same budget. A naive sliding-window approach discards turns arbitrarily and loses earlier context permanently. The layered memory model below allocates the budget deliberately, using recency for near-term turns and semantic retrieval for older history.

The index is the memory. The Kuzu Code Knowledge Graph is the agent's persistent long-term memory for code structure. The context window only needs to hold what is immediately relevant to the current turn — everything else is re-queryable on demand.

Recency-Weighted Recent Turns (L3a)

Near-term turns are the most contextually connected. Rather than a flat window, L3a weights turns by age — the most recent turn is included in full, older turns are progressively truncated:

Turn ageInclusion
Turn −1 (most recent)Full text
Turn −275% — trim assistant response tail
Turn −350% — user message + first half of response
Turn −425% — user message + first paragraph of response
Turn −5User message only
Turn −6+Evicted to L2 (summarized) or L3b (embedded)

Semantic History Retrieval (L3b)

Every turn is embedded and stored in LanceDB. When assembling context, the current user message is used to retrieve the most semantically relevant older turns — regardless of when they occurred. A turn from 30 exchanges ago resurfaces if it is relevant to the current question.

History embedding is skipped for the first 5 turns of a session (pure L3a recency is sufficient and cheaper when history is short).

Storage — per turn (end of turn, non-blocking)

// memory.ts — called after assistant response is streamed
async function storeTurn(session: Session, turn: ConversationTurn): Promise<void> {
  if (session.turnIndex < 5) return;  // skip embedding for early turns

  const vec = await embed(turn.userMessage);  // embed user message only

  // LanceDB: upsert into conversation_turns table
  await conversationTurnsTable.add([{
    id:        `${session.id}:${session.turnIndex}`,
    sessionId: session.id,
    idx:       session.turnIndex,
    user:      turn.userMessage,
    assistant: turn.assistantResponse,
    vector:    vec,            // float32[2048] for semantic retrieval
    entities:  turn.entities, // graph entity IDs referenced in this turn
    createdAt: new Date().toISOString(),
  }]);
}

Retrieval — per turn (start of turn, on critical path ~5ms)

// memory.ts — called before assembling context
async function retrieveSemanticHistory(
  session: Session,
  vec: number[]
): Promise<ConversationTurn[]> {
  if (session.turnIndex < 5) return [];  // nothing stored yet

  // LanceDB ANN vector search scoped to this session, excluding L3a window
  const minIdx = session.turnIndex - 5;  // exclude last 5 turns (already in L3a)
  return conversationTurnsTable
    .vectorSearch(vec)
    .where(`sessionId = '${session.id}' AND idx < ${minIdx}`)
    .distanceType('cosine')
    .limit(4)
    .toArray();
}

Rolling Summarization (L2)

When L3a overflows (turn −6 is evicted), that turn is compressed into the rolling L2 summary by the local model. The summary accumulates key decisions, entity names, and conclusions — not raw dialogue: 

// memory.ts
async function evictToSummary(session: Session): Promise<void> {
  const turn = session.recentTurns.pop();  // evict oldest (turn -6)

  const prompt = [
    { role: 'system', content:
        'Update the running summary to include the new exchange. ' +
        'Preserve: decisions made, file names, entity names, open questions. ' +
        'Be concise. Max 3 sentences total.' },
    { role: 'user', content:
        `Current summary:\n${session.summary}\n\nNew exchange:\n${formatTurn(turn)}` }

  session.summary = await ollamaProvider.complete(prompt, { maxTokens: 250 });
}

Dynamic Graph Retrieval (L4)

Task context is not accumulated — it is re-assembled from the code graph each turn. This keeps L4 fresh and avoids context drift from stale code that has since been edited by the indexer daemon. 

// context.ts
async function assembleContext(turn: string, session: Session): Promise<AssembledContext> {
  // Single embed call serves both code search (L4) and history retrieval (L3b)
  const vec = await embed(turn);

  const [codeHits, historyHits] = await Promise.all([
    db.searchCode(vec, { repos: session.closureRepos, limit: 8 }),   // L4
    retrieveSemanticHistory(session, vec),                            // L3b
  ]);

  const neighbors  = await db.neighbors(codeHits.map(h => h.id));
  const entities   = buildEntityContext(codeHits, neighbors, session.seenEntities);

  return fitToBudget({
    system:    session.systemContext,                    // L1
    summary:   session.summary,                          // L2
    recent:    weightedRecent(session.recentTurns),      // L3a
    semantic:  historyHits,                              // L3b
    code:      entities,                                 // L4
  });
}
Single embed per turn: The same embedding vector is reused for both the code graph search (L4) and the conversation history search (L3b). Only one Ollama call is needed.

Token Budget Enforcement

Context assembly respects hard per-layer ceilings. When a layer overflows, content is dropped in a defined priority order to preserve the most critical context. 

// budget.ts
const TOKEN_BUDGET = {
  system:   1_000,
  summary:  3_000,
  recent:   4_000,   // L3a
  semantic: 4_000,   // L3b

Overflow priority (what gets dropped first)

Each layer has a fixed ceiling. When assembling context exceeds the total budget, content is shed in this order — lowest priority first:

1st droppedL4 — code entitiesDrop lowest vector-score entities first; always keep entities directly named in the current turn
2nd droppedL3b — semantic historyDrop lowest-similarity retrieved turns first

History Retention & Pruning

Raw turn embeddings are ephemeral. At session end, the final L2 summary is promoted to a persistent conversation_session record and raw turns are deleted. This gives cross-session continuity without unbounded storage growth.

Session close

// session.ts — on disconnect or explicit /exit
async function closeSession(session: Session): Promise<void> {
  const now = new Date().toISOString();
  const expiresAt = new Date(Date.now() + 30 * 86_400_000).toISOString(); // 30 days

  // Promote summary to persistent LanceDB store
  await conversationSessionsTable.add([{
    id:           session.id,
    repo:         session.repoId,
    summary:      session.summary,
    seenEntities: [...session.seenEntities],
    createdAt:    now,
    expiresAt:    expiresAt,
    vector:       await embed(session.summary),  // for semantic cross-session seeding
  }]);

  // Delete all raw turns — summary is the durable artifact
  await conversationTurnsTable.delete(`sessionId = '${session.id}'`);
}

Cross-session seeding

On session start, recent session summaries for the same repo seed the initial L2:

// session.ts — on session start
// Embed the user's opening message as query vector for semantic similarity

// lifecycle.ts — runs daily via setInterval
async function pruneConversationHistory(): Promise<void> {
  const now = new Date().toISOString();

  // Delete expired summaries (LanceDB filter delete)
  await conversationSessionsTable.delete(`expiresAt < '${now}'`);

  // Cap at 20 summaries per repo — delete oldest beyond cap
  // Note: Map.groupBy requires Node.js ≥ 21. Use Object.groupBy polyfill for Node 18/20.
  const allSessions = await conversationSessionsTable.query().toArray();
  const byRepo = Map.groupBy(allSessions, s => s.repo);
  for (const [repo, sessions] of byRepo) {
    const sorted = sessions.sort((a, b) => b.createdAt.localeCompare(a.createdAt));
    const toDelete = sorted.slice(20).map(s => s.id);
    if (toDelete.length > 0) {
      const ids = toDelete.map(id => `'${id}'`).join(',');
      await conversationSessionsTable.delete(`repo = '${repo}' AND id IN (${ids})`);
    }
  }
}

Model Bootstrap

On daemon startup, after the embedding model check, the agent model is also verified and pulled if missing. Both checks run sequentially before any index or agent jobs are processed. 

// lifecycle.ts
const MODELS = {
  embedding: 'qwen3-embedding:0.6b',
  agent:     'qwen3-coder:latest',
};

async function ensureModels(ollama: Ollama): Promise<void> {
  const { models } = await ollama.list();
  const installed = new Set(models.map(m => m.name));

  for (const [role, model] of Object.entries(MODELS)) {
    if (installed.has(model)) continue;
    log.info(`Pulling ${role} model: ${model}`);
    const stream = await ollama.pull({ model, stream: true });
    for await (const event of stream) {
      const pct = event.total ? Math.round((event.completed / event.total) * 100) : 0;
      setDaemonState({ modelPull: { role, model, pct } });
    }
    log.info(`${model} ready`);
  }
}

Plan pre-flight — local-only condensed path

When the user issues a plan intent, the local model runs a condensed pre-flight analysis before Claude is invoked (to assemble the context package):

  1. Entity resolution — identify entities touched by the feature request using graph traversal only (auto-execute, no LLM)
  2. Scope estimate — count files, repos, and estimate token budget for the full plan context
  3. Local sketch — local model generates a rough ordered list of steps (~200 tokens) to give Claude a starting structure
  4. Claude enhancement — Claude receives: user request + entity signatures + local sketch; produces the final plan with dependencies, estimates, and ordered steps
  5. Plan persisted — plan written to Kuzu Plan / PlanStep tables via plan_step_update

If Ollama is unavailable, step 3 is skipped and Claude receives only steps 1–2 output. If the daemon is unavailable, plan persistence is skipped and the plan is printed to the terminal only (user can copy it).

The re-index is file-scoped (only the changed files are re-parsed and their graph nodes updated) rather than a full repo scan, keeping the feedback loop fast.

  1. Calls plan_get to fetch the current in-progress step
  2. Translates the step description into a task for the appropriate intent pipeline
  3. Executes the pipeline (implement / test / debug / etc.)
  4. On success: calls plan_step_update(status: 'done') and prints the next pending step
  5. On failure: calls plan_step_update(status: 'failed'), shows error, asks user to retry or skip

Open Questions

  1. Summarization model — Rolling summarization (L2 eviction) uses qwen3-coder:latest for convenience, but a smaller/faster model (e.g. qwen3:0.6b) would be more efficient for this internal housekeeping task. Worth benchmarking summary quality vs. latency tradeoff.
  2. Token counting accuracy — Accurate token counting requires the model's tokenizer. Approximating with character count (~4 chars/token) is fast but imprecise. Consider using tiktoken-style counting or a lightweight tokenizer aligned to Qwen's vocabulary to avoid budget overruns.
  3. Task classification accuracy — Simple heuristics (file count, repo count) may misclassify. A lightweight classifier model or rule-based NLU might improve routing over time. Consider logging escalation decisions for later review.
  4. Claude model configuration — Models are configurable per intent role in ~/.insrc/config.json. See the Model configuration section.
  5. Session persistence — Session summaries are persisted to LanceDB conversation_sessions as a lightweight "conversation memory" per repo, giving the agent recall across sessions without full history replay (30-day TTL, capped at 20 per repo).
  6. Streaming UX — Both providers support streaming. The REPL should stream tokens to the terminal in real time. Escalation announcement must appear before the first token from Claude.
