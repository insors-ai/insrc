Agent Personas

BRAINSTORM Brainstorm Agent

insrc coding agent — persona detail — March 2026

Overview

The brainstorm agent facilitates interactive idea exploration sessions that incrementally build a structured requirements specification as the conversation unfolds. Unlike the designer agent (which refines a fixed set of requirements) or the planner (which produces a step-by-step execution plan), the brainstorm agent operates in a fluid, iterative loop where ideas are generated, explored, challenged, and gradually crystallized into formal requirements.

The key insight: brainstorming and requirements capture are not sequential phases—they happen simultaneously. Every idea explored either strengthens an existing requirement, spawns a new one, or gets discarded. The running requirements spec is a live artifact that evolves with each brainstorming round.

Motivation

The current designer agent assumes the user already knows roughly what they want—it extracts, refines, and details pre-formed requirements. But many engineering tasks start with a vague problem statement: "we need better error handling", "auth is broken", "the indexer is too slow". These need exploratory thinking before requirements can be written.

The brainstorm agent fills this gap by providing structured divergent/convergent thinking cycles that produce a requirements document ready for the designer or planner pipelines.

Design Goals

  • Interactive, not batch — the user is an active participant, not a reviewer of finished output
  • Incremental spec — a requirements document builds up live during the session, never assembled after-the-fact
  • Codebase-grounded — ideas are anchored to real entities, not hypothetical abstractions
  • Diverge then converge — explicit thinking modes prevent premature commitment
  • Composable output — the resulting spec feeds directly into the designer or planner agent
  • Resumable — sessions can be paused and continued via the framework checkpoint system

Core Concepts

Idea Lifecycle

Every idea in the brainstorming session follows a lifecycle:

+-----------+ | proposed | <-- generated by LLM or user +-----+-----+ | +---------------+---------------+ | | | +-----v-----+ +-----v-----+ +----v------+ | accepted | | rejected | | parked | +-----+-----+ +-----------+ +-----------+ | (revisit later) +-----v-----+ | refinement | <-- explored further, challenged +-----+-----+ | +------- -+--------+ | | +-----v-----+ +-----v-----+ | promoted | | merged | <-- absorbed into existing req +-----+-----+ +-----------+ | +-----v-----+ | requirement | <-- now part of the live spec +-----------+

Proposed: initial idea from LLM divergent generation or user input.
Accepted: user signals interest; idea enters the active exploration pool.
Rejected: explicitly discarded; stored for context but not explored further.
Parked: interesting but not now; may be revisited in a later round.
Refinement: accepted idea is challenged, deepened, and sharpened.
Promoted: refined idea graduates to a formal requirement in the spec.
Merged: idea is absorbed into an existing requirement (strengthens it, adds acceptance criteria).

Live Requirements Spec

The requirements spec is a structured document that grows incrementally:

  • Each promoted idea becomes a numbered requirement with type, rationale, and acceptance criteria
  • Each merged idea appends criteria or notes to an existing requirement
  • Requirements are grouped into themes discovered during clustering
  • The spec maintains a revision log showing when and why each requirement was added or modified
  • At any gate, the user sees the current spec alongside new proposals

Thinking Modes

The agent alternates between two explicit thinking modes, preventing premature convergence:

Mode Goal LLM Behavior User Role
Diverge Maximize idea surface area Generate provocations, what-ifs, edge cases, analogies, inversions React, add own ideas, pick directions to explore
Converge Cluster, refine, and promote Find patterns, propose themes, sharpen ideas into requirements Approve/reject promotions, merge duplicates, prioritize
The agent starts in diverge mode and switches to converge after each idea burst. Multiple diverge/converge cycles can occur in a single session. The user can also explicitly request a mode switch at any gate.

Data Model

Types

/** Input from the REPL to start a brainstorming session. */
interface BrainstormInput {
  message: string;                    // problem statement or seed topic
  codeContext: string;                // pre-fetched code context from graph
  existingSpec?: string;              // optional: continue from prior spec
  session: {
    repoPath: string;
    closureRepos: string[];
  };
}

/** A single idea in the brainstorming pool. */
interface Idea {
  id: string;                         // deterministic: SHA256(session + index)
  index: number;                      // 1-based insertion order
  text: string;                       // the idea statement
  status: IdeaStatus;
  source: 'llm' | 'user';            // who proposed it
  round: number;                      // which diverge round produced it
  parentId?: string;                  // if refined from another idea
  promotedTo?: string;                // requirement ID if promoted
  mergedInto?: string;                // requirement ID if merged
  tags: string[];                     // freeform labels for clustering
  rationale?: string;                 // why this idea matters (from refinement)
  codeRefs: string[];                 // entity IDs grounding the idea
}

type IdeaStatus =
  | 'proposed'
  | 'accepted'
  | 'rejected'
  | 'parked'
  | 'refining'
  | 'promoted'
  | 'merged';

/** A theme grouping related ideas. */
interface Theme {
  id: string;
  name: string;
  description: string;
  ideaIds: string[];                  // ideas belonging to this theme
  requirementIds: string[];           // requirements generated from this theme
}

/** A requirement in the live spec. */
interface SpecRequirement {
  id: string;                         // deterministic: SHA256(session + 'req' + index)
  index: number;                      // display order
  statement: string;                  // formal requirement text
  type: 'functional' | 'non-functional' | 'constraint';
  priority: 'must' | 'should' | 'could';
  themeId: string;                    // parent theme
  acceptanceCriteria: string[];       // testable conditions
  rationale: string;                  // why this requirement exists
  sourceIdeaIds: string[];            // traceability back to ideas
  codeRefs: string[];                 // grounding entities
  revision: number;                   // bumped on each modification
  addedInRound: number;               // brainstorm round when first created
}

/** Revision log entry. */
interface SpecRevision {
  round: number;
  requirementId: string;
  action: 'added' | 'modified' | 'removed' | 'merged';
  detail: string;                     // human-readable change description
}

/** Final output of the brainstorm agent. */
interface BrainstormResult {
  kind: 'brainstorm-spec';
  output: string;                     // rendered HTML/markdown spec document
  requirements: SpecRequirement[];
  themes: Theme[];
  ideas: Idea[];                      // full idea pool for traceability
  revisions: SpecRevision[];
  summary: string;                    // L2 carry-forward tag
  stats: {
    rounds: number;
    totalIdeas: number;
    promoted: number;
    merged: number;
    rejected: number;
    parked: number;
  };
}

BrainstormState

/** Agent state — serialized at each checkpoint. */
interface BrainstormState extends AgentState {
  input: {
    message: string;
    codeContext: string;
    existingSpec?: string;
    repoPath: string;
    closureRepos: string[];
  };

  // Session tracking
  round: number;                      // current diverge/converge cycle (1-based)
  mode: 'diverge' | 'converge';      // current thinking mode
  maxRounds: number;                  // configured limit (default: 5)

  // Idea pool
  ideas: Idea[];
  nextIdeaIndex: number;              // monotonic counter

  // Themes (populated after first converge)
  themes: Theme[];

  // Live requirements spec
  requirements: SpecRequirement[];
  nextReqIndex: number;
  revisions: SpecRevision[];

  // Context management
  codebaseFindings: string;           // daemon search results
  compressedHistory: string;          // prior round summaries for context budget
  seedAnalysis: string;               // initial problem decomposition
  recentFeedback?: string;            // last user direction from gate feedback

  // Provider override (@-mention)
  providerOverride?: ProviderOverride;  // set by @haiku/@sonnet/@opus/@local at gates

  // Edit tracking
  editRounds: Record<string, number>;

  // Output
  assembledOutput?: string;
  summary?: string;
}

Agent Pipeline

Step Overview

Step Mode Provider Description
seed — LOCAL Decompose problem, search codebase, generate initial idea burst
validate-seed — gate User reviews seed ideas: accept/reject/add own, set initial direction
diverge diverge LOCAL Generate new ideas via provocations, what-ifs, edge cases, analogies
react diverge gate User reacts to new ideas: accept/reject/park/add, steer exploration
converge converge LOCAL + CLAUDE Cluster accepted ideas into themes, propose requirement promotions
validate-convergence converge gate User approves themes, accepts/rejects promotions, merges duplicates
update-spec converge CLAUDE Write/update requirements in the live spec based on approved promotions
review-spec converge gate User reviews updated spec: approve, edit, request another round
iterate — — Decision point: loop back to diverge or proceed to finalize
finalize — CLAUDE Polish spec, add cross-cutting concerns, assemble final document

Step Details

seed

The seed step decomposes the user's problem statement into a structured analysis and generates an initial burst of 5–10 ideas:

  1. Analyze problem: local model parses the message into problem facets (scope, stakeholders, constraints, ambiguities)
  2. Search codebase: LLM plans categorized searches (reusing search-planner.ts from the designer), daemon returns relevant entities
  3. Generate seed ideas: local model produces initial idea burst, grounded in codebase findings
  4. Write seed analysis: stored as state.seedAnalysis for context in subsequent rounds
Provider: resolveStepProvider(ctx, state, 'seed')
Resolution: @-mention override → config brainstorm.seed → default: local.
Recommended: local (fast iteration, broad idea generation).

validate-seed

Gate where the user reviews the initial idea burst and problem decomposition.

Actions:

  • approve — accept all seed ideas, start diverging
  • select — user marks individual ideas as accepted/rejected/parked and may add their own ideas (provided as feedback text)
  • reframe — user provides a different angle; agent re-seeds

diverge

The divergent thinking step generates new ideas using multiple provocation techniques. For each round, the LLM applies 2–3 techniques selected based on the current idea landscape:

Technique Description
What-if inversion Flip an assumption: "what if we didn't have X?"
Edge case mining Explore boundaries: "what happens when input is empty / huge / malformed?"
Analogy transfer Borrow from a similar domain: "how does X solve this in their context?"
Stakeholder lens View from different users: "what does the admin / new user / API consumer need?"
Constraint relaxation Remove a constraint: "if performance didn't matter, what would we build?"
Failure mode analysis Anticipate failures: "how could this go wrong in production?"
Decomposition Break a large idea into smaller pieces
Provider: resolveStepProvider(ctx, state, 'diverge')
Resolution: @-mention override → config brainstorm.diverge → default: local.
Higher temperature (0.8–0.9) for creative generation.

react

Gate where the user reacts to newly generated ideas.

Actions:

  • approve — accept all new ideas as-is
  • select — per-idea accept/reject/park decisions + user's own ideas (feedback text)
  • focus — user narrows the direction (feedback text steers next diverge round)
  • converge — user explicitly requests switching to converge mode
Auto-converge heuristic: if accepted ideas ≥ 8 or the user has done 2+ diverge rounds without converging, the agent suggests switching to converge mode. The user can override this at the gate.

converge

The convergent thinking step analyzes the accepted idea pool and proposes structure:

  1. Cluster: local model groups ideas by affinity into themes
  2. Deduplicate: identify overlapping ideas, propose merges
  3. Propose promotions: Claude evaluates which ideas are mature enough to become formal requirements, drafts requirement text + acceptance criteria
  4. Identify gaps: flag themes with thin coverage, suggest areas for further divergence
Dual-model strategy: LOCAL handles clustering via resolveStepProvider(ctx, state, 'cluster').
CLAUDE handles promotion via resolveStepProvider(ctx, state, 'promote').
Both are overridable via @-mentions or config. When the provider changes between cluster and promote, a full context handoff is built from state (see Context Handoff).

validate-convergence

Gate where the user reviews proposed themes and promotions.

Actions:

  • approve — accept all proposed themes and promotions
  • edit — modify themes, accept/reject individual promotions, merge ideas (feedback text)
  • diverge — user wants more exploration before committing; loop back to diverge

update-spec

Writes approved promotions into the live requirements spec. This is where the incremental spec building happens:

  1. Promoted ideas become new SpecRequirement entries
  2. Merged ideas append acceptance criteria to existing requirements
  3. Theme assignments are updated
  4. Revision log entries are written
  5. Claude polishes requirement text for clarity and consistency with existing spec entries
Provider: resolveStepProvider(ctx, state, 'update-spec')
Resolution: @-mention override → config brainstorm.update-spec → default: Claude (fast).
Requirement writing quality matters for downstream consumption. Full context handoff includes current spec, approved promotions, revision log, and codebase findings.

review-spec

Gate where the user reviews the updated requirements spec.

Actions:

  • approve — spec is good; proceed to iterate decision
  • edit — user modifies requirement text directly (feedback contains edits)
  • continue — spec is fine, want more brainstorming rounds

iterate

Decision point that determines whether to loop or finalize:

  • If state.round < state.maxRounds AND (user chose "continue" OR parked ideas remain OR gap analysis found thin themes): loop to diverge
  • If user chose "approve" at review-spec OR round limit reached: proceed to finalize
  • Compress current round into state.compressedHistory for context budget before next round

finalize

Assembles the complete requirements specification document:

  1. Claude reviews full spec for cross-cutting concerns (consistency, completeness, contradictions)
  2. Adds non-functional requirements inferred from the discussion (performance, security, etc.)
  3. Generates a prioritized summary
  4. Renders into the output template (HTML or markdown)
  5. Writes artifact and emits done message

Validation Gates

The brainstorm agent has 4 gate stages, compared to the designer's 3:

Gate Stage Tag Frequency Actions
validate-seed seed Once approve, select, reframe
react diverge-N 1–3 per round approve, select, focus, converge
validate-convergence converge-N Once per round approve, edit, diverge
review-spec spec-N Once per round approve, edit, continue
Edit round limits: Each gate stage tracks edit rounds via state.editRounds[stageTag]. Maximum 3 edit rounds per stage per brainstorm round, consistent with the designer agent. After 3 edits, the gate auto-approves with a warning.

Incremental Spec Building

Spec Evolution Strategy

The spec is not written once—it evolves through multiple update cycles. Each update-spec step receives the full current spec plus the batch of approved promotions/merges from the latest converge round.

Round 1 Round 2 Round 3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ diverge │ │ diverge │ │ diverge │ │ 8 ideas │ │ 6 ideas │ │ 4 ideas │ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ ┌────v─────┐ ┌────v─────┐ ┌────v─────┐ │ converge │ │ converge │ │ converge │ │ 3 themes │ │ +1 theme │ │ refine │ │ 4 promos │ │ 3 promos │ │ 2 promos │ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ ┌────v──────────┐ ┌────────v──────────┐ ┌─────────v─────────┐ │ Spec v1 │ │ Spec v2 │ │ Spec v3 (final) │ │ 3 themes │ │ 4 themes │ │ 4 themes │ │ 4 requirements│ │ 7 requirements │ │ 9 requirements │ │ 0 revisions │ │ 3 revisions │ │ 5 revisions │ └───────────────┘ └───────────────────┘ └───────────────────┘

The update-spec step uses Claude to ensure new requirements are consistent with existing ones. The prompt includes:

  • The full current spec (themes + requirements)
  • The batch of promoted/merged ideas with their rationale
  • Instructions to check for contradictions, redundancy, and gaps
  • The revision log so far (for traceability)

Conflict Resolution

When a new idea contradicts an existing requirement:

  1. Claude flags the conflict in the convergence output
  2. The validate-convergence gate presents both sides to the user
  3. User decides: keep original, replace, split into alternatives, or park for later
  4. Decision is recorded in the revision log

Provider Resolution

Every LLM call in the brainstorm agent goes through a three-tier resolution chain that determines which provider and model to use. This is critical because brainstorming sessions are long-running and interactive — the user may want to switch providers mid-session based on how the conversation is going.

Resolution Priority

Provider resolution follows a strict priority order:

Priority 1: @-mention override (user typed @haiku, @sonnet, @opus, @local at a gate) │ │ not present? ▼ Priority 2: config binding (config.models.agents.brainstorm.<step>) │ │ not configured? ▼ Priority 3: step default (hardcoded per step — local or claude)
Priority Source Scope Persistence
1 (highest) @-mention at gate Next LLM step only (one-shot), or @sticky for remainder of session Stored in state.providerOverride
2 ~/.insrc/config.json Permanent per-step binding Config file
3 (lowest) Step default Hardcoded in step implementation Code

@-Mention Overrides

At any validation gate, the user can prefix their feedback with a provider @-mention to override the provider for the next LLM step:

Mention Resolves To Use Case
@local Ollama (config.models.local) Fast iteration, keep it local
@haiku Claude Haiku (tiers.fast) Quick Claude assist without heavy cost
@sonnet Claude Sonnet (tiers.standard) Balanced quality for requirement writing
@opus Claude Opus (tiers.powerful) Maximum quality for complex analysis
@sticky <provider> Any of the above Lock provider for all remaining steps until cleared
@clear — Remove sticky override, revert to config/defaults

Examples at gates:

// At the react gate — user wants Claude to do the next diverge
> @sonnet focus on error handling edge cases

// At validate-convergence — user wants local for the next cluster pass
> @local approve

// Lock all subsequent steps to Haiku for the rest of the session
> @sticky haiku approve

// Clear a sticky override
> @clear continue

Parsing @-mentions from gate feedback

/** Parse provider override from gate reply feedback text. */
function parseProviderMention(feedback: string): {
  override: ProviderOverride | null;
  cleanFeedback: string;         // feedback with @-mention stripped
} {
  const mentionRe = /^@(local|haiku|sonnet|opus|sticky|clear)\b\s*/i;
  const stickyRe = /^@sticky\s+(local|haiku|sonnet|opus)\b\s*/i;

  const stickyMatch = feedback.match(stickyRe);
  if (stickyMatch) {
    return {
      override: { provider: resolveProvider(stickyMatch[1]), sticky: true },
      cleanFeedback: feedback.slice(stickyMatch[0].length),
    };
  }

  const match = feedback.match(mentionRe);
  if (!match) return { override: null, cleanFeedback: feedback };

  if (match[1] === 'clear') {
    return { override: { provider: null, sticky: false }, cleanFeedback: feedback.slice(match[0].length) };
  }

  return {
    override: { provider: resolveProvider(match[1]), sticky: false },
    cleanFeedback: feedback.slice(match[0].length),
  };
}

interface ProviderOverride {
  provider: { kind: 'local' } | { kind: 'claude'; tier: string } | null;
  sticky: boolean;
}

Resolution in step implementation

/** Resolve the LLM provider for this step, respecting override chain. */
function resolveStepProvider(
  ctx: StepContext,
  state: BrainstormState,
  stepName: string,
): LLMProvider {
  // Priority 1: @-mention override (one-shot or sticky)
  const override = state.providerOverride;
  if (override?.provider) {
    if (override.provider.kind === 'local') {
      return ctx.providers.local;
    }
    // Claude with specific tier
    const apiKey = ctx.config.keys.anthropic ?? process.env['ANTHROPIC_API_KEY'];
    if (apiKey) {
      return new ClaudeProvider({
        model: ctx.config.models.tiers[override.provider.tier as keyof typeof ctx.config.models.tiers],
        apiKey,
      });
    }
    // Fall through to config/default if no API key
  }

  // Priority 2 + 3: config binding → step default (handled by ProviderResolver)
  return ctx.providers.resolve('brainstorm', stepName);
}
One-shot vs sticky: After a non-sticky @-mention is consumed by one LLM call, state.providerOverride is cleared. Sticky overrides persist across all subsequent steps until the user sends @clear or the session ends.

Context Handoff

When the brainstorm agent switches between providers — whether due to config bindings (e.g., cluster on local, promote on Claude) or @-mention overrides — it must build a complete, self-contained context for the target provider. The target provider has no memory of prior steps. Each LLM call is stateless; continuity lives in the agent state, not in message history.

Anti-pattern: Forwarding only the last step's output to the next provider. If the diverge step ran on local and produced 5 ideas, and the converge step runs on Claude, Claude needs the full session context — not just the 5 new ideas.

Handoff Protocol

Every LLM call in the brainstorm agent builds its context from the agent state using a context builder function specific to the step. The context builder assembles a message array tailored to:

  1. The target provider's context window — local (128K) vs Claude (200K)
  2. The step's information needs — what this step actually requires
  3. Budget-aware truncation — compress or omit lower-priority context to fit
/** Build context messages for an LLM call, adapted to the target provider. */
function buildStepContext(
  state: BrainstormState,
  stepName: string,
  provider: LLMProvider,
  config: AgentConfig,
): LLMMessage[] {
  const isLocal = !(provider instanceof ClaudeProvider);
  const contextWindow = isLocal
    ? config.models.context.local
    : config.models.context.claude;
  const maxOutput = isLocal
    ? config.models.context.localMaxOutput
    : config.models.context.claudeMaxOutput;
  const charBudget = (contextWindow - maxOutput) * config.models.context.charsPerToken;

  // Assemble context layers with priority-based truncation
  const layers = buildContextLayers(state, stepName, charBudget);
  return layers.toMessages();
}

Context Layers per Provider

Each step defines which context layers it needs. Layers are assembled in priority order; lower-priority layers are truncated or omitted when the budget is tight (especially on local's smaller 128K window):

Layer Priority Content Needed By
System prompt 1 (always) Step-specific system prompt with role and format instructions All steps
Problem statement 2 (always) state.input.message + state.seedAnalysis All steps
Current spec 3 (high) Serialized requirements + themes from state.requirements converge, update-spec, finalize
Active ideas 4 (high) Current round's accepted/proposed ideas (full text) diverge, converge
Codebase findings 5 (medium) state.codebaseFindings — entity signatures and relationships seed, diverge, converge
Compressed history 6 (medium) state.compressedHistory — summaries of prior rounds diverge (round > 1), converge, finalize
Rejected/parked ideas 7 (low) Summary of rejected ideas (to avoid regenerating them) diverge
Revision log 8 (low) state.revisions — spec change history update-spec, finalize
User feedback trail 9 (low) Recent gate feedback from the user (last 2–3 interactions) diverge, converge (steers direction)

      formatIdeasForContext(state.ideas.filter(i => i.status === 'accepted')),
      '',
      '## Themes Identified So Far',
      formatThemesForContext(state.themes),
      '',
      '## Ideas Proposed This Round (Pending Triage)',
      formatIdeasForContext(state.ideas.filter(i =>
        i.status === 'proposed' && i.round === state.round)),
      '',
      '## Recent User Direction',
      state.recentFeedback ?? '(no specific direction given)',
      '',
      '## Codebase Context',
      state.codebaseFindings,
    ].join('\n'),
  },
];

@-Mention mid-session: user forces @opus for finalize

// User at review-spec gate: "@opus approve"
// → state.providerOverride = { provider: { kind: 'claude', tier: 'powerful' }, sticky: false }
// → finalize step resolves to Claude Opus instead of default Claude Sonnet

// The context is built the SAME way — from state, not from prior messages.
// Opus gets the full picture regardless of what model ran previous steps.
const provider = resolveStepProvider(ctx, state, 'finalize');
// → ClaudeProvider({ model: 'claude-opus-4-6', apiKey })
const messages = buildStepContext(state, 'finalize', provider, ctx.config);
// → Full context with spec, all ideas, revision log, codebase findings
const result = await provider.complete(messages, { maxTokens: maxOutput });
Key principle: Context is always built from BrainstormState, never from the prior step's raw LLM output. The state is the single source of truth. When a step completes, its results are parsed and stored in state fields (ideas, themes, requirements). The next step reads state, not message history. This makes provider switching seamless — the target provider gets exactly the context it needs, formatted for its strengths.

Round compression for context budget

After each round, the iterate step compresses the round's activity into a concise summary appended to state.compressedHistory. This prevents context from growing unboundedly across rounds: 

/** Compress a completed round into a history summary. */
function compressRound(state: BrainstormState, round: number): string {
  const roundIdeas = state.ideas.filter(i => i.round === round);
  const promoted = roundIdeas.filter(i => i.status === 'promoted');
  const rejected = roundIdeas.filter(i => i.status === 'rejected');
  const newReqs = state.revisions
    .filter(r => r.round === round && r.action === 'added')
    .map(r => state.requirements.find(req => req.id === r.requirementId)?.statement)
    .filter(Boolean);

  return [
    `## Round ${round}`,
    `Generated ${roundIdeas.length} ideas, promoted ${promoted.length}, rejected ${rejected.length}.`,
    newReqs.length > 0
      ? `New requirements:\n${newReqs.map((r, i) => `  ${i + 1}. ${r}`).join('\n')}`
      : 'No new requirements this round.',
    state.recentFeedback
      ? `User direction: ${state.recentFeedback}`
      : '',
  ].filter(Boolean).join('\n');
}

Prompt Design

Seed System Prompt

You are a brainstorming facilitator analyzing a software engineering problem.

Decompose the problem into facets:
- Core challenge: what is the fundamental issue?
- Stakeholders: who is affected?
- Constraints: what limits the solution space?
- Ambiguities: what is unclear or underspecified?

Then generate 5-10 initial ideas. Each idea should:
- Be a single, concrete suggestion (not vague)
- Reference existing code entities where relevant
- Include 1-2 tags for later clustering

Format each idea as:
[N] IDEA_TEXT — tags: tag1, tag2 — refs: entity1, entity2

Diverge System Prompt

You are a creative thinking partner in a brainstorming session.

Current accepted ideas:
{acceptedIdeas}

Apply these provocation techniques to generate 3-5 new ideas:
{selectedTechniques}

Rules:
- Each idea must be DISTINCT from existing accepted ideas
- Ground ideas in the codebase where possible
- Be specific and concrete, not generic
- It's OK to suggest unconventional or ambitious ideas
- Tag each idea for clustering

Format: [N] IDEA_TEXT — tags: tag1, tag2 — refs: entity1, entity2

Converge System Prompt

You are organizing brainstorming results into a coherent structure.

## All Accepted Ideas
{acceptedIdeas}

## Existing Themes (if any)
{existingThemes}

## Existing Requirements (if any)
{existingRequirements}

Tasks:
1. GROUP ideas into themes (reuse existing themes where they fit)
2. IDENTIFY duplicates or near-duplicates → propose merges
3. EVALUATE which ideas are mature enough to become requirements
4. FLAG any ideas that contradict existing requirements
5. IDENTIFY gaps: themes with fewer than 2 ideas

For each promotion candidate, draft:
- Requirement statement (formal, testable)
- Type: functional | non-functional | constraint
- Priority: must | should | could
- Acceptance criteria (2-3 testable conditions)
- Rationale (why this matters)

Update-Spec System Prompt

You are updating a requirements specification with new entries.

## Current Spec
{currentSpec}

## Approved Promotions
{promotions}

## Approved Merges
{merges}

## Revision Log
{revisionLog}

Tasks:
1. ADD new requirements from promotions, assigned to their themes
2. UPDATE existing requirements with merged idea content (acceptance criteria, notes)
3. CHECK for contradictions between new and existing requirements — flag any found
4. ENSURE consistent language and format across all requirements
5. WRITE revision log entries for each change

Return the updated requirements and revision entries as structured JSON.

Codebase Integration

The brainstorm agent leverages the code knowledge graph to ground ideas in reality:

Seed Phase

  • Reuses search-planner.ts from the designer to plan categorized searches based on the problem statement
  • Daemon returns relevant entities (functions, classes, modules) with signatures and relationships
  • Entity context included in the seed prompt so initial ideas reference real code

Diverge Phase

  • When an accepted idea references entities, follow 1-hop relationships (callers, callees, implementors) to discover related code
  • Feed discovered entities into the next diverge prompt as "nearby code" to spark grounded ideas

Converge Phase

  • Cross-reference proposed requirements against the entity graph to validate feasibility
  • Flag requirements that reference non-existent entities or impossible relationships

RPC Calls

// Search for entities related to an idea
ctx.rpc('search', {
  query: idea.text,
  repos: state.input.closureRepos,
  kinds: ['function', 'class', 'module', 'interface'],
  limit: 10,
});

// Expand entity relationships
ctx.rpc('expand', {
  entityId: ref,
  hops: 1,
  relations: ['CALLS', 'IMPLEMENTS', 'DEPENDS_ON'],
});

Output Format

The brainstorm agent produces two artifacts:

1. Requirements Specification (primary output)

A structured HTML or markdown document with:

  • Problem Statement — seed analysis
  • Themes — each theme with its requirements grouped
  • Requirements — numbered, typed, prioritized, with acceptance criteria
  • Traceability Matrix — mapping requirements back to source ideas
  • Parked Ideas — ideas deferred for future sessions
  • Session Statistics — rounds, idea counts, promotion rates

2. Session Log (artifact)

A JSON artifact containing the full idea pool, themes, revision log, and round summaries for session resumption or audit.

Downstream Integration

Into Designer Agent

The brainstorm output's requirements array maps directly to the designer's DesignerInput.requirementsDoc. The designer skips its extraction step and uses the pre-validated requirements from the brainstorm session. 

// REPL dispatch: brainstorm → designer handoff

// Feed into designer

  message: originalMessage,
  requirementsDoc: specDoc,       // <-- brainstorm output
  codeContext: ctx,
  template: resolveTemplate({ format: 'html' }),
  intent: 'design',
  session: { repoPath, closureRepos },
};

Session Carry-Forward

The BrainstormResult.summary is stored as an L2 tag in the session, allowing the agent REPL to maintain context across follow-up turns: 

// L2 summary format
"Brainstormed 24 ideas across 3 rounds → 4 themes, 9 requirements (5 must, 3 should, 1 could).
 Key themes: auth middleware, token storage, session management, admin controls.
 3 ideas parked for future exploration."

This agent extends the insrc agent framework with a new persona focused on exploratory thinking. It reuses the existing checkpoint, gate, and provider infrastructure while introducing the novel concept of a live, incrementally-built requirements specification that evolves during the brainstorming session.
