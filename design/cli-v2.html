CLI Overview and Architecture

The insrc CLI is a local-first hybrid coding assistant designed to help developers understand, modify, test, and debug code efficiently. It operates in a client-server architecture where the CLI frontend communicates with a daemon backend via Unix socket JSON-RPC. The system identity is established at startup via buildSystemContext(), which scopes every session to one or more repositories and grants the agent access to the Code Knowledge Graph (Kuzu + LanceDB).

Design Philosophy

The CLI follows several key design principles:

  • Fast startup – Minimize initialization time for quick interaction; the daemon pre-warms connections so the CLI itself starts in milliseconds
  • Streaming – Provide real-time token-by-token feedback during long-running operations such as insrc ask and insrc implement
  • UNIX conventions – Embrace standard Unix practices for composability (exit codes, stderr for diagnostics, stdout for output)
  • Pipes – Support input/output piping for shell integration (e.g. cat error.log | insrc ask "what is wrong?")
  • --json – Output structured JSON for programmatic consumption by scripts and editors
  • Shell completion – Provide tab completion for improved usability across bash, zsh, and fish

Key Design Principles: The insrc CLI prioritizes speed, composability, and integration with existing Unix workflows. It is built to be fast, streaming, and JSON-oriented, making it suitable for both interactive use and automation pipelines.

Architecture Boundary

The architecture is separated into three main layers. The CLI frontend is intentionally thin — all heavy lifting (agent orchestration, graph access, LLM routing) happens inside the daemon so that startup latency stays low and multiple CLI invocations can share a warm context.

Startup and Session Flow

When the daemon starts it logs its model configuration immediately, giving operators a quick diagnostic snapshot before any request is served:

$ insrc daemon start
Generating CLI design document section by section...
  local: llama3.1:8b-instruct-q4_K_M
  claude: claude-3-5-sonnet-20241022
  sections: 12

Each CLI invocation resolves the Unix socket, opens a JSON-RPC connection, and dispatches to the appropriate pipeline. The OneShotOpts interface captures the full set of options a single non-interactive invocation needs:

// Simplified from src/agent/cli.ts — OneShotOpts
interface OneShotOpts {
  userMessage: string;
  repoPath: string;
  localProvider: LLMProvider;
  claudeProvider?: LLMProvider;
  contextConfig: ContextConfig;
  contextProvider: ContextProvider;
  onEvent: (event: AgentEvent) => void;
  json?: boolean;       // --json flag: emit newline-delimited JSON
  stream?: boolean;     // --stream flag: emit tokens as they arrive
}

A typical streaming session looks like this:

$ insrc ask "how does the stuck-detector escalate to Claude?"
⠸ Fetching context from knowledge graph...
⠼ Local model reasoning...

The stuck-detector (src/agent/tasks/stuck-detector.ts) monitors token
output from the local model. When it detects a loop or stall it calls
buildEscalationPrompt() with a summary of what the local model tried,
then routes that prompt to Claude for a ~200-token course-correction.

Claude's directive is injected back into the local model's next turn.

Hybrid LLM Routing: The daemon routes requests between the local model and Claude based on task complexity, attachment type, and escalation signals from the stuck-detector. The runForcedClaudePipeline() path is triggered when an attachment (e.g. an image or binary diff) cannot be processed by the local model.

JSON and Pipe Mode

Every command supports --json for machine-readable output. Events are emitted as newline-delimited JSON (NDJSON) on stdout, making it straightforward to pipe into jq or a language server plugin:

$ insrc ask "list all exported functions in src/agent" --json | jq '.content'
"The following functions are exported from src/agent/index.ts:
  startRepl (line 42)
  handlePlanCommand (line 782)
  ..."

$ echo "explain the design pipeline" | insrc ask --json
{"type":"token","content":"The design pipeline runs in two stages:"}
{"type":"token","content":" a local sketch followed by a Claude review."}
{"type":"done","elapsed_ms":1823}

Pipe Behaviour: When stdin is not a TTY, insrc automatically reads from stdin and appends the content to the user message. Streaming progress indicators are suppressed and all output goes to stdout so pipelines remain clean.
