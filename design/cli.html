CLI Design Document

insrc command-line interface — v0.1 — March 2026

Overview

The insrc CLI serves as the primary terminal interface for the insrc agent, designed with a philosophy of speed, simplicity, and UNIX conventions. It supports two distinct modes: an interactive REPL for ongoing conversation and one-shot command execution for scripting and quick tasks. Both modes connect to the same daemon backend, ensuring consistent behavior regardless of how the user interacts. The CLI acts as a thin UI layer, delegating all intelligence—classification, routing, tool execution, and context assembly—to the agent core and daemon. This design prioritizes fast startup times, streaming output, compatibility with pipes, a --json flag for programmatic consumption, and shell completion for major shells.

Architecture Boundary

The CLI (src/cli/) owns input parsing, output formatting, and user interaction — including flags, streaming display, shell completion, and TTY detection.

The agent core (src/agent/) owns classification, routing, tool execution, and context assembly. No business logic lives in the CLI layer. This boundary keeps the CLI lightweight and replaceable while the core remains the single source of truth for agent behaviour.

Command Structure

Running insrc with no arguments enters the interactive REPL environment, providing a persistent session with full context retention, history, and real-time feedback. All other subcommands are non-interactive by default: they classify and execute their task, print output to stdout, and exit with an appropriate status code, making them suitable for scripting and CI pipelines.

insrc                                    # Interactive REPL (default)
insrc ask <message>                      # One-shot: classify, execute, exit
insrc plan <description>                 # One-shot: generate plan
insrc daemon <start|stop|status|logs>    # Daemon lifecycle
insrc repo <add|remove|list|status>      # Repository management
insrc session <list|show|resume|delete>  # Session management
insrc config <show|set|edit>             # Configuration management
insrc index <path>                       # Manual indexing
insrc graph <query>                      # Direct graph query (no LLM)
insrc health                             # System health check
insrc version                            # Version info
insrc completion <bash|zsh|fish>         # Shell completion

Command Reference

Example Invocations

# Enter the interactive REPL
$ insrc

# Ask a one-shot question and exit
$ insrc ask "Which functions call parseConfig?"

# Generate a plan
$ insrc plan "Refactor the authentication module to support OAuth2"

# Start the background daemon
$ insrc daemon start

# Check daemon status
$ insrc daemon status
# Add a repository to the index
$ insrc repo add /home/user/projects/myapp

# List all registered repositories
$ insrc repo list

# Manually re-index a path
$ insrc index /home/user/projects/myapp/src

# Query the graph directly without invoking the LLM
$ insrc graph "MATCH (f:Function)-[:CALLS]->(g:Function) RETURN f, g LIMIT 10"
# View current configuration
$ insrc config show

# Set a configuration value
$ insrc config set model.provider openai

# Open configuration in $EDITOR
$ insrc config edit

# Install shell completion (zsh example)
$ insrc completion zsh > ~/.zsh/completions/_insrc
Scripting tip: All one-shot commands exit with code 0 on success and a non-zero code on failure, and they write results to stdout with diagnostic messages on stderr. This makes them safe to use in shell pipelines and CI scripts without additional wrappers.

REPL prompt

The prompt character is ❯ . Its color reflects the current REPL state:

State Color
Ready white
Processing dim
Auto-accept yellow
Error red 
❯ explain the auth middleware          # Ready — white prompt
❯ (thinking…)                          # Processing — dim prompt
❯ refactor all handlers                # Auto-accept mode — yellow prompt
❯ (tool call failed)                   # Error — red prompt

Input features

  • Readline history persisted at ~/.insrc/repl_history; navigate with ↑/↓
  • Multi-line input via trailing \, opening triple backtick ```, or Ctrl+J
  • @file:path references inline — glob patterns supported (e.g., @file:src/**/*.ts)
  • @claude / @opus / @local provider prefixes to route a message to a specific model
  • /intent prefix to express high-level goals without triggering immediate tool calls
  • Automatic pipe detection — when stdin is not a TTY, insrc reads from the pipe and exits after one response
# Multi-line with backslash continuation
❯ summarise the following diff: \
  <paste diff here>

# File reference with glob
❯ @file:src/api/**/*.go what endpoints are unauthenticated?

# Provider prefix
❯ @local explain this function without sending data to the cloud

# Pipe detection
$ git diff HEAD~1 | insrc "what changed?"

Streaming Output

The CLI streams output token-by-token using a lightweight Markdown renderer with a rolling line buffer. This approach never triggers a full-screen redraw, ensuring responsive and efficient output even during long-running operations.

❯ debug why payments are failing
┌ intent: debug (0.94) → local (qwen3-coder)
│ ▸ Read src/payment/handler.ts (3 lines)
│ ▸ Bash kubectl logs payment-svc --tail=50 (OOMKilled detected)
│ Agent explanation text...
│ ── fix ──
│ coloured diff
│ ✓ Fix applied (1 file written)
└ 1.2s · 847 tokens · $0.00

Tool call rendering

Tool type Display format
Read-only Inline summary
Mutating-validated Preview + confirmation prompt
Mutating-auto-accepted Applied silently
Mutating-rejected Rejected with reason

Escalation notices

When the router escalates a request to a more capable model, a notice is emitted inline before the next response begins:

⇡ Escalating to claude-sonnet-4-6 (review) — ~1,200 tokens

Colour scheme

Element Colour ANSI
Agent text White 37
Intent badge Cyan 36
Tool names Dim 2;37
Diff additions Green 32
Diff deletions Red 31
Warnings Yellow 33
Errors Red bold 1;31
Escalation Magenta 35

The CLI automatically detects TTY environments and enables colour output accordingly. The --color flag forces colour on and --no-color forces it off, overriding auto-detection. When output is piped to another process or file, all ANSI escape codes are stripped to ensure clean, portable text.

Verbose output may expose sensitive data. Passing --verbose (or -v) causes the CLI to print full tool inputs and outputs — including file contents, environment variables, and command results — inline in the stream. Avoid using this flag in shared terminal sessions or when logging to persistent storage.

One-Shot Commands

insrc ask

Executes a single turn and exits. The prompt is auto-classified into an intent, or you can override classification with --intent. The process returns a structured result and terminates — no persistent session is created.

insrc ask "Add a README.md file"
insrc ask --intent plan "Create a new config file" --claude
insrc ask --intent fix "Fix the bug in main.py" --local
insrc ask --intent update "Update dependencies" --json
insrc ask --intent review "Review code changes" --file src/main.py --file src/utils.py
Flag Short Description
--intent -i Override auto-classification with an explicit intent string
--claude -c Route request to Claude (Sonnet) instead of the default provider
--opus Route request to Claude Opus (highest capability tier)
--local -l Route request to the configured local LLM endpoint
--json -j Emit structured JSON to stdout instead of human-readable output
--cwd Override the working directory for file operations
--auto-accept Automatically accept all proposed file changes without prompting
--verbose -v Print detailed reasoning, classification scores, and provider selection
--dry-run -n Simulate execution and show diff without writing any files
--file -f Attach a file to the context (repeatable: -f a.py -f b.py)
--timeout Abort and exit after N seconds (default: 120)

Exit codes

Code Meaning Description
0 Success Request completed and all file writes validated
1 Error General runtime error (see error field in JSON output)
2 No API key Required API key for the selected provider is missing or invalid
3 Daemon failed Background daemon could not be reached or crashed during execution
4 No LLM No provider is available (no API key and no local endpoint configured)

JSON output format

When --json is passed, a single JSON object is written to stdout on exit. All fields are always present; unused fields are null.

{
  "intent":       "create",
  "confidence":   0.95,
  "provider":     "claude",
  "model":        "claude-3-opus-20240229",
  "message":      "Created config.yaml with default settings.",
  "diff":         "--- /dev/null\n+++ config.yaml\n@@ -0,0 +1,5 @@\n+config:\n+  debug: false\n+  log_level: info\n+  port: 8080\n+  host: localhost\n",
  "filesWritten": ["config.yaml"],
  "validated":    true,
  "escalated":    false,
  "tokens":       124,
  "cost":         0.0018,
  "duration":     2.34,
  "error":        null
}
Field notes: confidence reflects the classifier score before any --intent override. escalated is true when the request was automatically routed to a more capable model. cost is in USD and is null for local providers.

insrc plan

Generates a numbered, step-by-step execution plan for the given prompt and prints it to stdout. This is a convenience shorthand for insrc ask --intent plan. With --execute, the CLI walks through each step interactively, prompting for confirmation before applying changes.

insrc plan "Implement user authentication"
insrc plan --execute "Create database schema"
insrc plan --execute --claude "Refactor legacy code to use dependency injection"
Flag Short Description
--execute After generating the plan, execute each step interactively
--claude Use Claude for both plan generation and step execution
--json -j Emit the plan as a JSON array of step objects instead of plain text
--cwd Override the working directory for all steps in the plan

Plan execution flow

When --execute is set, the CLI prompts between every step. Respond with Y to execute, n to skip, s to show the step diff without executing, or q to abort remaining steps.

Graph CLI

Run graph queries without an LLM — fast, deterministic, and free. Queries are executed directly against the code graph database, returning structured results instantly with no API calls or token costs. 

# Natural language queries
insrc graph "find all functions that call parseManifest"
insrc graph "who calls parseManifest"
insrc graph "what does parseManifest call"

# Raw Cypher query
insrc graph --cypher "MATCH (n:Function)-[:CALLS]->(m:Function {name:'parseManifest'}) RETURN n.name, n.file"

# Graph entity lookup by name
insrc graph entity parseManifest

# With additional flags
insrc graph "find all functions that call parseManifest" --depth 3
insrc graph "find all functions that call parseManifest" --json
insrc graph "find all functions that call parseManifest" --cwd /path/to/project
insrc graph --cypher "MATCH (n:Function) RETURN n.name, n.file" --json

Flags

Flag Description Default
--cypher Execute a raw Cypher query string directly against the graph database instead of translating natural language. —
--depth <n> Traversal depth for caller/callee queries. Minimum 1, maximum 5. 1
--json Output results as JSON instead of the default human-readable format. —
--cwd <path> Override the working directory used to locate the project graph. Defaults to the current directory. process.cwd()

Output formats

Caller/callee queries render as an indented tree showing the call hierarchy:

$ insrc graph "who calls parseManifest" --depth 2

┌──────────────────────┬──────────────────────────────────────┐
│ Name                 │ File                                 │

Interpretive questions are automatically re-routed. If your query asks why something works the way it does, or requests an explanation of intent or design (e.g. "why does parseManifest ignore optional fields?"), insrc graph will detect that a structural graph query alone cannot answer it and will re-route the request to insrc research automatically. You will see the following notice before the research output begins: 
⚠ Query requires interpretation — re-routing to `insrc research`...

Daemon Management

The insrc daemon command provides control over the background daemon process that manages model execution and caching.

insrc daemon start|stop|status|logs

Auto-start behaviour

When a REPL session or one-shot command is initiated, the daemon is automatically started as a detached process if it is not already running. The CLI waits up to 5 seconds for the socket to become available. If the daemon fails to start or become responsive within that window, the command exits with code 3.

The daemon is shared across all CLI sessions and VS Code instances. It persists after the CLI exits and is only terminated by explicitly running insrc daemon stop.

Repository Management

The repository management system allows users to add, remove, list, and check the status of repositories in the index. These operations are performed using the insrc repo command with the following subcommands:

insrc repo add <repo-path>
insrc repo remove <repo-path>
insrc repo list
insrc repo status <repo-path>

Note: The <repo-path> argument accepts either a local filesystem path or a remote repository URL.

Session Management

Sessions persist across interactions, each assigned a unique identifier stored within the daemon. The daemon maintains a complete record of turns and summaries for every session, enabling full resumption of previous conversations at any point. When a session is resumed, the agent restores complete context from all prior exchanges — including decisions, code review feedback, and intermediate reasoning — allowing work to continue exactly where it left off.

insrc session list
insrc session show <session-id>
insrc session resume <session-id>
insrc session delete <session-id>

Cross-session seeding vs. full resume: Starting a new session against the same repository seeds the L2 memory layer from prior session summaries, giving the agent lightweight background awareness of past work. Configuration

The insrc config command allows you to manage your configuration settings. It supports three subcommands:

insrc config show                          # Display current configuration
insrc config set <key> <value>            # Set a configuration value
insrc config edit                          # Open config file in default editor

Example usage:

# Set the default model

# Set a custom Ollama host
insrc config set ollama.host http://localhost:11434

Configuration precedence

When the same setting is defined in multiple places, insrc resolves values according to the following priority order (1 = highest priority):

Priority Source Scope
1 CLI flags Command-line only
2 Environment variables System-wide
3 config.json Per-user
4 Defaults Built-in

Environment variables

Environment variables take precedence over config.json and can be used to inject secrets in CI/CD pipelines or containerised environments without modifying the configuration file.

Permission Model

The permission model controls how tool calls are executed and validated. By default, all mutating tool calls are validated by Claude or Haiku before execution.

Validate mode (default)

All mutating tool calls are validated by Claude/Haiku before execution. This ensures safety and prevents unintended changes.

▸ Edit src/handler.ts → ⇡ Validating with claude-haiku-4-5... ✓ approved

Auto-accept mode

Mutating calls execute immediately without validation. This mode can be set at three scopes:

  • Per-session using the /mode command
  • Per-invocation using the --auto-accept flag
  • Globally using config set
Warning: Auto-accept mode bypasses all validation checks. Use with caution in production environments or when working with irreversible operations.

Stdin pipe

When stdin is not a TTY, insrc automatically enters one-shot mode, reading piped input and exiting after producing a response. This enables composing insrc with other Unix tools in pipelines.

# Review a file's contents
cat file.ts | insrc ask "review"

# Review staged diff with explicit intent
git diff | insrc ask "review" --intent review

# Generate a commit message and parse the result with jq
git diff --cached | insrc ask "commit message" --json | jq '.message'

# Debug a failing test suite
npm test 2>&1 | insrc ask "debug this"

Scripting patterns

Because all log output is written to stderr and the agent response to stdout, insrc composes cleanly in scripts and pipelines. Common patterns include CI quality gates, batch documentation, and file-watch loops.

# CI quality check — exit non-zero if quality gate fails
result=$(npm test 2>&1 | insrc ask "did all tests pass? answer yes or no" --json)
passed=$(echo "$result" | jq -r '.answer')
if [ "$passed" != "yes" ]; then
  echo "Quality gate failed" >&2
  exit 1
fi

# Batch document every Markdown file in the current directory
for file in *.md; do
  echo "Documenting $file..."
  insrc ask "improve the documentation in this file" --file "$file" > "${file%.md}.out.md"
done

# Watch mode — re-review any file that changes under src/
fswatch -r src/ | while read -r file; do
  echo "File changed: $file" >&2
  insrc ask "review changes" --file "$file"
done

Stderr / stdout separation: All progress logs, prompts, and diagnostic messages are written to stderr; the agent response is written to stdout. This means downstream tools such as jq, grep, or file redirections receive only the response, and log noise never corrupts piped output.

Errors are displayed with a red prefix, a concise message, and an actionable recovery hint to guide users toward immediate resolution.

  → Set your key with `export ANTHROPIC_API_KEY=` or add it via `ai config set api-key`.

Log levels

All log output is written to stderr so that piping and redirection of stdout work correctly — for example, insrc query "…" | jq receives only the result on stdout while diagnostic messages remain visible in the terminal.

Represents parsed command-line arguments passed into the CLI entry point.

interface CLIOptions {

This section compares the key aspects of using the tool as a command-line interface versus as a VS Code extension, highlighting the differences in user experience, functionality, and integration.

Input readline + stdin pipe Webview input box + @file picker
Output Streaming ANSI text Webview HTML + native diff editor
Diff review Coloured text diff Native VS Code diff editor
Plan tracking Numbered list + step prompts Kanban board
Graph results ASCII tree / table Interactive tree view
Annotations Not supported (use @file: refs) CodeLens + inline comments
Settings insrc config + $EDITOR Webview settings form
Session management insrc session commands Navigation panel tree
Daemon lifecycle insrc daemon commands Auto-managed by extension
Multi-repo --cwd flag Repo selector in chat header
Cost visibility /cost command, turn footer Status bar + session panel
Scriptable Yes (exit codes, --json, pipes) No

CLI Examples

Piping input and capturing structured output:

# Pipe a prompt from stdin
echo "Summarise the auth module" | insrc chat --json

# Use --cwd to target a specific repository
insrc --cwd ~/projects/api chat "List all public endpoints"

# Script with exit codes
insrc apply patch.diff && echo "Applied" || echo "Failed"

Checking cost and managing sessions from the terminal:

# View token cost for the current session
insrc> /cost
Session cost: $0.0042  (prompt: 1 840 tok, completion: 312 tok)

# List and resume sessions
insrc session list
  abc123  2024-06-10 14:02  "Refactor auth module"
  def456  2024-06-09 09:17  "Add pagination to /users"

insrc session resume abc123

Daemon lifecycle management:

insrc daemon start
Daemon started (pid 48291, socket /tmp/insrc-daemon.sock)

insrc daemon status
● insrc-daemon  running  pid 48291  uptime 00:04:32

insrc daemon stop
Daemon stopped.

Streaming ANSI diff output in the terminal:

--- a/src/auth/token.ts
+++ b/src/auth/token.ts
@@ -12,7 +12,7 @@ export function generateToken(user: User): string {
-  const expiry = 3600;

 }

Scripting tip: Use --json together with jq to extract structured fields from responses, making it straightforward to integrate insrc into CI pipelines or shell scripts.

The CLI offers direct control and full scripting capabilities via exit codes, --json output, and standard Unix pipes. The VS Code extension provides a more integrated experience with native editor features such as the diff editor, CodeLens annotations, and an auto-managed daemon — but is not scriptable.
