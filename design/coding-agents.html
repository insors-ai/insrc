Coding Agents

Pair programming and delegated execution for code tasks

1. Overview

Two coding agents handle all code-producing tasks: Pair (collaborative, multi-turn) and Delegate (autonomous, plan-driven). They replace the current single-turn pipelines (implement.ts, refactor.ts, debug.ts, test.ts) with checkpointable, resumable agents built on the existing agent framework.

Both agents follow a local-first hybrid pattern: local model generates code, Claude reviews/validates, escalation to Claude generation only on repeated validation failure. Users can override model assignments per step via config or per-turn via @mentions.

2. Architecture

2.1 Two Agents, Not One

Pair and Delegate are separate agents with fundamentally different interaction models:

Pair Agent Delegate Agent
Interaction Collaborative, multi-turn Autonomous, plan-driven
User role Co-driver — reviews every proposal Observer — intervenes on failures/ambiguity
Planning None — reactive, fluid Planner agent → approved plan
State model Working set (files, diffs, focus) Plan + step results
Gate frequency Every proposal On failure/ambiguity only (configurable)
Backtracking User redirects freely User edits remaining plan

2.2 Bug Fix Routing

Decision: Bug fixes route to the Pair Agent. Diagnosis is inherently interactive — the user provides symptoms, the agent investigates, proposes hypotheses, and the user confirms or redirects. Batch fixes ("fix all failing tests in indexer") route to Delegate.

The debug intent maps to Pair. The classifier distinguishes batch scope ("fix all X", "update every Y") → Delegate from single-issue scope → Pair.

2.3 Shared Infrastructure

Both agents share:

  • Investigate helper — reusable tool-calling function for reading files, grepping, graph queries, git operations. Any step can call it.
  • Diff management — generate, validate, apply, rollback unified diffs
  • Test execution — run tests, parse results, detect regressions
  • LLM routing — local generate → Claude validate → escalate pattern
  • Context assembly — entity lookup, semantic search, graph traversal via daemon RPC
  • Config-driven model overrides — per-agent, per-step model bindings

3. Pair Agent

3.1 Step Flow

+-- design/spec found --+ | | entry -----> [check-context] [analyze] | | v v [propose] <-----------------+ | | (simple: single diff) | (complex: TODO list + first diff) v [review-gate] --approve--> [apply] --> [validate] | | | edit/redirect pass | fail (retry < N) | | | | +-----+ more TODOs? | yes | no | | | done | [next-todo] +---> [review-gate] v | (loop continues) [summarize] [propose] (next item)
Step Type Description
check-context logic Look for design doc/spec: explicit attachment, [design] session tag, or artifact store. If found → propose. If not → analyze.
analyze LLM + tools Investigate: read files, query graph, grep codebase. Build understanding of scope. Uses the investigate helper. Produces scoping summary.
propose LLM Generate a proposal: code diff, diagnostic finding, design suggestion, or clarifying question. Local model generates; type depends on current mode. For multi-step changes, generates a TODO list so the user can approve scope and track progress.
review-gate gate User reviews the proposal. Actions: approve, edit (with feedback), reject (try different approach), expand ("also do X"), done (wrap up session).
apply logic Apply the approved diff to disk. Track in changesApplied.
validate Claude Claude validates the applied change: type safety, edge cases, integration correctness. Pass → back to review-gate (loop continues). Fail → generate refined proposal.
summarize LLM Produce session summary: files changed, decisions made, open items. Stored as session L2 tag.

3.2 State Shape

interface PairState extends AgentState {
  input: {
    message: string;
    codeContext: string;
    designSpec?: string;         // from session tag, attachment, or artifact
    session: SessionRef;
  };
  mode: 'implement' | 'refactor' | 'debug' | 'explore';
  hasDesignContext: boolean;      // skips analyze if true

  // Working set (grows organically)
  filesInScope: string[];
  changesApplied: DiffEntry[];   // accumulated approved diffs
  pendingProposal: Proposal | null;

  // TODO tracking (per-proposal, not persistent across proposals)
  activeTodos: TodoItem[] | null;  // generated when proposal has multiple sub-tasks
  currentTodoIndex: number;

  // Debug mode
  findings: Finding[];           // diagnostic evidence chain
  hypotheses: Hypothesis[];      // tracked hypotheses (confirmed/rejected/pending)

  // Context management
  conversationSummary: string;   // compressed history of prior iterations
  currentFocus: string;          // what the agent is working on now
  iterationCount: number;

  // Provider overrides
  providerOverride?: ProviderOverride;
  editRounds: Record<string, number>;
}

interface TodoItem {
  index: number;
  description: string;
  status: 'pending' | 'in-progress' | 'done' | 'skipped';
  diff?: string;               // generated diff for this item
}

interface Proposal {
  summary: string;             // what the agent proposes to do
  diff?: string;               // for simple single-change proposals
  todos?: TodoItem[];          // for multi-step proposals
  findings?: Finding[];        // for debug mode proposals
  question?: string;           // for clarifying questions
}

3.3 Gate Actions

ActionEffect
approve Apply the proposed diff. Agent loops back to propose next change or waits for direction.
edit <feedback> Agent refines the current proposal using feedback. Does not apply anything.
reject <reason> Discard proposal entirely. Agent tries a different approach based on the reason.
expand <request> Shift focus to the new request. Prior context carries forward. No re-planning — pair is fluid.
done End the session. Agent produces summary and exits.
Fluid direction, structured execution: The pair agent has no upfront plan — "also do X" simply becomes the next iteration, and each gate response reshapes focus. However, when a proposal involves multiple sub-tasks (e.g., "refactor this interface" → update interface, update 5 callers, update tests), the agent generates a TODO list as part of the proposal. The user approves the TODO at the gate, and the agent works through items, reporting progress as each completes. TODOs are ephemeral — scoped to the current proposal, not carried across gate cycles.

3.4 Operating Modes

Mode is inferred from intent classification but can shift during the session:

ModeProposal TypeEntry Intent
implement Code diffs (new code) implement
refactor Code diffs (restructuring) refactor
debug Findings, hypotheses, then fix diffs debug
explore Analysis, questions, suggestions research (code-focused)

Mode can shift naturally: a debug session that identifies the bug transitions to implement mode for the fix. An implement session that discovers a design issue might shift to explore.

3.5 Context Entry

Decision: If a design doc or spec is available (explicit attachment, session [design] tag from a prior designer run, or matching artifact), skip analyze and go straight to propose. Otherwise, the agent investigates first — reads files, queries the graph, builds a scoping summary.

Context sources checked at entry (in priority order):

  1. Explicit attachment (@design spec.md)
  2. Session tag [design] or [requirements] from prior designer/brainstorm run
  3. Artifact store — most recent design artifact for the current repo

4. Delegate Agent

4.1 Step Flow

entry --> [invoke-planner] --> [approve-plan-gate] --> [execute-step] | | edit plan pass | fail | | | +------+ v v | [advance] [failure-gate] | | | | v edit plan / retry / abort | more steps? | | | | | | yes no | | | | | | v v | | [execute] [report] | +---------------------+
Step Type Description
invoke-planner sub-agent Run the planner agent as a sub-agent. Receives the approved plan with steps, dependencies, and suggested commit points.
approve-plan-gate gate User reviews the plan. Can approve, edit steps, adjust commit strategy, set gate level. Plan is a living document — editable at any subsequent gate too.
execute-step LLM + tools Execute one plan step: investigate → generate diff (local) → validate (Claude) → apply → optionally run tests. Retry loop on validation failure.
advance logic Record step result. Commit if this is a commit point. Move to next step or report.
failure-gate gate Fires on: validation failure after retries, test failure, scope creep, ambiguity. User can: retry, edit remaining plan, skip step, abort.
report LLM Produce execution report: steps completed, files changed, test results, open items. Final commit if pending. Write artifact.

4.2 State Shape

interface DelegateState extends AgentState {
  input: {
    message: string;
    codeContext: string;
    designSpec?: string;
    session: SessionRef;
  };

  // Plan (from planner agent, editable at gates)
  plan: DelegatePlan;
  currentStepIndex: number;
  stepResults: StepResult[];

  // Configuration
  gateLevel: 'minimal' | 'normal' | 'cautious';
  commitStrategy: CommitStrategy;
  testAfterEach: boolean;
  rollbackOnFailure: boolean;

  // Execution tracking
  filesChanged: string[];
  testsRun: TestRunResult[];
  pendingCommitFiles: string[];

  // Provider overrides
  providerOverride?: ProviderOverride;
}

interface DelegatePlan {
  steps: DelegatePlanStep[];
  commitPoints: number[];        // step indices after which to commit
  estimatedCost?: string;        // pre-assessed LLM cost estimate
}

interface DelegatePlanStep {
  index: number;
  title: string;
  description: string;
  status: 'pending' | 'running' | 'done' | 'failed' | 'skipped';
  commitAfter: boolean;          // suggested by planner, overrideable
  result?: StepResult;
}

type CommitStrategy =
  | { kind: 'per-step' }
  | { kind: 'at-end' }
  | { kind: 'at-points'; points: number[] }; // from plan, user-adjusted

4.3 Planning Phase

Decision: Delegate always invokes the planner agent for task decomposition. The planner handles analysis, step ordering, dependency detection, and user validation. Delegate does not reinvent planning — it is an execution engine.

The planner agent runs as a sub-agent within the delegate's invoke-planner step. The planner's output (approved plan) becomes the delegate's execution input. The planner suggests commit points as part of its output; the user can override during plan approval.

4.4 Execution Phase

Each plan step executes as a mini-pipeline:

  1. Investigate — assemble context for this step (files, entities, graph)
  2. Generate — local model produces unified diff
  3. Validate — Claude reviews correctness
  4. Apply — write diff to disk, re-index changed files
  5. Test (if testAfterEach) — run relevant tests
  6. Fix loop (if tests fail) — local retries, then Claude escalation
  7. Commit (if commit point) — stage and commit changed files

Steps 2–6 reuse the existing implement/refactor pipeline logic as a shared helper, not by calling the pipeline directly.

4.5 Gating Strategy

The user is an observer who intervenes when needed. Gate level is configurable:

LevelGates on
minimal Validation failure after all retries. Test failure after fix loop exhausted.
normal (default) Above + scope creep (step touches unexpected files). Ambiguous requirements.
cautious Above + every applied diff shown (but auto-proceeds after 10s unless user intervenes).

4.6 Commit Strategy

Decision: The planner suggests commit frequency as part of the plan. The user can override during plan approval: "commit after every step", "commit at end", or accept the planner's suggestion. The delegate respects this throughout execution.

Commit messages are auto-generated from the step description and files changed. The delegate does not push — commits are local only. The user pushes when ready.

4.7 Plan Editing at Gates

When the delegate gates (failure, ambiguity, scope creep), the user sees the current plan state and can modify it:

  • Skip step — mark as skipped, advance to next
  • Edit step — reword description, change scope
  • Reorder — move steps
  • Add step — insert new step at any position
  • Abort — stop execution, keep changes applied so far
  • Retry — re-attempt the failed step (possibly with feedback)

The plan is a living document throughout execution. The delegate re-reads the plan after every gate interaction.

5. LLM Routing

5.1 Universal Pattern

Decision: Local generates, Claude validates. Always. Local models like qwen-coder are capable enough for code generation — even complex multi-file changes. Claude's value is as a reviewer. Claude generates only when local fails validation repeatedly.
PhaseDefault ProviderEscalation
Investigation (tool calls) Local —
Code generation Local Claude after N validation failures
Validation / review Claude Haiku Claude Sonnet if Haiku uncertain
Fix loops Local Claude after local retries exhausted

5.2 Escalation Triggers

  • Validation failure — local diff fails Claude review N times → Claude generates next attempt
  • User rejection (pair agent) — if user rejects and local re-proposal is also rejected, escalate
  • Fix loop exhaustion — local fix attempts exhausted → Claude gets full error context

5.3 Config Overrides

Users can override model assignments per agent and per step in ~/.insrc/config.json, following the same pattern as designer/brainstorm agents:

{
  "models": {
    "agents": {
      "pair": {
        "propose": "local",
        "validate": "claude-haiku-4-5",
        "escalate": "claude-sonnet-4-5"
      },
      "delegate": {
        "execute": "local",
        "validate": "claude-haiku-4-5",
        "fix-escalate": "claude-sonnet-4-5"
      }
    }
  }
}

5.4 Per-Turn Mentions

At any gate, the user can prefix their response with a provider mention to override the model for the next LLM call:

  • @local — force local model
  • @haiku / @sonnet / @opus — force specific Claude tier
  • @sticky — make the override persist until cleared
  • @clear — revert to default routing

Same mechanism as the brainstorm agent's parseProviderMention().

6. Investigate Helper

A reusable function that both agents call within any step that needs to understand code before acting. Not a step itself — a building block.

async function investigate(
  query: string,
  ctx: StepContext,
  opts?: {
    maxToolCalls?: number;     // default 10
    tools?: ToolFilter;        // subset of tools to expose
    provider?: LLMProvider;    // default: local
  },
): Promise<InvestigationResult>

Internally runs a mini tool loop: the LLM drives tool calls (Read, Grep, Glob, graph_search, graph_callers, git log/diff) to gather information. Returns structured findings:

interface InvestigationResult {
  summary: string;              // LLM-generated summary of findings
  filesExamined: string[];      // files read during investigation
  entitiesFound: Entity[];      // graph entities discovered
  relevantCode: CodeSnippet[];  // key code sections identified
  toolCallCount: number;
}

Used by:

  • Pair agent analyze step — initial scoping
  • Pair agent propose step (debug mode) — gathering evidence for hypotheses
  • Delegate agent execute-step — assembling context per plan step

7. Intent Routing

How intents map to coding agents:

IntentAgentMode / Notes
implement Pair mode: implement
refactor Pair mode: refactor
debug Pair mode: debug (hypothesis-driven investigation)
test Pair mode: implement (test generation is code generation)
implement (batch) Delegate Classifier detects batch scope: "implement all endpoints", "add X to every Y"
refactor (batch) Delegate Classifier detects batch scope: "rename X across all modules"
debug (batch) Delegate "Fix all failing tests", "resolve all lint errors"
Scope detection: The classifier needs to distinguish single-item from batch-scope requests. Signals: "all", "every", "across", explicit file lists, reference to a plan with multiple steps. Single-item is the default — Delegate requires clear batch indicators.
