Config Management Framework

Hierarchical config with vector-indexed templates, feedback, and conventions

1. Overview

insrc currently has a single flat config at ~/.insrc/config.json loaded by loadConfig() in src/agent/config.ts. This works for LLM provider settings and model routing, but doesn't support:

  • Project-level overrides — different repos need different test conventions, provider preferences, or agent behavior
  • Accumulated knowledge — test feedback, coding conventions, learned patterns that should influence agent behavior
  • Searchable content — templates, feedback, and conventions need semantic retrieval, not just flat file reads
  • Cross-project transfer — language-level patterns (vitest conventions, Go test idioms) apply across repos

This design introduces a two-tier config hierarchy (global + project) with a dedicated vector store for indexing config entries, templates, and feedback. The existing code entity store (~/.insrc/lance/) remains untouched — config content lives in a separate LanceDB table.

2. Config Hierarchy

Lookup order (most specific wins): 1. Project config <project>/.insrc/ 2. Global config ~/.insrc/ 3. Hardcoded defaults Merge: deep merge with project overriding global. Vector store: entries from both levels indexed together, tagged with scope (global vs project).

2.1 Global Config (~/.insrc/)

User-wide settings, cross-project knowledge, and language-level conventions. This is the existing ~/.insrc/ directory, extended with new subdirectories.

Owns:

  • LLM provider config (Ollama host, API keys, model tiers)
  • Default agent behavior (permission mode, routing mode)
  • Language-level templates and feedback (vitest patterns, pytest conventions, Go idioms)
  • The config vector store (shared across all projects)

2.2 Project Config (<project>/.insrc/)

Per-repo overrides and project-specific knowledge. Created on demand (not required). Can be checked into the repo or gitignored — up to the team.

Owns:

  • Project-specific model overrides (e.g., always use Claude for this repo's tests)
  • Project-specific conventions (entity ID format, custom logger API, internal service URLs)
  • Project-specific templates (test setup patterns for this repo's DB layer)
  • Project-specific feedback (lessons learned from testing this codebase)

2.3 Merge Strategy

Config resolution uses deep merge with project overriding global:

function resolveConfig(repoPath: string): ResolvedConfig {
  const global = loadGlobalConfig();          // ~/.insrc/config.json
  const project = loadProjectConfig(repoPath); // <repo>/.insrc/config.json

  return deepMerge(global, project);
  // Arrays: project replaces global (no concat)
  // Objects: recursive merge, project keys win
  // Primitives: project wins
}

For non-JSON content (templates, feedback, conventions): both levels are indexed into the vector store with a scope tag. At query time, project-scoped entries rank higher than global entries via a boost factor.

Content typeMerge behavior
JSON configDeep merge, project wins
TemplatesProject shadows global by name; both searchable
FeedbackBoth levels returned; project entries ranked higher
ConventionsBoth levels returned; project entries ranked higher

Separation from code store: Config content lives in ~/.insrc/config-store/ (a separate LanceDB database), not in the existing ~/.insrc/lance/ entity store. This avoids polluting code search results with config/template content and allows independent schema evolution. The code store indexes source entities; the config store indexes knowledge entries. Both stores are managed by the same IndexerService in the daemon — the indexer holds two store handles and routes jobs by kind.

4. Config Vector Store

A dedicated LanceDB database at ~/.insrc/config-store/ that indexes all templates, feedback, and conventions from both global and project scopes. Enables semantic search over accumulated knowledge.

4.1 Schema

Single table: config_entries

// Apache Arrow schema (LanceDB)
{
  id:           Utf8,          // SHA256(scope + namespace + category + path), 32 hex chars
  scope:        Utf8,          // 'global' | 'project:<repoPath>'
  namespace:    Utf8,          // 'tester' | 'pair' | 'delegate' | 'designer' | 'planner' | 'common'
  category:     Utf8,          // 'template' | 'feedback' | 'convention'
  language:     Utf8,          // 'typescript' | 'python' | 'go' | 'all'
  name:         Utf8,          // Human-readable name (e.g., 'vitest-unit', 'db-integration')
  filePath:     Utf8,          // Source file path (for change detection)
  body:         Utf8,          // Full markdown content (max 16K chars)
  tags:         Utf8,          // Comma-separated tags for filtering
  updatedAt:    Utf8,          // ISO datetime
  contentHash:  Utf8,          // SHA256 of body (change detection)
  vector:       FixedSizeList[Float32 x DIM]   // Embedding vector
}
ID generation: SHA256(scope + '\x00' + namespace + '\x00' + category + '\x00' + filePath) truncated to 32 hex chars. Deterministic: re-indexing the same file produces the same ID, enabling upsert semantics.

Namespaces

Namespaces partition config entries by the agent that produces and consumes them. This enables agents to search only their own config space (fast, focused) or widen to include shared conventions (common).

NamespaceOwnerContains
tester Tester agent Test templates (vitest-unit, pytest-live, ...), test feedback (fix loop patterns, classification corrections), test conventions (file naming, framework idioms)
pair Pair agent Coding templates (error handling, logging patterns), coding feedback (validation loop outcomes), implementation conventions
delegate Delegate agent Batch execution templates, planning feedback, commit conventions
designer Designer agent Design document templates, review checklists, architecture patterns
planner Planner agent Plan structure templates, step granularity feedback, dependency patterns
common All agents Language conventions, project-wide patterns, import style, logging rules — anything not specific to one agent

Search behavior: An agent queries its own namespace plus common by default. The namespace field on ConfigSearchOpts accepts a single namespace or an array for multi-namespace search:

// Tester agent's analyze step — searches tester + common
const results = await searchConfig({
  query: 'vitest mock patterns',
  namespace: ['tester', 'common'],
  language: 'typescript',
  boostProject: repoPath,
});

// Explicit single-namespace lookup
const results = await searchConfig({
  query: 'commit message format',
  namespace: 'delegate',
  category: 'convention',
});
Namespace from directory: The namespace is derived from the directory structure (templates/tester/, feedback/pair/) or from the frontmatter namespace field. Directory takes precedence. Files directly under templates/ or feedback/ (no agent subdirectory) default to common.

4.3 Indexing Pipeline

Config entries are indexed by the same background daemon indexer (IndexerService) that indexes code entities. The indexer holds a separate LanceDB store handle for config content — same infrastructure, different table, no cross-contamination with code search.

Integration with IndexerService

The existing indexer processes jobs via processJob() dispatching on job.kind ('full', 'file', 'reembed'). Config indexing adds two new job kinds:

// Extended IndexJob union
type IndexJob =
  | { kind: 'full';        repoPath: string }
  | { kind: 'file';        filePath: string; event: 'create'|'update'|'delete' }
  | { kind: 'reembed';     repoPath: string }
  | { kind: 'config-full';    scope: ConfigScope }           // NEW: full config index
  | { kind: 'config-file';    filePath: string; scope: ConfigScope;
      event: 'create'|'update'|'delete' }                   // NEW: single config file event
  | { kind: 'config-reindex'; scope: ConfigScope }          // NEW: drop + re-index all config

The IndexerService constructor receives both store handles:

class IndexerService {
  private readonly db:          DbClient;      // Kuzu + code LanceDB (existing)
  private readonly configStore: ConfigStore;    // Config LanceDB (new)
  private readonly queue:       IndexQueue;
  private readonly watcher:     Watcher;
  // ...

  async processJob(job: IndexJob): Promise<void> {
    switch (job.kind) {
      case 'full':        await this.fullIndex(job.repoPath);                  break;
      case 'file':        await this.fileEvent(job.filePath, job.event);       break;
      case 'reembed':     await this.reembed(job.repoPath);                   break;
      case 'config-full':    await this.configFullIndex(job.scope);              break;
      case 'config-file':    await this.configFileEvent(job.filePath, job.scope, job.event); break;
      case 'config-reindex': await this.configReindex(job.scope);               break;
    }
  }
}

Watched directories

On daemon startup, the watcher registers config directories alongside repo directories:

async start(repos: RegisteredRepo[]): Promise<void> {
  // Existing: watch repos
  for (const repo of repos) {
    await this.watcher.addRepo(repo.path);
  }

  // NEW: watch global config directories
  await this.watcher.addConfigDir(join(PATHS.insrc, 'templates'));
  await this.watcher.addConfigDir(join(PATHS.insrc, 'feedback'));
  await this.watcher.addConfigDir(join(PATHS.insrc, 'conventions'));

  // NEW: watch project config directories for each registered repo
  for (const repo of repos) {
    const projectConfig = join(repo.path, '.insrc');
    if (await dirExists(projectConfig)) {
      await this.watcher.addConfigDir(projectConfig);
    }
  }

  // Enqueue initial config index
  this.queue.enqueue({ kind: 'config-full', scope: { kind: 'global' } });
}

When a repo is added via addRepo(), its <repo>/.insrc/ directory is also watched (if it exists). File events from config directories are routed to config-file jobs instead of code file jobs.

Config file event routing

The watcher's onEvents callback distinguishes config files from code files by path:

this.watcher.onEvents(events => {
  for (const e of events) {
    const scope = classifyConfigPath(e.path);
    if (scope) {
      // Config file — route to config indexer
      this.queue.enqueue({ kind: 'config-file', filePath: e.path, scope, event: e.type });
    } else if (this.supported.has(extname(e.path).toLowerCase()) || basenameParser.handles(e.path)) {
      // Code file — existing behavior
      this.queue.enqueue({ kind: 'file', filePath: e.path, event: e.type });
    }
  }
});

function classifyConfigPath(filePath: string): ConfigScope | null {
  // Global: ~/.insrc/{templates,feedback,conventions}/...
  if (filePath.startsWith(join(PATHS.insrc, 'templates')) ||
      filePath.startsWith(join(PATHS.insrc, 'feedback')) ||
      filePath.startsWith(join(PATHS.insrc, 'conventions'))) {
    return { kind: 'global' };
  }
  // Project: <repo>/.insrc/{templates,feedback,conventions}/...
  const insrcMatch = filePath.match(/^(.+)\/.insrc\/(templates|feedback|conventions)\//);
  if (insrcMatch) {
    return { kind: 'project', repoPath: insrcMatch[1]! };
  }
  return null;
}

Single file indexing

private async configFileEvent(
  filePath: string, scope: ConfigScope, event: 'create'|'update'|'delete',
): Promise<void> {
  if (event === 'delete') {
    await this.configStore.deleteEntry(configEntryId(scope, filePath));
    return;
  }

  // create or update
  await this.indexConfigFile(filePath, scope);
}

private async indexConfigFile(filePath: string, scope: ConfigScope): Promise<void> {
  const body = await readFile(filePath, 'utf-8');
  const hash = sha256(body).slice(0, 16);

  // Parse YAML frontmatter (category, namespace, language, name, tags)
  const meta = parseConfigFrontmatter(body);
  const content = stripFrontmatter(body);

  // Namespace: frontmatter field takes precedence, else derive from directory
  // e.g., templates/tester/vitest-unit.md → namespace 'tester'
  //        feedback/common/typescript.md  → namespace 'common'
  const namespace = meta.namespace ?? inferNamespaceFromPath(filePath);

  // Check if unchanged
  const id = configEntryId(scope, namespace, filePath);
  const existing = await this.configStore.getEntry(id);
  if (existing?.contentHash === hash) return;

  // Embed using same embedder as code entities
  const vector = await embedQuery(content);

  // Upsert into config store
  await this.configStore.upsertEntry({
    id,
    scope: formatScope(scope),
    category: meta.category,
    namespace,
    language: meta.language,
    name: meta.name,
    filePath,
    body: content.slice(0, 16_384),
    tags: meta.tags.join(','),
    updatedAt: new Date().toISOString(),
    contentHash: hash,
    vector,
  });
}

Config reindex

The config-reindex job drops all existing entries for the given scope and re-indexes from scratch. This handles stale entries from deleted files, renamed templates, or corrupted embeddings — cases where incremental indexing via config-full (which skips unchanged files) would leave orphans.

private async configReindex(scope: ConfigScope): Promise<void> {
  const scopeKey = formatScope(scope);
  log.info({ scope: scopeKey }, 'config reindex: dropping existing entries');

  // 1. Delete all entries for this scope
  await this.configStore.deleteByScope(scopeKey);

  // 2. Re-index everything (same as configFullIndex but no hash-skip)
  const dirs = scope.kind === 'global'
    ? [join(PATHS.insrc, 'templates'), join(PATHS.insrc, 'feedback'), join(PATHS.insrc, 'conventions')]
    : [join(scope.repoPath, '.insrc', 'templates'),
       join(scope.repoPath, '.insrc', 'feedback'),
       join(scope.repoPath, '.insrc', 'conventions')];

  let count = 0;
  for (const dir of dirs) {
    if (!await dirExists(dir)) continue;
    for (const filePath of walkConfigDir(dir)) {
      await this.indexConfigFile(filePath, scope);  // hash will differ since entry was deleted
      count++;
    }
  }

  log.info({ scope: scopeKey, count }, 'config reindex complete');
}

Difference from config-full: config-full is incremental — it skips files whose content hash hasn't changed. config-reindex deletes first, then re-indexes everything. Use config-full on startup (fast, skips unchanged); use config-reindex for explicit insrc config reindex CLI commands or when the store may be out of sync.

Same queue, same worker loop. Config jobs share the IndexQueue with code jobs. This means config re-indexing is naturally throttled alongside code re-indexing and doesn't compete for resources. The queue processes jobs sequentially, so a large code re-index won't block a config update for long (config files are small and fast to embed).

4.4 Search API

The config store exposes a search API parallel to the entity search in src/db/search.ts, but operating on the config_entries table.

interface ConfigSearchOpts {
  query:         string;                           // Natural language query
  namespace?:    string | string[];                // 'tester' | ['tester', 'common'] | ...
  category?:     'template' | 'feedback' | 'convention';
  language?:     Language;
  scope?:        ConfigScope;                      // filter to global or specific project
  limit?:        number;                           // default: 10
  boostProject?: string;                           // repoPath — boost project-scoped results
}

interface ConfigSearchResult {
  entry:    ConfigEntry;
  score:    number;       // cosine similarity
  boosted:  boolean;      // true if project boost applied
}

async function searchConfig(opts: ConfigSearchOpts): Promise<ConfigSearchResult[]>

Search modes

ModeUse caseHow it works
Semantic "How should I mock the database layer?" Embed query → vector search on config_entries with pre-filter on namespace/category/language

Project boost

When an agent searches config in the context of a specific project, project-scoped entries are boosted:

// Post-search re-ranking
results.sort((a, b) => {
  const aScore = a.scope.startsWith('project:' + repoPath)
    ? a.score * PROJECT_BOOST   // 1.5x
    : a.score;
  const bScore = b.scope.startsWith('project:' + repoPath)
    ? b.score * PROJECT_BOOST
    : b.score;
  return bScore - aScore;
});
Relationship to code search: Config search is a separate function from searchEntities() in src/db/search.ts. They operate on different LanceDB databases (config-store/ vs lance/) and return different types (ConfigEntry vs Entity). An agent may call both in the same step — e.g., the test agent's analyze step searches code entities for source structure and config entries for test feedback.

5. Feedback System

Agents accumulate feedback from user corrections, fix loop outcomes, review comments, and classification corrections. Feedback is persisted as markdown files and indexed into the config vector store for semantic retrieval.

5.1 Sources

SourceAgentWhenWhat gets recorded
User gate edits Tester, Pair, Delegate User provides edit feedback at any gate Generalized rule, not the specific instance
Claude review corrections Tester Claude flags issues during test/code review Pattern that led to successful fix (discard false positives)
Fix loop outcomes Tester, Pair Fix attempt succeeds after specific change Root cause pattern (skip one-off typos)
Fix loop exhaustion Tester Fix loop exhausts without resolution Failure pattern + what was tried (avoid repeating dead ends)
Classification corrections Tester Re-classification changes category Misclassification pattern
Code generation patterns Pair, Delegate Validate-and-fix loop succeeds Pattern the validator caught (import style, type errors, missing await)

5.2 Recording

Feedback recording happens at two points in an agent's lifecycle:

Inline (during execution)

When a user provides gate feedback or a fix loop reveals a generalizable pattern, the agent appends to the appropriate feedback file immediately. This ensures feedback survives even if the agent crashes later.

async function recordFeedback(opts: RecordFeedbackOpts): Promise<void> {
  const { content, namespace, language, repoPath, provider } = opts;

  // Determine scope and file path
  const isProjectSpecific = classifyFeedbackScope(content, repoPath);
  const dir = isProjectSpecific
    ? join(repoPath, '.insrc', 'feedback', namespace)
    : join(PATHS.insrc, 'feedback', namespace);
  const file = join(dir, `${language}.md`);

  // Generalize: ask LLM to extract the rule from the instance
  const generalized = await provider.chat({
    messages: [{
      role: 'system',
      content: GENERALIZE_FEEDBACK_SYSTEM,
    }, {
      role: 'user',
      content: `Raw feedback: ${content}\nExisting rules:\n${await readFileIfExists(file)}`,
    }],
  });

  // Deduplicate: LLM also checks if rule already exists
  if (generalized.text === 'DUPLICATE') return;

  // Append
  await appendFile(file, '\n' + generalized.text);

  // Re-index this file in config store
  const scope = isProjectSpecific
    ? { kind: 'project' as const, repoPath }
    : { kind: 'global' as const };
  await indexConfigEntry(file, scope);
}

Batch (during report step)

The agent's report step reviews all outcomes from the run and extracts generalizable patterns that weren't caught inline. The LLM receives the full run summary and existing feedback, and produces only net-new rules.

5.3 Retrieval

Agents retrieve relevant feedback during their analysis/setup phase using semantic search:

// In tester agent's analyze step — searches tester + common namespaces
const feedback = await searchConfig({
  query: `test generation patterns for ${state.detectedFramework}`,
  namespace: ['tester', 'common'],
  category: 'feedback',
  language: frameworkToLanguage(state.detectedFramework),

// Inject into state for use in generation prompts

The retrieved feedback is injected into LLM prompts as a "Previous learnings" section, scoped to the relevant language and task.

6.2 Resolution

Template lookup uses a combination of exact match and semantic search:

async function resolveTemplate(opts: TemplateQuery): Promise<ConfigEntry | null> {
  const { namespace, language, name, repoPath } = opts;

  // 1. Exact match: project-level template by name
  if (repoPath) {
    const projectPath = join(repoPath, '.insrc', 'templates', namespace, `${name}.md`);
    if (await fileExists(projectPath)) {
      return loadConfigEntry(projectPath, { kind: 'project', repoPath });
    }
  }

  // 2. Exact match: global template by name
  const globalPath = join(PATHS.insrc, 'templates', namespace, `${name}.md`);
  if (await fileExists(globalPath)) {
    return loadConfigEntry(globalPath, { kind: 'global' });
  }

  // 3. Semantic fallback: search config store (own namespace + common)
  const results = await searchConfig({
    query: `${name} template for ${language}`,
    namespace: [namespace, 'common'],
    category: 'template',
    language,
    boostProject: repoPath,
    limit: 1,
  });

  return results[0]?.entry ?? null;
}

Shadowing: If a project template has the same name as a global template (e.g., both have templates/tester/vitest-unit.md), the project version is used for exact lookup. Both remain searchable in semantic search, but the project version ranks higher due to the project boost.

7. Agent Integration

Each agent loads relevant config context during its setup/analyze phase. The config store is queried for templates, feedback, and conventions matching the agent's task.

Per-agent config loading

AgentStepNamespaces queriedWhat it loads
Tester analyze ['tester', 'common'] Test templates (by framework), test feedback (by language), language conventions
Pair propose ['pair', 'common'] Coding templates (by language), coding feedback, language conventions
Delegate invoke-planner ['delegate', 'common'] Execution conventions, project conventions (for plan context)
Planner analyze ['planner', 'common'] Plan structure templates, step granularity feedback
Designer analyze ['designer', 'common'] Design document templates, review checklists, project conventions
Default namespace resolution: Each agent definition includes a configNamespace field (e.g., testerAgent.configNamespace = 'tester'). The framework runner automatically queries [agent.configNamespace, 'common'] unless the step overrides it. This means agents don't need to hard-code their namespace in every search call.

Config context injection

Loaded config content is assembled into a configContext string on agent state and injected into LLM prompts:

// Prompt assembly (in write-tests, propose, execute-step, etc.)
const systemPrompt = [
  STEP_SYSTEM_PROMPT,
  state.configContext
    ? `\n## Project conventions and learnings\n${state.configContext}`
    : '',
].join('\n');
Token budget: Config context is capped at 2,000 tokens (~6,000 chars). If the combined templates + feedback + conventions exceed this, the agent prioritizes: (1) project conventions, (2) language-specific feedback, (3) templates. Lower-priority content is truncated.

Feedback write-back

Agents that generate feedback (tester, pair) call recordFeedback() to persist and re-index. The recording function is passed via the step context:

// In StepContext (framework/types.ts)
interface StepContext<S> {
  // ... existing fields ...
  recordFeedback?: (opts: RecordFeedbackOpts) => Promise<void>;
}
