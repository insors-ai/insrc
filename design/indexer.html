Indexer Design Document

insrc coding agent — v0.1 — March 2026

Overview

The indexer is the foundational subsystem of the insrc coding agent. Its responsibility is to parse source code across one or more repositories, extract meaningful code entities and their relationships, embed them for semantic search, and persist everything into SurrealDB — which serves as both a graph database (structural relationships) and a vector store (semantic search).

The result is a living Code Knowledge Graph that the agent queries to understand context, trace dependencies, find relevant code, and reason across repo boundaries.

Goals

  • Parse Python, Go, and TypeScript/JavaScript source files into structured entities
  • Model code relationships as a traversable graph (calls, imports, inheritance, etc.)
  • Support semantic search via vector embeddings stored alongside graph nodes
  • Handle large codebases efficiently using native file watching and incremental updates
  • Support multiple repositories simultaneously with cross-repo relationship linking
  • Be language-extensible — new grammars can be added without changing the core pipeline

Non-Goals

  • Real-time type inference or full compiler-level analysis
  • Execution or dynamic analysis of code
  • Replacing a language server (LSP) — this is a knowledge graph, not an IDE feature
  • Remote/published package deep indexing (stub nodes only for external deps)

User Data Directory

All insrc state is stored under ~/.insrc/ in the user's home directory. This includes the embedded database, daemon runtime files, and logs. 

~/.insrc/
  db/            ← SurrealDB surrealkv embedded store (persistent)
  daemon.pid     ← PID of the running background indexer process
  daemon.sock    ← Unix domain socket for CLI ↔ daemon IPC
  logs/
    daemon.log   ← indexer daemon output
SurrealDB is opened exclusively by the daemon process. The CLI communicates with it indirectly via the Unix socket — it never opens the database directly.

Repo Registry

Users register local repositories for indexing. The registry is stored as a registered_repo table in SurrealDB, so it persists alongside the index itself. No separate config file is needed. 

-- Repo registry table
DEFINE TABLE registered_repo SCHEMAFULL;
DEFINE FIELD path        ON registered_repo TYPE string;   -- absolute path on disk
DEFINE FIELD name        ON registered_repo TYPE string;   -- display name (inferred or user-set)
DEFINE FIELD addedAt     ON registered_repo TYPE datetime;
DEFINE FIELD lastIndexed ON registered_repo TYPE option<datetime>;
DEFINE FIELD status      ON registered_repo TYPE string;   -- 'pending' | 'indexing' | 'ready' | 'error'
DEFINE FIELD errorMsg    ON registered_repo TYPE option<string>;
DEFINE INDEX repo_path   ON registered_repo FIELDS path UNIQUE;

Background Daemon Architecture

The indexer runs as a persistent background process separate from the CLI. This allows the index to stay live across multiple agent sessions without re-scanning on every invocation.

Why a Separate Process (Not a Worker Thread)

ConcernSeparate ProcessWorker Thread

Shared by multiple sessions✓ Yes✗ No
Crash isolation✓ Yes✗ No

Daemon Lifecycle

  1. insrc daemon start forks a detached child process (process.detach()), redirects stdio to ~/.insrc/logs/daemon.log, writes PID to daemon.pid
  2. Daemon opens SurrealDB at surrealkv://~/.insrc/db
  3. Daemon checks Ollama availability and ensures the embedding model is installed (see below)
  4. Daemon reads all registered_repo rows, starts @parcel/watcher on each
  5. Daemon runs an initial full index on any repo with status = 'pending'
  6. Daemon opens Unix socket at ~/.insrc/daemon.sock and listens for JSON-RPC messages
  7. On insrc daemon stop: CLI sends shutdown message, daemon flushes in-flight writes, closes DB, deletes daemon.pid + daemon.sock, exits
On startup, if daemon.pid exists but the process is not alive (stale PID from a crash), delete the stale files and start fresh. Do not refuse to start.

Embedding Model Bootstrap

Before any indexing begins, the daemon checks that qwen3-embedding:0.6b is installed in Ollama and pulls it automatically if not. Pull progress is streamed to the daemon log and exposed via the daemon.status IPC method so the CLI can surface it to the user. 

// lifecycle.ts — run before first IndexJob is processed
async function ensureEmbeddingModel(ollama: Ollama): Promise<void> {
  const MODEL = 'qwen3-embedding:0.6b';

  const { models } = await ollama.list();
  if (models.some(m => m.name === MODEL)) return; // already installed

  log.info(`Pulling ${MODEL} from Ollama registry...`);
  setDaemonState({ modelPullStatus: 'pulling' });

  const stream = await ollama.pull({ model: MODEL, stream: true });
  for await (const event of stream) {
    // event: { status, completed, total }
    const pct = event.total ? Math.round((event.completed / event.total) * 100) : 0;
    setDaemonState({ modelPullStatus: 'pulling', modelPullPct: pct });
    log.info(`  ${event.status} ${pct}%`);
  }

  setDaemonState({ modelPullStatus: 'ready' });
  log.info(`${MODEL} ready`);
}
Status surface: While pulling, daemon.status returns { modelPullStatus: "pulling", modelPullPct: 42 } so the CLI can show a progress bar. Index jobs are held in the queue but not processed until the model is ready.
If Ollama itself is not running or unreachable at startup, the daemon logs a warning and continues. Entities are written to the DB without embeddings. A reembed job is enqueued for all registered repos and retried when Ollama becomes available.

Indexing Queue

The daemon maintains an in-memory FIFO queue of indexing jobs. File watcher events and repo.add messages both enqueue jobs. A single async worker drains the queue sequentially to avoid overloading Ollama with concurrent embedding requests. 

type IndexJob =
  | { kind: 'full',   repoPath: string }
  | { kind: 'file',   filePath: string, event: 'create' | 'update' | 'delete' }
  | { kind: 'reembed', repoPath: string }   // triggered if embedding model changes

Language Extraction Rules

TypeScript / JavaScript

Entity Extraction

  • function foo() / const foo = () => → Function node
  • class Foo → Class node
  • class Foo extends Bar → Class node + INHERITS edge
  • class Foo implements IBar → IMPLEMENTS edge (TypeScript only)
  • interface IFoo → Interface node
  • type Foo = ... → Type node
  • export keyword sets isExported: true

Import Extraction

  • import x from './foo' → IMPORTS edge, resolve to File node
  • import { x } from './foo' → same, record imported symbol
  • require('./foo') → same treatment as ESM import
  • import x from 'external-pkg' → IMPORTS edge to Module stub

Cross-Repo

  • Parse package.json dependencies + devDependencies
  • Emit DEPENDS_ON edges from Repo to Module stub nodes
  • If a dependency repo is locally indexed, resolve stubs to real Repo nodes

Python

Entity Extraction

  • def foo(): → Function node
  • async def foo(): → Function node, isAsync: true
  • class Foo: → Class node
  • class Foo(Bar): → Class node + INHERITS edge (supports multiple inheritance)
  • Decorators (@dataclass, @abstractmethod) stored as metadata
  • No explicit export — top-level definitions exported by default; __all__ overrides

Import Extraction

  • import os → IMPORTS edge to Module stub
  • from pathlib import Path → IMPORTS edge, record imported symbol
  • from . import foo / from ..utils import bar → resolve to File node

Cross-Repo

  • Parse requirements.txt, pyproject.toml ([tool.poetry.dependencies]), setup.cfg
  • Emit DEPENDS_ON edges

Go

Entity Extraction

  • func Foo() → Function node; exported if name starts with uppercase
  • func (r *Receiver) Foo() → Method node, linked to Receiver type
  • type Foo struct { ... } → Class node
  • type Foo interface { ... } → Interface node with method signatures extracted
  • type Foo = Bar → Type alias node

Import Extraction

  • import "fmt" → IMPORTS edge to Module stub
  • import "github.com/org/repo/pkg" → Module stub; resolve if locally indexed
  • Named imports: import alias "pkg" → same, store alias

Interface Satisfaction (Implicit Implements)

Go has structural typing — a struct implicitly implements an interface if it has all required methods. This must be computed statically after parsing.
  1. At index time, store all method signatures per struct and per interface
  2. After all files in a repo are parsed, run a method set matching pass:
    • For each interface, compute its required method set
    • For each struct, compute its implemented method set
    • If struct methods ⊇ interface methods → emit IMPLEMENTS edge
  3. Re-run this pass on incremental updates affecting structs or interfaces

Cross-Repo

  • Parse go.mod require directives
  • Emit DEPENDS_ON edges

Search Strategy

The agent must search code that spans repositories — a function in the current repo may call into a shared library in another repo. However, searching all registered repos is too broad and returns irrelevant noise. The solution is the dependency closure: at session start, resolve the transitive set of repos the current repo depends on, then scope every search to that set.

Scope rule: Not "current repo only" — too narrow, misses library code. Not "all registered repos" — too broad, returns unrelated results. Search the dependency closure of the active repo.

Resolving the Dependency Closure

At agent session start, traverse the DEPENDS_ON graph from the current repo node to collect all transitively reachable repos. This is a one-time startup cost, cached for the session duration. 

-- Resolve all repos reachable via DEPENDS_ON from the current repo
LET $closure = SELECT VALUE id FROM entity
  WHERE kind = 'Repo'
  AND id IN (
    SELECT VALUE out FROM depends_on
    WHERE in = $currentRepoId
    FETCH out, out->depends_on->entity
  );

-- Include the current repo itself
LET $closureRepos = array::union([$currentRepoId], $closure);

Scoped Vector Search

All semantic searches filter by repo IN $closureRepos to restrict results to the dependency closure.

SELECT id, name, kind, file, repo,
       vector::similarity::cosine(embedding, $query_vec) AS score
FROM entity
WHERE repo IN $closureRepos
  AND score > 0.75
ORDER BY score DESC
LIMIT 20

Three-Tier Search Strategy

TierScopeMethodWhen Used
1 — Primary Dependency closure Vector similarity + BM25 hybrid Every query; returns ranked candidates
2 — Expansion Graph neighbors of Tier 1 hits Graph traversal (CALLS, IMPORTS, DEFINES) When Tier 1 results need deeper context
3 — Never All registered repos — Not used; avoids noise from unrelated repos

Query Patterns

Semantic Search (Vector, Closure-Scoped)

LET $closureRepos = [...]; -- resolved at session start

SELECT id, name, kind, file, repo,
       vector::similarity::cosine(embedding, $query_vec) AS score
FROM entity
WHERE repo IN $closureRepos
  AND score > 0.75
ORDER BY score DESC
LIMIT 20

Find All Callers of a Function

SELECT <-calls<-entity.* FROM entity WHERE id = $target_id

Transitive Call Graph (What Does This Function Depend On?)

SELECT ->calls->entity.* FROM entity
WHERE id = $fn_id
FETCH ->calls->entity->calls->entity

Cross-Repo Impact (Who Uses This Exported Type?)

-- Scoped to closure repos for relevance
SELECT <-references<-entity.*
FROM entity
WHERE id = $type_id
  AND (<-references<-entity.repo) IN $closureRepos

Hybrid: Semantic Search + Graph Expansion (Closure-Scoped)

-- Tier 1: vector search within closure
LET $results = (
  SELECT id, vector::similarity::cosine(embedding, $query_vec) AS score
  FROM entity
  WHERE repo IN $closureRepos AND score > 0.8
  LIMIT 5
);

-- Tier 2: expand to graph neighbors
SELECT *, ->calls->entity.*, <-defines<-entity.*
FROM $results

Incremental Indexing

@parcel/watcher provides native OS-level file system events. On each event:

EventAction
createParse file → insert entities + relations
updateParse file → diff against existing → upsert changed, delete removed
deleteRemove all entities and edges originating from that file
Change detection: Each File node stores a content hash. On update events, recompute hash — skip re-indexing if unchanged. Handles editor save-without-change events.
Relation cleanup: When a file is re-parsed, all edges originating from its entities must be deleted before reinserting to prevent stale call/import edges from removed code.

After any struct or interface update in a Go repo, re-run the method set matching pass scoped to the affected repo.

Embedding Strategy

PropertyValue
Modelqwen3-embedding:0.6b via Ollama
Dimensions2048
Context window32K tokens — handles entire files without chunking
Instruction-awareYes — query and document embeddings use different prefixes
GranularityEntity-level (function body, class body)
TimingAfter entity extraction, before DB write — batched per file
FallbackWrite entity without embedding if Ollama unavailable; backfill later via reembed job
Why qwen3-embedding:0.6b? Ollama is already a dependency for the LLM. Qwen3-Embedding adds no new process or library. The 32K context window eliminates the need to chunk large entities. At 0.6B parameters it runs efficiently on CPU alongside the daemon.

Input Format — Instruction-Aware

Qwen3-Embedding distinguishes document embeddings (indexed code) from query embeddings (user question). Use the task instruction prefix only on queries, not on documents being indexed. 

// Indexing — no instruction prefix on document text
const docInput = 'function calculateTotal(items: Item[]): number { return items.reduce(...) }';

// Querying — prepend task instruction to user query
const queryInput = 'Instruct: Given a user question, retrieve relevant code snippets\nQuery: how is the cart total calculated';

const docVec  = await ollama.embed({ model: 'qwen3-embedding:0.6b', input: docInput });
const queryVec = await ollama.embed({ model: 'qwen3-embedding:0.6b', input: queryInput });

Document Format by Entity Kind

// Functions: signature + body
"function calculateTotal(items: Item[]): number { return items.reduce(...) }"

// Classes: declaration line + body
"class OrderService extends BaseService implements IOrderService { ... }"

// Go structs: type declaration + field list
"type OrderService struct { db *sql.DB; logger *zap.Logger }"

// Interfaces/Types: full declaration
"interface IOrderService { create(order: Order): Promise<Order>; cancel(id: string): void }"
Store the embedding model name alongside each entity. If the model changes, all embeddings must be regenerated. The reembed IndexJob handles this: it iterates all entities for a repo, re-embeds, and updates the DB without re-parsing source.

Open Questions

  1. Embedding model versioning — If the model changes from qwen3-embedding:0.6b to another, all embeddings must be regenerated. Store model name per entity as a migration signal; trigger a reembed job for all affected repos.
  2. Call resolution accuracy — Static call resolution is approximate. Dynamic dispatch, higher-order functions, and monkey-patching cannot be resolved without runtime information. Edges should be treated as best-effort hints, not ground truth.
  3. Go interface pass scaling — For large Go repos, method set matching could be expensive. Consider running it as a background job with a dirty-flag mechanism rather than blocking the incremental update path.
  4. Multi-language files — .tsx, .vue, .svelte, .mdx contain mixed content. Out of scope for Phase 1 but the parser registry should support compound parsers later.
  5. Symbol re-export chains — export { foo } from './bar' creates chains that require multi-hop resolution. Full resolution adds complexity to the resolver; consider deferring to Phase 5.
  6. Daemon auto-start — Should the CLI auto-start the daemon if it is not running when a command is issued (e.g. repo add)? Or should it require an explicit daemon start first? Auto-start is more ergonomic but harder to reason about.
  7. Multiple SurrealDB writers — SurrealDB embedded (surrealkv) is single-writer. The daemon is the sole writer; the CLI must never open the DB directly. This must be enforced architecturally — only daemon/ imports db/client.ts.
