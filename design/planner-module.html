Design a generic planning module for the insrc agent framework

Generated by insrc Designer Agent

Overview

Design document for a generic planning module in the insrc agent framework. The module has two layers: a utility library (Plan<T> types, status engine, markdown serialization, progress tracking) and a planner agent that orchestrates plan creation through an 8-step LLM-driven pipeline with user validation gates. 6 requirements addressed; 3 open questions documented in appendix.

Requirements Summary

  1. [FUNCTIONAL] The planner module shall define a generic Plan<T> and Step<T> type with fields for title, description, id, status, dependencies (array of step ids on Step only), optional assignee, optional notes, metadata (created/updated timestamps, author), support for nested sub-steps (tree structure), and a derived plan-level status computed from aggregate step statuses — DONE
  2. [FUNCTIONAL] The planner module shall implement status transition functions for steps that validate dependencies before allowing status changes to "done", detect circular dependencies in the dependency graph, and identify blocked steps with their blocking reasons — DONE
  3. [FUNCTIONAL] The planner module shall serialize plans to Markdown format with YAML frontmatter (title, author, created, updated) and checkbox lists, support round-trip fidelity for parsing back into the data structure, and support partial in-place updates to a single step's status without rewriting the entire file — DONE
  4. [FUNCTIONAL] The planner module shall provide progress tracking capabilities including step counts by status, percentage complete, and history/changelog of status transitions with timestamps — DONE
  5. [SYSTEM] The planner module shall support extensible plan types (ImplementationPlan, TestPlan, MigrationPlan) that inherit core Plan/Step types via generics while adding domain-specific fields: ImplementationPlan adds file paths, code references, and estimated complexity per step; TestPlan adds test categories (unit/integration/e2e), coverage targets, and fixtures; MigrationPlan adds rollback steps and data validation checkpoints — DONE
  6. [SYSTEM] The planner module shall be designed as a standalone TypeScript library within src/agent/planner/ with a clean public index.ts barrel export, ensuring all core engine behaviors (status transitions, progress tracking, MD serialization) are reusable by specialised plan types without duplication — DONE

Open questions (Q7–Q9) are documented in the Appendix.

Planner Agent Architecture

The planner is implemented as an agent using the src/agent/framework/ runner, following the same patterns as the designer agent. The utility layer (engine, progress, markdown) provides the building blocks; the agent orchestrates plan creation through an iterative, LLM-driven pipeline with user validation gates.

Step Pipeline

The agent runs an 8-step pipeline. Steps 4 and 7 are user validation gates. Steps 5→4 form a feedback loop when dependency issues are found.

┌─────────────────┐
│ analyze-request  │  Local LLM: parse user intent, detect plan type,
│                  │  identify goals and constraints
└───────┬─────────┘
        ▼
┌─────────────────┐
│ gather-context   │  Daemon RPC: search knowledge graph for relevant
│                  │  entities, files, dependencies in closure repos
└───────┬─────────┘
        ▼
┌─────────────────┐
│ draft-plan       │  Local LLM → draft steps with titles, descriptions,
│                  │  dependencies.  Claude → review and refine.
│                  │  Populates Plan<T> with typed steps.
└───────┬─────────┘
        ▼
┌─────────────────┐     ┌──────────┐
│ validate-plan    │◄────│  GATE    │  User reviews plan overview + step list
│                  │     │ approve  │  approve → resolve-dependencies
│                  │────►│ edit     │  edit    → re-draft with feedback (max 3)
│                  │     │ reject   │  reject  → restart from analyze-request
└───────┬─────────┘     └──────────┘
        ▼ (approve)
┌─────────────────┐
│ resolve-deps     │  detectCycles(), validateDependencies(),
│                  │  detectBlockedSteps()
│                  │  If issues → report and loop back to validate-plan
│                  │  If clean  → detail-steps
└───────┬─────────┘
        ▼ (clean)
┌─────────────────┐
│ detail-steps     │  Per plan type, enrich steps with domain data:
│                  │  - Implementation: file paths, code refs, complexity
│                  │  - Test: categories, coverage targets, fixtures
│                  │  - Migration: rollback steps, validation checkpoints
│                  │  Uses local LLM + daemon RPC for code analysis
└───────┬─────────┘
        ▼
┌─────────────────┐     ┌──────────┐
│ validate-details │◄────│  GATE    │  User reviews detailed plan
│                  │     │ approve  │  approve → serialize
│                  │────►│ edit     │  edit    → refine details (max 3)
│                  │     │ skip     │  skip    → serialize as-is
└───────┬─────────┘     └──────────┘
        ▼ (approve/skip)
┌─────────────────┐
│ serialize        │  toMarkdown(plan) → write artifact
│                  │  ctx.emit(output) → send to user
│                  │  next: null → done
└─────────────────┘

export const analyzeRequestStep: AgentStep<PlannerState> = {
  name: 'analyze-request',
  async run(state, ctx) {
    ctx.progress('Analyzing planning request...');

    const analysis = await ctx.providers.local.complete({
      system: ANALYZE_SYSTEM_PROMPT,
      messages: [{ role: 'user', content: buildAnalysisPrompt(state.input) }],
    });

    // Detect plan type from analysis if not explicitly provided
    const inferredType = detectPlanType(analysis.text, state.input.planType);
    ctx.progress(`Plan type: ${inferredType}`);

    return {
      state: { ...state, analysis: analysis.text, inferredPlanType: inferredType },
      next: 'gather-context',
    };
  },
};

3. draft-plan

export const draftPlanStep: AgentStep<PlannerState> = {
  name: 'draft-plan',
  async run(state, ctx) {
    ctx.progress('Drafting plan (local model)...');

    const draftSteps = await ctx.providers.local.complete({
      system: DRAFT_SYSTEM_PROMPT,
      messages: [{ role: 'user', content: buildDraftPrompt(state) }],
    });

    // Claude review and refinement
    ctx.progress('Reviewing plan (Claude)...');
    const claude = ctx.providers.claude;
    let refined = draftSteps.text;
    if (claude) {
      const review = await claude.complete({
        system: REVIEW_PLAN_PROMPT,
        messages: [{ role: 'user', content: refined }],
      });
      refined = review.text;
    }

    // Parse LLM output into Plan<T> structure
    const plan = parsePlanFromLLM(refined, state.inferredPlanType);

    ctx.writeArtifact('draft-plan.json', JSON.stringify(plan, null, 2));

    return {
      state: { ...state, draftSteps: refined, plan },
      next: 'validate-plan',
    };
  },
  artifacts: () => ['draft-plan.json'],
};

4. validate-plan (GATE)

const MAX_EDIT_ROUNDS = 3;

export const validatePlanStep: AgentStep<PlannerState> = {
  name: 'validate-plan',
  async run(state, ctx) {
    const plan = state.plan!;
    const summary = formatPlanForReview(plan, state.dependencyIssues);

    const reply = await ctx.gate({
      stage: 'plan-review',
      title: `Plan Review: ${plan.title}`,
      content: summary,
      actions: [
        { name: 'approve', label: 'Approve' },
        { name: 'edit',    label: 'Edit', hint: '<feedback>' },
        { name: 'reject',  label: 'Reject & restart' },
      ],
    });

    if (reply.action === 'approve') {
      return { state: { ...state, dependencyIssues: [] }, next: 'resolve-dependencies' };
    }

    if (reply.action === 'reject') {
      return {
        state: { ...state, plan: null, draftSteps: '', editRounds: {} },
        next: 'analyze-request',
      };
    }

    // Edit loop
    const rounds = (state.editRounds['plan'] ?? 0) + 1;
    if (rounds > MAX_EDIT_ROUNDS) {
      ctx.progress('Max edit rounds reached — proceeding with current plan.');
      return { state, next: 'resolve-dependencies' };
    }

    ctx.progress(`Refining plan with feedback (round ${rounds}/${MAX_EDIT_ROUNDS})...`);
    const refined = await refinePlanWithFeedback(state, reply.feedback ?? '', ctx);
    const plan2 = parsePlanFromLLM(refined, state.inferredPlanType);

    return {
      state: {
        ...state, plan: plan2, draftSteps: refined,
        editRounds: { ...state.editRounds, plan: rounds },
      },
      next: 'validate-plan',
    };
  },
};

5. resolve-dependencies

export const resolveDependenciesStep: AgentStep<PlannerState> = {
  name: 'resolve-dependencies',
  async run(state, ctx) {
    const plan = state.plan!;
    ctx.progress('Validating step dependencies...');

    const issues: string[] = [];

    // Check for cycles
    const cycle = detectCycle(plan.steps);
    if (cycle) {
      issues.push(`Circular dependency detected: ${cycle.join(' → ')}`);
    }

    // Validate each step's dependencies exist
    for (const step of flattenSteps(plan.steps)) {
      for (const depId of step.dependencies) {
        if (!findStepById(plan.steps, depId)) {
          issues.push(`Step "${step.title}" depends on non-existent step ID: ${depId}`);
        }
      }
    }

    // Detect blocked steps
    const blocked = detectBlockedSteps(plan);
    if (blocked.length > 0) {
      for (const b of blocked) {
        issues.push(`Step "${b.id}" is blocked: ${b.reasons.join(', ')}`);
      }
    }

    if (issues.length > 0) {
      ctx.progress(`Found ${issues.length} dependency issue(s) — returning to plan review.`);
      return {
        state: { ...state, dependencyIssues: issues },
        next: 'validate-plan',
      };
    }

    ctx.progress('Dependencies valid — no cycles or missing references.');
    return { state: { ...state, dependencyIssues: [] }, next: 'detail-steps' };
  },
};

Integration with Existing Plan Workflow

The planner agent replaces the current buildPlan() function in src/agent/tasks/plan.ts. Where the old flow was a single-shot LLM call, the agent provides:

  • Iterative refinement — user validation gates at plan review and detail review stages
  • Dependency validation — automatic cycle detection and blocked-step analysis before finalization
  • Type-specific enrichment — domain data added based on plan type (implementation, test, migration)
  • Crash recovery — checkpointed state with artifact validation on resume
  • Codebase awareness — daemon RPC searches the knowledge graph for relevant entities

The agent is invoked from the REPL when the classifier detects a plan intent:

// In src/agent/index.ts (REPL handler for 'plan' intent)

  channel: replChannel,
  options: { input: plannerInput, repo: session.repoPath },
  config,
  providers: { local: ollamaProvider, claude: claudeProvider },
});

Existing Plan Update Flow

When existingPlan is provided in the input, the analyze-request step detects update mode and adjusts the pipeline:

  • analyze-request → diffs the existing plan against the new request to identify what changed
  • draft-plan → modifies existing steps rather than creating from scratch
  • resolve-dependencies → validates the updated dependency graph
  • All other steps proceed as normal

Key Risks & Design Decisions

Agent Layer

  • LLM plan parsing reliability: parsePlanFromLLM() must robustly extract structured Plan<T> from free-form LLM output. Expect JSON extraction failures — implement fallback parsing and validation. Consider asking the LLM to output JSON directly with a schema.
  • Plan type auto-detection: detectPlanType() infers implementation/test/migration/generic from the analysis. If wrong, the detail-steps enricher adds irrelevant data. Mitigate via the validate-plan gate where the user can correct.
  • Dependency loop between resolve-dependencies and validate-plan: If the LLM keeps generating cycles on re-draft, the pipeline could loop. Limit total resolve→validate iterations (e.g., 2 attempts, then force-proceed with issues flagged).
  • Existing plan update mode: When existingPlan is provided, the agent must diff and merge rather than replace. This is significantly more complex than fresh creation — consider deferring to a later version.
  • Gate overhead on CPU: Each gate round-trip adds latency. On CPU-only inference, minimize the number of LLM calls per edit loop. Consider skipping Claude review in draft-plan when no Claude provider is available.
  • State serialization size: PlannerState includes the full Plan object plus analysis text and codebase findings. For large plans with many steps, checkpoint files could grow large. Monitor and compress if needed.

Utility Layer

  • Nested sub-step traversal: All functions (getProgressSummary, detectBlockedSteps, validateDependencies, computePlanStatus) must recursively walk the sub-step tree. A flat implementation would silently under-count steps.
  • Terminal state definition: validateDependencies and detectBlockedSteps must agree on what constitutes "terminal" (done, skipped). Export an isTerminalStatus() type-guard from types.ts.
  • Circular import prevention: Dependency direction: types.ts ← engine.ts / markdown.ts / progress.ts ← steps.ts / agent.ts ← index.ts. No cross-imports between engine, markdown, and progress.
  • Status history ownership: updateStepStatus in engine.ts should call recordStatusTransition from progress.ts (one-directional import: engine → progress).
  • Plan status write-back: computePlanStatus is pure — updateStepStatus must call it and update Plan<T>.status on every mutation.
  • Cross-tree dependencies: detectCycles and validateDependencies must use a flattened ID-keyed index of all steps (including nested).
  • Open questions Q7-Q9 (error recovery strategy, status precedence rules, persistence scope) must be resolved before finalizing the fromMarkdown API contract.
