2. DATA FLOW

  1. Caller invokes updateStepStatus(plan, stepId, newStatus)

    • Input: Plan, stepId, newStatus
    • Output: Updated Plan or error

  2. If newStatus === "done"

    • Call validateDependencies(plan, stepId)
      • If any dependency is not in {done, skipped}, throw an error with unresolved IDs
      • Else proceed to status update

  3. If plan's dependencies are modified

    • Call detectCircularDependencies(plan)
      • Run DFS over all steps using a flattened index of all steps (including nested)
      • If cycle detected, reject modification and return cycle path in error

  4. On successful transition

    • Update step status and metadata.updated timestamp
    • Store the transition in history via appendStatusTransition

  5. Post-transition

    • Call detectBlockedSteps(plan)
      • Recompute which pending steps should be blocked
      • Return list of { id, reasons }
    • Pass result to getStatusHistory for changelog entries

  6. All functions are re-exported via src/agent/planner/index.ts

2. DATA FLOW

Serialization (toMarkdown)

  1. Input: Plan<T> instance from src/agent/planner/types.ts
  2. Process:
    • Extract metadata fields (title, author, created, updated) from plan.metadata
    • Serialize metadata to YAML using js-yaml.dump() into a block
    • Traverse plan.steps recursively (depth-first) to build step list:
      • For each Step<T>:
        • Map status to checkbox format ([ ], [x], [-], etc.)
        • Embed step.id as HTML comment: <!-- id: step-1 -->
        • Include nested sub-steps in indented list
    • Concatenate YAML frontmatter fence (---) + body
  3. Output: Markdown string

Parsing (fromMarkdown)

  1. Input: Raw Markdown string
  2. Process:
    • Split input on first ---/--- fence pair
    • Parse YAML block using js-yaml.load() into PlanMarkdownMetadata
    • Scan remaining lines for checkbox patterns:
      • Match regex /^(\s*)- \[( |x|-|>)\]/
      • Reconstruct tree structure based on indentation depth
      • Extract embedded id from HTML comments (<!-- id: ... -->)
  3. Output: Plan<T> instance with populated metadata and steps

Partial Update (updateStepStatusInMarkdown)

  1. Input: Markdown string, step ID, new status
  2. Process:
    • Parse markdown to PlanMarkdown
    • Locate step by id in the tree structure
    • Update its status field
    • Re-serialize using toMarkdown logic (preserving all other metadata)
  3. Output: Updated Markdown string

Concerns from Sketch

  1. ID Embedding Strategy

    • Risk: HTML comments (<!-- id: ... -->) may not be robust across all Markdown parsers or editors.
    • Mitigation: Consider alternative embedding strategies such as:
      • Using custom attributes (e.g., data-id="step-1")

    • Risk: Parsing indentation depth and checkbox states may lose fidelity if Markdown is edited manually.

  3. Performance of In-Place Updates

    • Risk: Re-parsing large Markdown files for small updates could be inefficient.
    • Mitigation: Cache parsed structures or implement a streaming parser that only modifies the relevant section.

  4. YAML Dependency

    • Risk: js-yaml may not be present in package.json.
    • Mitigation: Add js-yaml as a dependency if missing, and verify compatibility with existing tooling.

  5. Nested Step Handling

    • Risk: Deeply nested steps might cause parsing ambiguity or stack overflow.
    • Mitigation: Implement recursion depth limits in parser; test with large step trees.

2. DATA FLOW

  1. Caller invokes updateStepStatus(plan, stepId, newStatus) in plan-engine.ts.
  2. Validation occurs: validateDependencies(plan, stepId) is called to ensure dependencies are met before transitioning.
  3. Transition recorded: On successful validation:
    • recordStatusTransition(step, fromStatus, newStatus, author) is invoked from progress-tracker.ts.
    • A new StatusTransitionEntry is created with timestamp = new Date().toISOString().
    • The entry is appended to step.statusHistory.
  4. Metadata updated: step.metadata.updated is set to the same timestamp.
  5. Consumer queries progress:
    • getProgressSummary(plan) walks the tree recursively.
    • Counts steps by status (pending, in-progress, done, blocked, skipped).
    • Computes percentageComplete = done / (total − skipped) * 100.
  6. Consumer queries history:
    • getStatusHistory(plan, stepId?) either returns a specific step's statusHistory or flattens all transitions into a sorted list by timestamp.

5. RISKS

Circular Import Risk

  • Mitigation: Ensure that progress-tracker.ts does not import from plan-engine.ts. Only plan-engine.ts imports from progress-tracker.ts.
  • This prevents a circular dependency and maintains clean module boundaries.

Timestamp Consistency

  • Risk: Inconsistent use of timestamps across the system.
  • Mitigation: All timestamps are enforced to be ISO-8601 UTC using new Date().toISOString(). A utility function like now() can be introduced for consistency if needed.

Performance for Large Plans

  • Risk: getStatusHistory(plan) flattens and sorts all transitions, which may become slow on very large plans.
  • Mitigation: This is flagged as a future concern. Consider pagination or lazy evaluation once performance characteristics are known.

Denominator Ambiguity in Percentage Calculation

  • Risk: Misinterpretation of how skipped steps affect percentageComplete.
  • Mitigation: Documented convention: Skipped steps are excluded from the denominator, so only non-skipped steps contribute to completion percentage. This is reflected in JSDoc for getProgressSummary.

Tree Traversal Assumption

  • Risk: Incorrect assumption that all plans have a tree structure.
  • Mitigation: Explicitly documented in getProgressSummary() that recursive traversal is mandatory due to requirement 1's definition of plan structure.
