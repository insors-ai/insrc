Test Agent

Step-based testing agent with scenario-level planning, failure classification, and Pair agent handoff for implementation bugs.

1. Overview

The Test Agent replaces the single-turn runTestPipeline with a checkpointable, resumable agent on the existing framework. It generates, writes, and executes tests for user-specified files or modules, iterating until all tests pass or surfacing implementation bugs to the Pair agent.

The agent operates on user-directed scope — the user requests tests for specific files, modules, or components (e.g., "write real tests for the planner agent"). The agent does not autonomously discover untested code across the repo.

Key decision: The test agent never modifies implementation code. When it detects an implementation bug, it invokes the Pair agent (debug mode) as a sub-agent with full context, after user approval.

2. Design Principles

  1. Test code only. The agent owns test files exclusively. Implementation bugs are handed off to the Pair agent.
  2. Scenario-level planning. Test plans enumerate specific scenarios (happy path, edge cases, error conditions) — not just "test file X."
  3. Mock vs live decision up front. The agent asks the user whether mock or real tests are needed (if not already specified). This fundamentally changes the test plan shape.
  4. Claude reviews the plan. The generated test plan is validated by Claude for coverage and correctness before user review.
  5. Per-file write+execute. Tests are grouped by file. Each file is written and executed to green before moving to the next.
  6. One gate, then autonomous. The user reviews the test plan once. After approval, the agent runs autonomously (unless it needs to hand off an implementation bug).
  7. Optional test code review. If the user requests it at the gate, Claude reviews generated test code before execution.

3. Agent Flow

+─────────+ │ analyze │ +────┬────+ │ +───────────▼────────────+ │ generate-test-plan │ │ (LLM + Claude review) │ +───────────┬────────────+ │ +───────────▼────────────+ │ review-test-plan │◄──── edit feedback │ (user gate) │────► reject → analyze +───────────┬────────────+ │ approve ┌───────────▼────────────┐ │ per-group loop │ │ │ ┌─────▼──────┐ │ │ write-tests │ │ │ (codegen) │ │ +─────┬───────+ │ │ │ ┌─────▼──────┐ │ │ review-tests│ Claude + user │ │ (gate) │ (can be auto) │ +─────┬───────+ │ │ approve │ ┌─────▼──────┐ │ │execute-tests│ │ │ (fix loop) │ │ +─────┬───────+ │ │ │ ├── all pass ──► next group │ │ ├── test issue ──► fix test ──► re-run │ │ └── impl bug ──► accumulate, next group │ ┌────────────────────────-┘ │ all groups done │ ┌─────▼──────────┐ │ impl-bug gate │ (if any accumulated) │ → Pair handoff │ +─────┬──────────+ │ ┌─────▼─────┐ │ report │ +────────────+
Per-group loop and checkpoints: Each step transition writes a framework checkpoint, so a group cycles through 3 checkpoints: write-tests → review-tests → execute-tests. This is by design — if the agent is interrupted mid-group, it resumes at the last completed step rather than re-doing all prior groups.

3.1 analyze

local LLM + tools

Investigates the user-specified files/modules to understand what needs testing.

  • Uses investigate() helper with a test-focused query built from the user's message
  • Calls detectFramework(repoPath) to identify the test framework (vitest, jest, pytest, go, etc.)
  • For each source file in scope, calls findTestFile() to discover existing tests
  • Reads existing test files to understand current coverage and test style
  • For live tests: queries the daemon to discover config patterns, service URLs, DB setup
  • Loads config context via loadConfigContext(ctx, 'tester', language, repoPath) — retrieves conventions, feedback, and templates from the config vector store (see Section 9)
  • Performs targeted feedback search via ctx.searchConfig({ query: 'test generation patterns for ${framework}', namespace: ['tester', 'common'], category: 'feedback', ... }) for framework-specific learnings
  • Config context is stored on state.configContext and injected into prompts for generate-test-plan and write-tests

State updates: investigationSummary, detectedFramework, existingTests[], sourceFiles[], configContext

Next: generate-test-plan

3.2 generate-test-plan

local generate → Claude review

LLM generates a structured test plan at scenario-level granularity, then Claude reviews it.

Generation:

  • Builds prompt from investigation summary, source files, existing tests, detected framework, and user message
  • LLM produces JSON matching the TestPlan schema
  • Each entry specifies: target file, test file, kind (unit/live), scenarios, fixtures/setup, priority
  • For live tests: auto-populates setup config (service URLs, env vars, DB paths) discovered during analysis
  • Retry up to 2x on malformed JSON; fallback: one entry per source file

Claude review:

  • Claude validates the plan for: scenario completeness, edge case coverage, correct test categorization, realistic fixtures
  • If CHANGES_NEEDED: feeds Claude's feedback back to local LLM for refinement (up to 2 rounds)
  • Writes test-plan.json artifact

Mock vs live decision: If the user hasn't specified, the agent asks at this stage (or infers from the message and confirms). This determination drives the kind field on each TestPlanEntry.

3.3 review-test-plan

gate

Presents the test plan to the user for review.

Gate display: Formatted markdown showing each entry with target file, test file, kind, scenarios, setup config (for live tests), and priority.

Actions:

ActionBehaviorNext step

3.4 write-tests

local generate → Claude validate

Generates test code for the current group of test plan entries and writes it to disk.

For the current group:

  1. Read the target source file(s) and existing test file(s) for context
  2. Build prompt from: entry scenarios, fixtures, setup config, test framework conventions
  3. Call generateAndValidate() — local generates unified diff, Claude validates
  4. Call applyApprovedDiff() to write the test file to disk
  5. Record TestFileResult with status: 'written'

On codegen failure: Record error in TestFileResult, skip to next group. Don't block the pipeline.

Next step return values:

  • Codegen succeeds → next: 'review-tests'
  • Codegen fails → next: 'write-tests' (advance to next group) or next: 'report' (last group)

3.5 review-tests

Claude review → gate

Reviews the generated test code before execution. This step always runs but the gate can be auto-approved.

Claude review: Checks test code for assertion correctness, test independence, scenario coverage vs plan, framework conventions.

Gate display: Shows the test file diff with Claude's review summary (issues found or "looks good").

Actions:

ActionBehaviorNext step
approveProceed to executionexecute-tests
edit <feedback>Re-generate with feedbackwrite-tests (same group)
skipSkip this group's testswrite-tests (next group) or report
Auto-approve: When the user selected "approve" (not "approve with review") at the test plan gate, the review-tests gate is auto-approved — Claude's review still runs and its feedback is logged, but the gate doesn't pause for user input. When the user selected "approve with review" at the plan gate, each group's test code is presented for explicit user approval.

3.6 execute-tests

local fix → Claude classify + escalation

Runs the current group's test file. On failure, classifies first, then routes to the appropriate fix strategy. This step does NOT use the existing runTestsAndFix() helper — it implements its own classify-then-fix loop (see Section 6.1).

For live tests: Prerequisite check before execution. If required infrastructure is unavailable (Ollama down, daemon unreachable, DB not connected), skip with a clear message rather than running and failing.

Flow per group:

  1. Run tests via runTests() (from test-runner.ts)
  2. If all pass → mark status: 'passing', proceed to next group
  3. Classify each failure (see Section 6) — sends test output, test code, and implementation code to LLM
  4. Route by classification:
    • Test issue → enter test fix loop (local fix up to 3 attempts, Claude escalation up to 2 rounds)
    • Implementation bug → accumulate in implementationBugs[], mark status: 'impl-bug', move on
    • Setup/config issue → attempt config fix (1 try), else skip with status: 'setup-skipped'
  5. After fix attempt: re-run tests, re-classify if still failing (classification may change — what looked like a test issue may reveal an impl bug after the obvious test error is fixed)
  6. Exhaustion: If test fix loop exhausts (3 local + 2 Claude rounds) without passing, mark status: 'fix-exhausted'

Next step return values:

  • More groups remaining → next: 'write-tests' (advance currentEntryIndex)
  • Last group done, impl bugs accumulated → next: 'impl-bug-gate'
  • Last group done, no impl bugs → next: 'report'

3.7 impl-bug-gate

gate

Presents accumulated implementation bugs and offers Pair agent handoff. Only reached if implementationBugs.length > 0 after all groups are done. See Section 7 for full details.

Next step return values:

  • User selects fix/auto-fix and Pair completes → next: 'report'
  • User selects skip → next: 'report'

3.8 report

local

Produces a final summary after all test files have been processed.

  • Summary of: files tested, scenarios covered, pass/fail counts per file
  • List of any implementation bugs found and whether they were fixed (via Pair agent)
  • List of skipped live tests with prerequisite failures
  • Writes test-report.md artifact
  • Emits done message

5. Unit vs Live Tests

The agent handles these as fundamentally different execution strategies:

Unit / Mock TestsLive / Integration Tests
Dependencies None — self-contained with mocks/stubs External services (Ollama, daemon, Claude API, DB)
Setup Standard test framework setup Service connectivity, env vars, config loading
Prerequisites None Ping services, verify connectivity. Skip if unavailable.
Execution Generate → run → fix loop → done Generate → check prereqs → run → fix loop → done
Fix targets Test code only (assertions, mocks, imports) Test setup/config, or report as infra issue
Failure mode Test issue or implementation bug Test issue, implementation bug, or infra/config issue
Config discovery N/A Auto-populated via daemon search during analyze step
Mock vs live decision: If the user hasn't specified, the agent infers from the message and confirms at the test plan review gate. Keywords like "real tests", "integration", "live", "end-to-end" signal live tests. Default is unit/mock.

5.1 Test File Naming Conventions

The agent enforces language-specific naming conventions that distinguish unit from live test files. This is not cosmetic — the execute-tests step uses the file name pattern to decide whether to run prerequisite checks before execution.

LanguageUnit testsLive / integration testsLocation
TypeScript / Node *.test.ts, *.spec.ts *.live.test.ts, *.live.spec.ts __tests__/ alongside source, or tests/ at repo root
Python test_*.py test_*_live.py tests/ at package root, mirroring source structure
Go *_test.go *_live_test.go Same package directory (Go convention)

Detection logic in execute-tests:

function isLiveTest(testFile: string): boolean {
  // Node / TypeScript
  if (/\.live\.(test|spec)\.[tj]sx?$/.test(testFile)) return true;
  // Python
  if (/test_\w+_live\.py$/.test(testFile))            return true;
  // Go
  if (/_live_test\.go$/.test(testFile))                return true;
  return false;
}

When isLiveTest() returns true, the step runs prerequisite checks (service pings, env var presence, DB connectivity) before test execution. If prerequisites fail, the file is skipped with status: 'prereq-not-met' rather than run-and-fail.

Generation enforcement: The generate-test-plan step includes the naming convention in its prompt, and the review-tests step validates that generated file names match the convention for their kind. If a live test is named steps.test.ts (missing .live infix), the review flags it for correction before execution.
Existing test discovery: The analyze step's findTestFile() call discovers existing tests regardless of naming. The convention applies only to generated test files. If the project has pre-existing live tests under a different naming scheme, the agent matches the existing pattern instead.

6. Failure Classification

When a test fails, the agent must classify the failure before deciding how to fix it. This is a critical capability that distinguishes the test agent from a simple generate-and-run pipeline.

Classification Strategy

The agent sends the failing test output, test code, and implementation code to the LLM with a classification prompt. The LLM determines:

CategorySignalAction
Test issue Wrong expected value, bad mock setup, missing import, incorrect assertion syntax Fix the test code (local fix loop, Claude escalation)
Implementation bug Function returns unexpected result, missing error handling, wrong logic, null where value expected Surface to user → gate → invoke Pair agent (debug mode)
Setup/config issue Connection refused, timeout, missing env var, wrong URL (live tests only) Fix test setup, or skip with prerequisite-not-met message

Classification Prompt

The LLM receives: failing test output, test code snippet, implementation code snippet, and the original test scenario description. It must respond with a JSON classification:

{
  "category": "test_issue" | "implementation_bug" | "setup_issue",
  "confidence": "high" | "medium" | "low",
  "reasoning": "The test expects X but the function returns Y because...",
  "suggestedFix": "Change the assertion from X to Y" | "The function has a bug in..."
}
Low confidence: When classification confidence is low, the agent defaults to assuming a test issue first. If the test fix loop exhausts without success, it reclassifies as a potential implementation bug.

6.1 Test Execution Framework

The test agent does not use the existing runTestsAndFix() from shared/test-runner-helper.ts. That helper has a blind fix loop: it treats every failure as a test issue and applies diffs without classification. The test agent needs classification-first routing, so it builds its own execution loop from lower-level primitives.

Why not runTestsAndFix()

runTestsAndFix() behaviorTest agent needs
Runs fix loop immediately on any failure Classify failure before deciding whether to fix
Always targets test code (or impl code blindly) Route test issues to fix loop, impl bugs to Pair, setup issues to config fix or skip
Escalates to Claude only after N local fix attempts Uses Claude for classification on first failure, not just escalation
No re-classification after fix attempt Re-classifies after each fix attempt (category may change)
Tracks fix history internally, returns single result Agent state tracks per-failure classification, fix attempts, category transitions

Primitives reused from existing code

The execution framework reuses these functions directly:

  • runTests(testFile, repoPath, framework) — execute tests and get structured TestResult
  • formatTestResultForLLM(result) — format failures for LLM consumption
  • applyFixDiff(diff, repoPath) — apply a unified diff from an LLM fix response
  • detectFramework(repoPath) — auto-detect test framework

Classify-then-fix loop

function executeAndFix(testFile, repoPath, state):
  result = runTests(testFile, repoPath, state.detectedFramework)
  if result.passed → return 'passing'

  for attempt in 1..MAX_ATTEMPTS:
    classification = classifyFailure(result, testCode, implCode)

    switch classification.category:
      'implementation_bug':
        accumulate(state.implementationBugs, classification)
        return 'impl-bug'

      'setup_issue':
        if attempt === 1:
          tryFixSetup(classification.suggestedFix)
          result = runTests(...)
          if result.passed → return 'passing'
        return 'setup-skipped'

      'test_issue':
        diff = generateTestFix(result, testCode, classification)
        applyFixDiff(diff, repoPath)
        result = runTests(...)
        if result.passed → return 'passing'
        // re-classify on next iteration (category may have changed)

  // Exhausted — try Claude escalation
  for round in 1..MAX_CLAUDE_ROUNDS:
    diff = claudeEscalation(result, testCode, fixHistory)
    applyFixDiff(diff, repoPath)
    result = runTests(...)
    if result.passed → return 'passing'
    // re-classify after Claude fix too
    classification = classifyFailure(result, testCode, implCode)
    if classification.category !== 'test_issue':
      accumulate(state.implementationBugs, classification)
      return 'impl-bug'

  return 'fix-exhausted'
Constants: MAX_ATTEMPTS = 3 (local fix attempts per test issue), MAX_CLAUDE_ROUNDS = 2 (Claude escalation rounds). These match the existing runTestsAndFix() defaults for consistency, but are now applied only to test-issue failures rather than all failures.

7. Pair Agent Handoff

When the test agent detects implementation bugs, it accumulates them across all test files and hands them off to the Pair agent as a single consolidated batch. This avoids N separate Pair invocations and gives Pair the full picture of related bugs.

Accumulation

During the execute-tests loop, when a failure is classified as an implementation bug:

  1. Record the bug in implementationBugs[] with full context (test file, test name, source file, expected vs actual, classification reasoning)
  2. Mark the test file as status: 'impl-bug'
  3. Do not invoke Pair immediately — continue to the next test file

Batch Handoff Gate

After all test files have been processed through write+execute, if implementationBugs.length > 0, the agent presents a single gate:

Gate display:

## Implementation Bugs Found (3)

1. [src/agent/planner/steps.ts] analyzeRequestStep
   Test: "classifies 'test' plan type from test-related prompt"
   Expected: planType === 'test'
   Actual: planType === 'implementation'
   Confidence: high — function lacks keyword matching for test-related prompts

2. [src/agent/planner/steps.ts] draftPlanStep
   Test: "generates dependencies that reference existing step IDs"
   Expected: all deps in stepIds set
   Actual: dep "step-0" not found — off-by-one in ID generation
   Confidence: high — step ID prefix mismatch

3. [src/db/search.ts] hybridSearch
   Test: "returns empty array for zero-match query"
   Expected: []
   Actual: throws NullReferenceError
   Confidence: high — missing null check on empty result set

Actions:

ActionBehavior
fixInvoke Pair agent (interactive) with full bug TODO — user participates in debugging
auto-fixInvoke Pair agent (auto-approve) with full bug TODO — autonomous debugging

Single Pair Invocation

The Pair agent is invoked once in debug mode with a consolidated TODO:

PairInput {
  message: `Fix the following implementation bugs found by test execution:

  TODO:
  1. [ ] [src/agent/planner/steps.ts:42] analyzeRequestStep — lacks keyword
       matching for test-related prompts, returns 'implementation' instead of 'test'
  2. [ ] [src/agent/planner/steps.ts:108] draftPlanStep — off-by-one in step
       ID generation, dep "step-0" not found
  3. [ ] [src/db/search.ts:67] hybridSearch — missing null check on empty
       LanceDB result set, throws NullReferenceError

  Fix each bug. The associated tests will be re-run after all fixes are applied.`,

  codeContext: /* concatenated: all affected source files + test files + failure output */,
  mode: 'debug',
}

Pair works through the TODO items sequentially using its standard propose → review-gate → apply → validate loop.

Re-run After Fix

  1. After Pair completes, re-execute only the previously-failing test files (those with status: 'impl-bug')
  2. Update TestFileResult statuses based on results
  3. If still failing: reclassify — may now be a test issue that needs fixing, or Pair's fix was incomplete
  4. One re-run cycle only — if bugs persist after Pair's fix, they're surfaced in the report as unresolved
Sub-agent invocation: The Pair agent is invoked via runAgent() with a TestChannel (auto-approve) when the user selects "auto-fix", or with a ReplChannel (interactive) when the user selects "fix". This allows the user to participate in the debugging if they want to.
Why batch, not per-file: Related bugs often share root causes (e.g., two tests fail because the same utility function has a bug). A single Pair invocation with full context can fix the root cause once rather than patching symptoms file-by-file. It also avoids the overhead of N sub-agent launches with separate investigation phases.

8. Review Strategy

Test Plan Review (always)

The generated test plan is always reviewed by Claude before presentation to the user. Claude checks for:

  • Scenario completeness — missing edge cases, error paths, boundary conditions
  • Correct categorization — unit vs live correctly assigned
  • Realistic fixtures — mock strategies that match the actual code
  • Setup correctness — for live tests, valid service URLs and config

Test Code Review (every group)

Claude reviews every generated test file at the review-tests step (Section 3.5). Claude checks for:

  • Assertion correctness — are the expected values realistic?
  • Test independence — no shared mutable state between tests
  • Coverage of specified scenarios — does the code match the plan?
  • Framework conventions — correct use of describe/it/beforeEach etc.

The review always runs. What differs is the gate behavior:

  • "approve" at plan gate → review-tests gate auto-approves (Claude's feedback is logged but doesn't pause)
  • "approve with review" at plan gate → review-tests gate pauses for user input on each group

On issues (whether from Claude or user feedback), the test is re-generated via write-tests with the feedback incorporated.

9. Test Feedback, Config & Memory Integration

The test agent uses the Config Management Framework (see design/config-management.html) and the Memory Framework (see agent.html §Context Window & Memory) for storing and retrieving test feedback, templates, and conventions. The agent definition sets configNamespace: 'tester'.

9.1 Config Context Loading

During the analyze step, the tester loads config context using the same pattern as other agents:

// 1. Load general conventions + feedback + templates
let configContext = await loadConfigContext(ctx, 'tester', language, state.input.repoPath);

// 2. Targeted feedback search for framework-specific learnings
if (ctx.searchConfig) {
  const feedback = await ctx.searchConfig({
    query: `test generation patterns for ${state.detectedFramework}`,
    namespace: ['tester', 'common'],
    category: 'feedback',
    language: frameworkToLanguage(state.detectedFramework),
    limit: 5,
    boostProject: true,
  });
  if (feedback.length > 0) {
    configContext += '\n\n### Tester Learnings\n' + feedback.map(f => f.entry.body).join('\n');
  }
}

// 3. Resolve testing template (e.g., vitest-unit, pytest-live)
if (ctx.resolveTemplate) {
  const template = await ctx.resolveTemplate({
    namespace: 'tester',
    language: frameworkToLanguage(state.detectedFramework),
    name: `${state.detectedFramework}-unit`,
    repoPath: state.input.repoPath,
  });
  if (template) configContext += '\n\n### Test Template\n' + template.body;
}

state.configContext = configContext;

The config context is stored on state.configContext (loaded once in analyze, reused across generate-test-plan, write-tests, and fix loop steps). The loadConfigContext() helper from src/agent/tasks/shared/config-context.ts applies a 6,000-char cap with priority ordering: conventions > feedback > templates.

9.2 What the tester records

The tester agent calls ctx.recordFeedback() with namespace: 'tester' at these points (fire-and-forget, never blocks the agent):

SourceWhenWhat gets recorded
User edits at review-tests gate User provides edit feedback on generated tests Generalized rule (e.g., "don't mock the database" → "use real DB connections for integration tests")
Claude review corrections Claude flags issues during test code review Pattern that led to successful fix (discard false positives)
Fix loop outcomes Fix attempt succeeds after specific change Root cause pattern (skip one-off typos)
Fix loop exhaustion Fix loop exhausts without resolution Failure pattern + what was tried (avoid repeating dead ends)
Classification corrections Re-classification changes category after fix attempt Misclassification pattern
High failure rate (report step) Session-level: >50% of test entries failed or were skipped Quality signal about test generation approach for this project 
// Example: recording feedback at review-tests gate edit action
if (ctx.recordFeedback && feedback) {
  ctx.recordFeedback({
    content: `User edited test code: ${feedback}`,
    namespace: 'tester',
    language: frameworkToLanguage(state.detectedFramework),
    repoPath: state.input.repoPath,
    provider: ctx.providers.local,
  }).catch(() => {});  // fire-and-forget
}

9.3 Feedback storage

Feedback files follow the config framework's markdown-with-frontmatter format, organized by namespace:

  • Global: ~/.insrc/feedback/tester/{language}.md — framework-level patterns (vitest conventions, pytest idioms)
  • Project: <project>/.insrc/feedback/tester/{language}.md — project-specific patterns (custom types, internal API quirks)

The recordFeedback() function handles: LLM-based generalization (extracts the rule from the instance), deduplication (checks if rule already exists), scope classification (global vs project), file persistence with frontmatter, and immediate re-indexing into the config vector store. See Config Management §5 for details.

9.4 Conversation persistence & session tags

The tester agent integrates with the persistent conversation store and session tagging:

  • Turn recording: The REPL records the tester's output as a conversation turn (same as all pipeline agents). This persists in the conversation_turns LanceDB table and is available for L3b semantic retrieval in future turns.
  • Session tag: After completion, the REPL sets ctx.setTag('[test]', summary) with test outcome details. Downstream agents (e.g., Pair debug mode) can retrieve this via ctx.getTag('[test]').
  • Compaction safety: Conversation turns with directive patterns (e.g., "never mock the database in tests") are preserved during compaction as type: 'directive' entries that survive all tier transitions.

9.5 Configurable budget

The tester agent's context window budget is derived from config.models.context.local (default 32K) via createBudget(contextWindowSize). Budget layers scale proportionally: L4 (code entities) gets ~25% of the window, ensuring sufficient room for source files + test files + investigation results.

  sourceFiles:           string[];

  // Test plan
  testPlan:              TestPlan | null;
  configContext?:        string | undefined;  // assembled from config store (feedback + templates)

  // Execution
  currentEntryIndex:     number;
  fileResults:           TestFileResult[];
  reviewTests:           boolean;        // Claude reviews test code before execution
  allPassing:            boolean;

  // Bug tracking
  implementationBugs:    ImplementationBug[];

  // Tracking
  editRounds:            Record<string, number>;
  filesChanged:          string[];
  summary?:              string | undefined;
}

TestFileResult

interface TestFileResult {
  testFile:       string;
  targetFile:     string;
  kind:           'unit' | 'live';
  status:         'pending' | 'written' | 'passing' | 'failing'
                  | 'impl-bug' | 'fix-exhausted'
                  | 'prereq-not-met'     // live test: infrastructure unavailable
                  | 'setup-skipped'      // setup/config fix failed
                  | 'codegen-failed'     // write-tests couldn't generate code
                  | 'skipped';           // user skipped at review-tests gate
  scenarios:      string[];              // from test plan entry
  fixAttempts:    number;
  claudeRounds:   number;
  testResult?:    TestResult | undefined;
  error?:         string | undefined;
  filesWritten:   string[];
}

}
