VS Code Plugin — Implementation Plan

18 installable, testable segments — from scaffold to marketplace

Overview

The insrc VS Code extension is a thin UI layer over the insrc daemon. All intelligence (LLM routing, tool execution, graph queries, memory) lives in the daemon. The extension owns: chat panel, inline diff, status bar, navigation, settings, and setup wizard.

Each segment below is independently installable via code --install-extension insrc-X.X.X.vsix and testable. Segments build on each other but each produces a working extension with incremental functionality.

Key decisions: Plugin-only (no VS Code fork). Inline diff via virtual document + vscode.diff inline mode. Cross-repo navigation via daemon-backed TreeView. Daemon auto-managed by extension (spawn, health poll, auto-restart).

Segment 2: Daemon Lifecycle

~3 days depends: Seg 1

Scope

  • daemon/lifecycle.ts — ensureDaemon(): check socket, check PID, spawn detached, poll for ready
  • daemon/rpc.ts — JSON-RPC client: call(method, params) over Unix socket with timeout
  • On activation: call ensureDaemon(), show progress notification
  • Health polling: 30s interval via daemon.status RPC
  • Auto-restart on crash (3 attempts, then notification)
  • On deactivation: disconnect (do NOT stop daemon — shared across windows)

Files

src/daemon/lifecycle.ts, src/daemon/rpc.ts
src/extension.ts (updated)

Test Milestone

  1. Stop daemon manually (insrc daemon stop). Open VS Code. Extension auto-starts daemon. Verify with insrc daemon status.
  2. Kill daemon process (kill -9). Extension auto-restarts within 30s.
  3. Open two VS Code windows. Both connect to the same daemon (check PID).
  4. Close all VS Code windows. Daemon keeps running (detached).

Segment 7: Inline Diff

~4 days depends: Seg 5

Scope

  • ui/diffManager.ts — TextDocumentContentProvider for proposed content
  • On activation: write "diffEditor.renderSideBySide": false to .vscode/settings.json
  • When daemon sends a diff: register virtual document, open with vscode.diff(original, proposed)
  • CodeLens provider: Accept / Reject / Edit buttons above the diff
  • Accept: write virtual doc to disk, close diff tab, notify daemon
  • Reject: close diff tab, restore original, notify daemon
  • Edit: open input box, send feedback to daemon, re-propose
  • Multi-file diffs: one tab per file, progress indicator in chat
  • Pipeline status in editor title: "local generating..." → "Claude validating..." → "APPROVED"
