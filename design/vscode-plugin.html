VS Code Plugin Design Document

insrc VS Code extension — v0.1 — March 2026

Overview

The insrc VS Code extension provides a native graphical interface for the insrc agent directly inside the IDE. It is a thin UI layer — all intelligence, routing, and tool execution lives in the insrc daemon. The extension connects to the already-running daemon over a local IPC channel and renders its output.

Installing the extension is the complete setup experience. The extension handles daemon installation, Ollama detection, model pulling, and first-run key configuration through an in-IDE wizard. The user never needs to touch a terminal to get started.

Architecture boundary: The extension owns UI only — chat panel, diff review, status bar, settings UI, and setup wizard. The daemon owns everything else: intent classification, LLM routing, tool execution, graph queries, session memory. The extension communicates with the daemon exclusively over a local WebSocket or Unix socket.

Daemon Lifecycle

The extension manages the daemon process lifecycle. On activation, it checks if the daemon is already running (a previous VS Code window may have started it). If not, it spawns the daemon as a detached child process that outlives any single VS Code window.

Startup sequence

Extension activated (VS Code opens workspace)
  │
  ├─ 1. Check: is daemon running?
  │         ping localhost:<port> or ~/.insrc/daemon.sock
  │         ✓ already running → connect
  │         ✗ not running    → spawn daemon process
  │
  ├─ 2. Check: Ollama installed?
  │         ✓ continue
  │         ✗ → show setup wizard step: "Install Ollama"
  │
  ├─ 3. Check: qwen3-coder + qwen3-embedding:0.6b pulled?
  │         ✓ continue
  │         ✗ → daemon pulls both automatically, extension shows per-model progress bars
  │
  ├─ 4. Check: ANTHROPIC_API_KEY configured?
  │         ✓ continue
  │         ✗ → show setup wizard step: "Add API key" (skippable — local-only mode)
  │
  └─ 5. Ready → show status bar ✓, enable chat panel
Daemon is shared across windows: If the user opens multiple VS Code windows, all connect to the same daemon instance. The daemon handles concurrent sessions per repo. Only the first window to activate spawns the daemon; subsequent windows connect to the existing process.

Daemon installation

The insrc daemon (Node.js + npm dependencies) is bundled inside the VSIX. On first activation the extension extracts it to ~/.insrc/daemon/ and registers it for auto-start. Subsequent updates replace the daemon binary when a new extension version is installed.

First-Run Setup Wizard

Shown automatically on first activation when any required component is missing. Each step is independently skippable. The wizard re-opens from the status bar or settings panel at any time.

Step 1 Ollama

  • Detect Ollama at localhost:11434 or custom host
  • If missing: show install button (opens browser) or run curl … | sh inline
  • If remote: show host field — validate with health check ping

Step 2 Local models

  • Check if qwen3-coder:latest is pulled — agent model
  • Check if qwen3-embedding:0.6b is pulled — embedding model (graph search + session memory)
  • If either missing: pull automatically with per-model progress bars in wizard
  • Custom agent model: dropdown to select any installed Ollama model

Step 3 Anthropic API key

  • Text field — validates by making a minimal test call
  • Skippable: extension works in local-only mode without it
  • Stored in ~/.insrc/config.json, not in VS Code settings

Step 4 Brave Search (optional)

  • Text field — validates with a test query
  • Skippable: falls back to Claude web search tool
  • Link to free signup (2,000 queries/month)
Local-only mode: If the user skips the Anthropic API key, the extension operates fully locally — Ollama handles all requests. Intents that normally escalate to Claude (requirements, design, plan, review) are handled by the local model with a visible LOCAL-ONLY badge. Quality is lower but no API calls are made.

Settings

Settings are split into two layers: VS Code extension settings (workspace/user scope, in settings.json) for IDE behaviour, and insrc config (~/.insrc/config.json) for agent behaviour. The extension settings panel provides a UI for both, but writes to the correct backing store for each.

Ollama host

SettingDefaultDescription
insrc.ollama.host http://localhost:11434 Ollama server URL. Change to a remote host if Ollama runs on a different machine (e.g. a GPU workstation). The daemon health-checks this URL on startup and on every model request. If unreachable, the agent falls back to Claude for all intents and shows a status bar warning.

Local model

SettingDefaultDescription
insrc.model.local qwen3-coder:latest The Ollama model used for local inference. Must be a model name available on the configured Ollama host. The extension shows a dropdown populated by querying GET /api/tags on the Ollama host. Changing this setting restarts the daemon's model connection.

Claude model tiers

Maps to the models section of ~/.insrc/config.json. The settings UI exposes tier defaults and per-role overrides. See the agent design document for the full role list.

SettingDefaultDescription
insrc.model.tiers.fast claude-haiku-4-5 Model for call validation, diff validation, fix escalation. Dropdown: Haiku / Sonnet / Opus.
insrc.model.tiers.standard claude-sonnet-4-6 Model for requirements, design, plan, review. Dropdown: Haiku / Sonnet / Opus.
insrc.model.tiers.powerful claude-opus-4-6 Model for explicit @opus escalation only. Dropdown: Sonnet / Opus.
insrc.model.roles.* (tier defaults) Per-role overrides. Exposed as an expandable table in the settings UI — each row is a role name (e.g. review, implement.validate) with a tier dropdown. Empty = use tier default.

API keys

SettingStored inDescription
insrc.anthropic.apiKey ~/.insrc/config.json Anthropic API key. Displayed as a masked field with a "Validate" button that makes a minimal test call. Stored in ~/.insrc/config.json, not in VS Code settings — so it is shared with the CLI and never committed to a workspace.
insrc.search.braveApiKey ~/.insrc/config.json Brave Search API key. Optional. If empty, web searches fall back to Claude's hosted WebSearch tool. Validated with a test query on save.

Permission mode

SettingDefaultDescription
insrc.permissions.mode validate Controls whether mutating tool calls are validated by Claude/Haiku before execution.

validate — Claude/Haiku validates all mutating calls (Write, Edit, Bash mutations, kubectl apply/exec/delete, git commit/push/reset) before execution. Recommended default.

auto-accept — Mutating calls execute immediately without Claude validation. Removes the ~$0.0004/call validation cost but increases risk of unintended mutations. Suitable for trusted local workflows where the user monitors output.

The current mode is always shown in the status bar. Can be toggled per-session from the chat panel without changing the persistent setting.
auto-accept mode risk: In auto-accept mode, the local LLM can write files, run shell commands, and execute kubectl mutations without any Claude review. The extension shows a persistent AUTO-ACCEPT badge in the status bar whenever this mode is active to keep the user aware.

Chat panel — floating

The chat panel lives in the secondary sidebar by default. A Pop out button in the chat header opens it as a WebviewPanel editor tab, which can then be dragged outside VS Code into a floating OS window (VS Code 1.85+). This is ideal for dual-monitor setups — chat floats on a second screen while the main window shows code.

  • Float: click Pop out in chat header, then drag tab outside VS Code — or right-click tab → Move into New Window
  • Re-attach: drag tab back into VS Code — or right-click tab → Move into Main Window
  • Session state is preserved across float/attach cycles (retainContextWhenHidden: true)

Chat panel — layout

The chat panel is split vertically into two sections: History (top, scrollable) and Action (bottom, fixed height). A persistent repo indicator sits in the chat header, always showing the active repo context. The split is resizable by dragging the divider. The action panel content is context-driven — it changes based on what the agent is doing. When a plan is active, the action panel is locked to plan progress and cannot be replaced by other content until the plan is complete or cancelled.

Repo context indicator

The active repo is shown as a clickable pill in the chat header: [insrc ▾]. Every message sent in the session is scoped to this repo's graph context.

  • Click the pill → dropdown lists all indexed repos; select one to switch context for subsequent turns in this session
  • Switching repo does not start a new session — the history stays; a separator line is inserted: — context switched to api-gateway —
  • In the input box, type @repo:<name> to switch context inline without opening the dropdown — fuzzy-matched against known repo names: 
    @repo:api-gateway why does /health return 503?
  • @repo: with no name opens the dropdown picker at the cursor position
  • The navigation panel highlights the newly active repo node when context switches
  • Repo context is persisted per session — re-opening a session restores the last active repo

History panel

  • Scrollable stream of conversation turns — user messages and agent responses
  • Tool calls rendered as collapsible blocks: tool name + input summary, expandable output
  • Escalation notices inline: provider, model, estimated token cost before each Claude call
  • @opus cost confirmation rendered as an inline card — confirm or cancel without leaving the panel
  • Timestamps and intent badges on each agent turn

Action panel

Shows the most relevant action context for the current state. Priority order (highest wins):

ConditionAction panel content
Plan is active Always shown — cannot be displaced. Step tracker: current step name, step N of M, status badge, Continue / Skip / Cancel plan buttons. Progress bar across all steps. Tool spinners and diff decisions appear as transient overlays inside the tracker card rather than replacing it.
Agent processing a turn Spinner + current tool name (read_file src/index.ts…). Input box disabled with a Cancel button.
Diff awaiting decision Accept / Reject / Edit buttons for the pending diff. File name and change summary shown above the buttons.
Idle Input box — supports @file mentions (fuzzy picker), @repo:<name> to switch repo context inline, @claude / @opus / @local prefixes, file drag-and-drop attachments, and / command menu.
Plan always wins the action panel. While a plan is active, the step tracker is pinned. The input box is still accessible via a collapsed drawer at the bottom of the action panel for mid-plan questions to the agent.

Repo management

  • Active repo — the repo matching the current VS Code workspace folder; expanded by default; sessions are scoped to this repo's graph context
  • Indexed repos — any repo previously indexed by the daemon appears in the list; user can expand it and resume any past session regardless of which folder VS Code has open
  • Stale badge ⚠ — shown when the repo's index is out of date; click to trigger a re-index
  • ＋ Add repo button at the top — opens a folder picker; the selected path is registered with the daemon and indexing starts immediately with a progress indicator under the repo node
  • Right-click a repo node: Re-index · Remove from list (does not delete the repo, only removes it from the daemon's tracked list)

Session list

  • Sessions grouped by date under each repo: Today · Yesterday · This week · older
  • Each session node shows: first message preview, intent badge, timestamp
  • Click a session → switches active repo context in the daemon, loads the session into the chat panel, restores last active code view
  • Right-click a session: Rename · Delete · Export
  • Search box at the top searches sessions across all repos (queries daemon graph)

Settings panel

A dedicated webview panel (accessible via insrc: Open Settings command or the /settings chat command) that provides a structured form UI for all insrc configuration. Changes are written immediately to the appropriate backing store (~/.insrc/config.json or VS Code settings.json) and take effect without restarting VS Code.

Sections: Ollama · Local model · Claude models · API keys · Permissions · Daemon

Intent-Specific Views

Each intent has a distinct primary artifact rendered in the Code panel (centre editor area). The chat panel (right sidebar) remains accessible at all times alongside the code view. When the daemon signals an intent, the extension opens the matching WebviewPanel in the editor area automatically.

View switching is automatic but not forced. The extension opens the intent-specific code view when a turn starts. The user can close it, drag it to a different column, or ignore it entirely — the chat panel always works standalone. Per-intent view preference is remembered.

implement / refactor

Primary artifact: file diff

  • Split view: chat stream on left, native VS Code diff editor on right
  • Pipeline status bar above the diff: Stage 1 — local generating… → Stage 2 — Claude validating… → APPROVED / CHANGES_NEEDED (retry 1/2)
  • Per-file diff tabs when multiple files are changed
  • Accept / Reject / Edit buttons pinned to the bottom of the diff panel
  • On exhausted retries: diff border turns yellow, manual decision prompt appears inline

test

Primary artifact: test run results

  • Test results panel (VS Code native Testing API) shows pass / fail per test case
  • Fix loop counter shown in panel header: Fix loop — attempt 2/3 (local) or Escalating to Claude…
  • Failing test highlighted with the current error and stack trace inline
  • Fix diff shown in a secondary split alongside the failing test body so the user can see what the model is changing and why
  • Terminal condition prompt appears inline when claudeRounds cap is reached: accept / discard / open editor

debug

Primary artifact: tool call timeline

  • Timeline view showing each tool call as a collapsible row: tool name, command/input, output summary, auto-execute vs. validated badge
  • Read-only calls collapsed by default (low noise); mutating calls and their validation result expanded by default
  • Running cost counter in the panel header (Claude calls only): Claude spend: $0.0008
  • Stuck escalation shown as a distinct row: local model's ≤200-token summary sent to Claude, Claude's direction returned — both visible
  • Fix diff appears in a split when the local model produces a patch, same as implement

plan

Primary artifact: plan step tracker

  • Kanban-style column view: Pending · In Progress · Done · Failed · Skipped
  • Each step card shows: step description, estimated effort, dependent steps
  • Active step highlighted with a pulsing border; click to start it immediately
  • Step state transitions animated in real time as plan_step_update events arrive from the daemon
  • Click any completed step to expand its diff and Claude validation result
  • Toolbar: Start next · Skip step · View full plan

review

  • Review output rendered as a structured document rather than a chat message: sections for Security · Performance · Correctness · Style
  • Each finding linked to the relevant file and line — click to jump to the location in the editor

> "Response time < 200ms"

  • Annotation mode is cleared after sending; annotations can also be discarded individually or all at once with Clear all

Code Annotations

Annotations are a cross-cutting feature available in any file open in the VS Code editor — not just requirements and design views. The user selects a range of code, adds a comment, and when done sends all annotations to the chat panel as structured context for the next agent turn. The mechanism is the same as in the requirements/design split panel but implemented using native VS Code editor APIs.

How it works

  • Select any text range in any open editor file
  • A CodeLens appears above the selection: ＋ Add insrc annotation — or use the right-click context menu → insrc: Add annotation — or keybind Ctrl+Shift+A / Cmd+Shift+A
  • A VS Code InputBox appears — type the annotation and press Enter
  • The annotated range is decorated with an amber highlight and a gutter icon; a CodeLens above it shows the annotation text (truncated to 60 chars)
  • Multiple annotations accumulate across multiple files
  • A badge in the chat panel action area and status bar shows N annotations pending
  • Click Send annotations to chat (chat action panel or status bar) to compile and send

Annotation message format

Sent to the chat as a user turn with structured context:

Code annotations:

src/payment/handler.ts  lines 42–51
> const result = await db.query(sql, params)
  → this runs inside a loop — N+1 query problem

src/payment/handler.ts  lines 78–80
> if (err.code === 'TIMEOUT') retry()
  → retry has no backoff or limit — can storm the DB

src/auth/middleware.ts  line 12
> req.user = jwt.decode(token)
  → decode does not verify signature — use jwt.verify

VS Code API used

  • vscode.languages.registerCodeLensProvider — renders the Add annotation and annotation preview lenses above selections/annotated ranges
  • vscode.window.createTextEditorDecorationType — amber highlight + gutter icon on annotated ranges
  • vscode.window.showInputBox — annotation text input
  • Annotations stored in extension memory (not on disk) — cleared after sending or on session close
Works alongside requirements / design views. The same annotation queue is shared — a user can annotate code files and the requirements preview in the same session, then send all annotations to chat in one batch.

IPC Protocol

The extension and daemon communicate over a local WebSocket (TCP on localhost:<port>, port stored in ~/.insrc/daemon.json) or a Unix domain socket (~/.insrc/daemon.sock). All messages are newline-delimited JSON.

Message types (extension → daemon)

// Send a user turn
{ "type": "turn", "sessionId": "sess_abc", "repoId": "repo_xyz",
  "message": "debug why payments are failing",
  "attachments": [{ "kind": "text", "name": "crash.log", "content": "..." }] }

// Cancel in-progress turn
{ "type": "cancel", "sessionId": "sess_abc" }

// Toggle permission mode for session
{ "type": "setPermissionMode", "sessionId": "sess_abc", "mode": "auto-accept" }

// Re-index current repo
{ "type": "reindex", "repoId": "repo_xyz" }

Message types (daemon → extension)

// Streaming token delta
{ "type": "delta", "sessionId": "sess_abc", "text": "Found OOMKilled in describe..." }

// Tool call notification (shown as collapsible block)
{ "type": "toolCall", "sessionId": "sess_abc",
  "tool": "Bash", "input": "kubectl logs payment-svc --tail=100",
  "validation": "auto-execute" }

// Tool result
{ "type": "toolResult", "sessionId": "sess_abc",
  "tool": "Bash", "output": "...", "isError": false }

// Escalation notice (shown before Claude call)
{ "type": "escalation", "sessionId": "sess_abc",
  "provider": "claude", "model": "claude-haiku-4-5",
  "role": "call_validation", "estimatedTokens": 320 }

API keys not in VS Code settings: ANTHROPIC_API_KEY and BRAVE_API_KEY are intentionally absent from the VS Code contributes.configuration schema. They are stored in ~/.insrc/config.json only — this prevents them from appearing in workspace settings.json files and being accidentally committed to git.

Implementation Phases

Phase 1 Scaffold + Daemon lifecycle

  • Extension activates, spawns daemon
  • WebSocket IPC established
  • Status bar connected to daemon status
  • Setup wizard (Ollama + model pull)

Phase 2 Chat panel

  • Webview chat UI with streaming
  • Tool call blocks (collapsible)
  • Escalation notices inline
  • @file mentions + attachments

Phase 3 Diff viewer + permissions

  • Native VS Code diff editor for patches
  • Accept / reject / edit controls
  • Permission mode toggle in panel
  • @opus cost confirmation flow

Phase 4 Settings panel + polish

  • Webview settings form
  • Per-role model override table
  • API key validation UI
  • Plan progress header
