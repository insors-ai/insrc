LOCAL-FIRST

insrc — Installation & Setup

insrc indexes your repos into a structural graph + vector store and serves citation-grounded exploration (analyze) and multi-step engineering workflows (define → design → plan → build → test) to a coding agent over a local socket.

1 · Requirements

ComponentRoleRequired?
Node.js ≥ 20Runtime. Native modules build against Node 22.22.1 headers; the daemon spawns under Node 22.Mandatory
Ollama (running, http://localhost:11434)The semantic layer: entity + query embeddings, and the analyze tool-loop. Cloud CLIs can't do these.Mandatory
claude / codex CLICloud reasoning for analyze synthesis + workflow generation (higher accuracy). Uses the CLI's own OAuth — no API keys stored.Recommended
gitRepo indexing + tracker linkage.Mandatory (indexing)
gh CLIGitHub tracker integration (tracker.push/sync/post, Task sub-issues).Optional
Why Ollama is mandatory even when reasoning runs on the cloud CLI. insrc is local-first by design: embeddings never leave your machine — the claude/codex CLIs expose no embeddings API, so the vector index + semantic search (concept.resolve, capability.reuse-check) can only run through a local embedder. Separately, the analyze tool-loop (freeform.probe) needs a tool-calling provider, which the one-shot CLI wrapper can't drive — so that path is Ollama-only too. Cloud CLIs do the reasoning; Ollama does the semantic layer. Both are needed.

2 · Install

The daemon lives at ~/.insrc/daemon. Two paths:

A · Release installer (recommended)

./insrc-daemon-install.sh      # clones + builds into ~/.insrc/daemon

B · From source (developers / IDE fork)

git clone https://github.com/insors-ai/insrc ~/.insrc/daemon
cd ~/.insrc/daemon
npm install                    # builds native modules against Node 22 headers
npm run build                  # tsc + copy-assets → out/

Binaries after build:

  • out/cli/index.js — the insrc interactive TUI
  • out/bin/insrc-mcp.js — the MCP server a coding agent connects to
  • out/daemon/index.js — the background daemon
The IDE fork (insors-ai/insrc-ide) performs path B automatically and spawns the compiled daemon; the IPC socket at ~/.insrc/daemon.sock is the only surface it consumes.

3 · Ollama models

Pull these, or let the Setup pane (§6) recommend + pull them for your hardware.

ModelPurposeRequired?Default id
EmbeddingSemantic index + query embeddings (1024-dim)Mandatory (local-only)qwen3-embedding:0.6b
Analyzer core (shaper)Analyze / workflow reasoning when shaperProvider: "ollama"Only when reasoning runs locally (skip if using the claude / codex CLI)qwen3.6:35b-a3b
 
ollama pull qwen3-embedding:0.6b
ollama pull qwen3.6:35b-a3b        # analyzer core — skip if reasoning runs on claude/codex
Using a CLI for reasoning (cli-claude/cli-codex) means you don't need the shaper model — but you still need the embedding model (the semantic layer is always local).

4 · Configuration

Two files under ~/.insrc/.

~/.insrc/config.json — models

{
  "models": {
    "providers": {
      "local": {
        "host":           "http://localhost:11434",
        "embeddingModel": "qwen3-embedding:0.6b",
        "embeddingDim":   1024,
        "coreModel":      "qwen3.6:35b-a3b"
      }
    },
    "analyze": {
      // Reasoning provider. Default "ollama".
      //   "ollama"     — local shaperModel below
      //   "cli-claude" — the claude CLI
      //   "cli-codex"  — the codex CLI
      "shaperProvider": "ollama",
      "shaperModel":    "qwen3.6:35b-a3b",   // Ollama id; OMIT for a cli-* provider
      "shaper": { "ollamaNumCtx": 32768, "maxToolTurns": 8, "structuredOutputRetries": 3 }
    }
  }
}

Provider selection — one decision, applied consistently:

  • Explicit shaperProvider → that provider is used for all reasoning.
  • Not set → reasoning follows the invoking agent (Claude Code → claude, Codex → codex); Ollama is the standalone fallback.
  • Regardless, Ollama is always used for embeddings + the tool-loop (see §1).
Local (Ollama) structured output: the local path constrains JSON via Ollama's grammar engine, which can reject some strict schemas ("failed to parse grammar"). For analyze/workflow reasoning, prefer a cli-* provider until that's hardened; embeddings are unaffected.

~/.insrc/github.json — tracker (optional)

GitHub Issues integration. Opt-in — absent / "type": "none" disables it.

{
  "default": {
    "type":  "github",          // or "none" to disable
    "owner": "myorg",
    "repo":  "myrepo",
    "epicLabel": "insrc:epic", "storyLabel": "insrc:story", "taskLabel": "insrc:task",
    "useMilestones": false,
    "pushTasks":     false,     // push Task-tier issues (type Task, sub-issues of the Story)
    "taskIssueType": "Task"     // org must have this issue type; falls back to untyped
  },
  "repos": { "/abs/path/to/repo": { "type": "github", "owner": "…", "repo": "…" } }
}

Requires an authenticated gh CLI. pushTasks + typed sub-issues also need the org to have GitHub issue types + sub-issues enabled (both degrade gracefully if absent).

5 · Connect a coding agent (Claude Code / Codex)

Register the insrc MCP server so the agent can call insrc_analyze / insrc_analyze_step / insrc_workflow_step:

# Claude Code (per-project, from the repo root):
claude mcp add insrc --scope local --env INSRC_REPO="$PWD" \
  -- node "$HOME/.insrc/daemon/out/bin/insrc-mcp.js"

You don't need to paste anything: when you register a repo (insrc TUI → Repos → a, or the repo.add IPC), insrc offers to install the steering block into the repo's CLAUDE.md (Claude Code) and/or AGENTS.md (Codex) — a safe, idempotent insrc:steering marked section that is never clobbered on re-add. To install by hand instead, copy src/prompts/steering-block.md into the file wrapped in those markers.

MCP tools load on a fresh agent session — after registering, restart Claude Code / Codex.

6 · Using the insrc TUI

Launch the interactive TUI (requires a real terminal / TTY):

insrc                 # if built + on PATH
# or, from ~/.insrc/daemon:
npm run insrc         # dev (tsx)

Global keys: Tab / Shift+Tab or 1–4 switch panes · r refresh · : command · ? help · q quit.

PanePurposeKeys
DaemonHealth + lifecycle.s start · x stop · u update (FF → install-if-changed → build → restart) · b backup · c compact
ReposRegister + index repositories.a add · i reindex · d remove
WorkflowsPer-Epic chain status; approve/reject artifacts.a approve · x reject
SetupDetects CPU/RAM/Ollama, recommends models, writes config + pulls models.a apply config · p pull models
Fastest first run: Setup → a (write hardware-matched config) → p (pull missing models) → Daemon → s (start) → Repos → a (add + index your repo).

7 · Verify

  • Daemon pane shows running + a bound socket (~/.insrc/daemon.sock).
  • Repos pane shows your repo indexed (queue depth 0).
  • From a registered agent session, ask a codebase question — insrc_analyze returns a citation-grounded bundle with real file paths.

If analyze returns "Local Ollama unavailable", confirm ollama serve is running and the embedding model is pulled (§3).
